Mengapa kami menyertakan kriptografi?

Dalam tiga kata: karena kami bisa.

Proyek OpenBSD berbasis di Kanada.

Daftar Kendali Ekspor Kanada tidak menentukan batasan berati pada pengeksporan peranti lunak kriptografi, dan meskipun lebih eksplisit tentang ekspor bebas peranti lunak kriptografi yang tersedia gratis. Marc Plumb telah menyelesaikan beberapa riset untuk menerapkan hukum kriptografi.

Karena itu proyek OpenBSD menyertakan kriptografi ke sejumlah tempat pada sistem operasi. Kami mensyaratakan peranti lunak kriptografi yang kami gunakan tersedia gratis dan berlisensi yang bagus. Kami tidak langsung menggunakan kriptografi dengan paten yang menyusahkan. KAmi juga mensyaratkan beberapa peranti lunak berasal dari negara-negara dengan lisensi ekspor yang berguna karena kami tidak ingin melanggar hukun di negara manapun. Komponen peranti lunak kriptografik yang kami gunakan saat ini ditulis di Argentina, Australia, Kanada, Jermany, Yunani, Norwegia, dan Swedia.

Ketika kami menciptakan rilis OpenBSD atau snapshot kami bangun binari rilis kami di negara yang benar-benar bebas untuk memastikan bahwa kode sumber dan bianri yang kami sediakan kepada pengguna bebas dari kecacatan. Di masa lampau binari rilis kami bangun di Kanada, Swedia, dan Jerman.

OpenBSD ships with Kerberos IV and Kerberos V included. The two codebases we use are the exportable KTH-based release from Sweden. Our X11 source has been extended to make use of Kerberos as well.

OpenBSD was the first operating system to ship with an IPsec stack. We've been including IPsec since the OpenBSD 2.1 release in 1997. Our fully conformant in-kernel IPsec stack, with hardware acceleration based on a number of cards, and our own free ISAKMP daemon, is used as one of the machines in the IPsec conformance testbed run by VPNC.

Today cryptography is an important means for enhancing the security of an operating system. The cryptography utilized in OpenBSD can be classified into various aspects, described as follows.

OpenSSH

What is the first thing most people do after installing OpenBSD? They install Secure Shell (ssh(1)) from the ports tree or the packages on the FTP sites. Until now, that is.

As of the 2.6 release, OpenBSD contains OpenSSH, an absolutely free and patent unencumbered version of ssh. As of the OpenBSD 2.6 release date, OpenSSH interoperated with ssh version 1 and had many added features,

  • all components of a restrictive nature (i.e., patents, see ssl(8)) had been directly removed from the source code; any licensed or patented components used external libraries.
  • had been updated to support ssh protocol 1.5.
  • contained added support for kerberos(1) authentication and ticket passing.
  • supported one-time password authentication with skey(1).

Roughly said, we took a free license release of ssh, OpenBSD-ifyed it. About a year later, we extended OpenSSH to also do SSH 2 protocol, the result being support for all 3 major SSH protocols: 1.3, 1.5, 2.0.

Pseudo Random Number Generators

A Pseudo Random Number Generator (PRNG) provides applications with a stream of numbers which have certain important properties for system security:

  • It should be impossible for an outsider to predict the output of the random number generator even with knowledge of previous output.
  • The generated numbers should not have repeating patterns which means the PRNG should have a very long cycle length.

A PRNG is normally just an algorithm where the same initial starting values will yield the same sequence of outputs. On a multiuser operating system there are many sources which allow seeding the PRNG with random data. The OpenBSD kernel uses the mouse interrupt timing, network data interrupt latency, inter-keypress timing and disk IO information to fill an entropy pool. Random numbers are available for kernel routines and are exported via devices to userland programs. So far random numbers are used in the following places:

  • Dynamic sin_port allocation in bind(2).
  • PIDs of processes.
  • IP datagram IDs.
  • RPC transaction IDs (XID).
  • NFS RPC transaction IDs (XID).
  • DNS Query-IDs.
  • Inode generation numbers, see getfh(2) and fsirand(8).
  • Timing perturbance in traceroute(8).
  • Stronger temporary names for mktemp(3) and mkstemp(3)
  • Randomness added to the TCP ISS value for protection against spoofing attacks.
  • random padding in IPsec esp_old packets.
  • To generate salts for the various password algorithms.
  • For generating fake S/Key challenges.
  • In isakmpd(8) to provide liveness proof of key exchanges.

Cryptographic Hash Functions

A Hash Function compresses its input data to a string of constant size. For a Cryptographic Hash Function it is infeasible to find:

  • two inputs which have the same output (collision resistant),
  • a different input for a given input with the same output (2nd preimage resistant).

In OpenBSD MD5, SHA1, and RIPEMD-160 are used as Cryptographic Hash Functions, e.g:

  • In S/Key(1) to provide one time passwords.
  • In IPsec(4) and isakmpd(8) to authenticate the data origin of packets and to ensure packet integrity.
  • For FreeBSD-style MD5 passwords (not enabled by default), see passwd.conf(5)
  • In libssl for digital signing of messages.

Cryptographic Transforms

Cryptographic Transforms are used to encrypt and decrypt data. These are normally used with an encryption key for data encryption and with a decryption key for data decryption. The security of a Cryptographic Transform should rely only on the keying material.

OpenBSD provides transforms like DES, 3DES, Blowfish and Cast for the kernel and userland programs, which are used in many places like:

  • In libc for creating Blowfish passwords. See also the USENIX paper on this topic.
  • In IPsec(4) to provide confidentiality for the network layer.
  • In Kerberos and a handful of kerberized applications, like telnet(1), cvs(1), rsh(1), rcp(1), and rlogin(1).
  • In isakmpd(8) to protect the exchanges where IPsec key material is negotiated.
  • In AFS to protect the messages passing over the network, providing confidentiality of remote filesystem access.
  • In libssl to let applications communicate over the de-facto standard cryptographically secure SSL protocol.

Dukungan Peranti Keras Kriptografik

OpenBSD, diawali pada 2.7, telah mulai mendukung beberapa peranti keras kriptografi seperti akselerator dan pembangkit angka acak.

  • Dequeue kripto IPsec
    Stack IPsec kami telah dimodifikasi sehingga fungsi kriptografi diselesaikan diluar garis besar. Kebanyakan stack IPsec peranti lunak sederhana perlu melakukan kriptografi ketika memproses setiap paket. Menghasilkan kinerja sinkronus. Untuk menggunakan peranti keras dengan tepat dan mempercepatnya perlu dilakukan pemisahan dua komponen ini, seperti yang telah kami lakukan. Sebenarnya, hal ini meningkatkan cukup kinerja meskipun untuk kasus peranti lunak.

  • Hifn 7751
    Kartu yang menggunakan Hifn 7751 dapat dipakai sebagai pemercepat kriptografi asimetrik, seperti Soekris VPN1201 atau VPN1211 (untuk membeli) atau PowerCrypt. Kinerja saat ini dengan menggunakan Hifn 7751 tunggal pada setiap akhir kanal adalah 64Mbit/detik untuk ESP 3DES/SHA1, peningkatannya mendekati 600% dengan CPU P3/550. Peningkatan lebih lanjut masih dalam pengerjaan, tetapi pada 13 April 2000 kode sumber diakui telah stabil. Kami menulis sendiri pengendali untuk chip ini, daripada menggunakan pengendali (ditulis seorang USA) PowerCrypt, lebih baik menggunakan pengendali kami yang terhubung dengan benar ke stack IPsec. 7751 saat ini dianggap lambat oleh standar industri dan banyak vendor telah memiliki chip lebih cepat (meskipun Hifn sekarang lebih cepat tetapi lebih mahal). Puncak kinerja dengan 3DES SHA1 ESP sekitar 64Mbit/detik.

    Setelah pengiriman 2.9, dukungan telah ditambahkan untuk chip Hifn 7951, sebuah versi tersederhanakan dari 7751 yang menambahkan sebuah pemercepat kunci umum (tidak didukung) dan pembangkit angka acak (didukung). Kartu tersebut disumbangkan oleh Soekris Engineering.

    Setelah pengiriman 3.0, dukungan telah ditambahkan pada chip Hifn 7811, versi yang lebih cepat pada seri 7751 (sekitar 130Mbit/detik) dengan sebuah pembangkit angka acak. Kartu tersebut disumbangkan oleh GTGI.

    Setelah pengiriman 3.2, dukungan telah ditambahkan untuk algoritma pemampatan LZS yang digunakan ipcomp(4).

    Hifn awalnya sebuah perusahaan yang sulit bekerja sama (mengancam untuk menuntut kami sebagai pelaku pengembang curang non-USA pada algoritma pembukaan kunci kripto mereka), tetapi akhir-akhir ini mereka sangat membantu dalam menyediakan kartu dan bantuan.

  • Hifn 6500
    Divais ini merupakan unit kripto asimetrik. Mempunyai dukungan algoritma RSA, DSA, dan DH, dengan fungsi utama lain berupa angka besar. Juga mempunyai pembangkit angka acak berkinerja sangat tinggi. Kami memiliki satu divais, dokumentasi penuh, dan contoh kode sumber. Pada OpenSBD 3.1, pembangkit angka acak dan unit angka besar bekerja dengan baik.

  • Hifn 7814/7851/7854
    Divais ini adalah sebuah paket prosesor bersama unit kripto asimetrik. Mendukung algoritma RSA, DSA, dan DH, dengan fungsi utama angka besar dan juga mempunyai pembangkit angka acak. Saat ini, hanya mesin angka besar dan pembangkit angka acak yang didukung (tanpa pergantian bentuk paket).

  • Broadcom BCM5801/BCM5802/BCM5805/BCM5820/BCM5821/BCM5822 (atau chip Bluesteelnet 5501/5601 beta)
    Setelah rilis OpenBSD 2.7, kami berhasil menambahkan dukungan persiapan untuk bagian chipset yang baru dirilis ini yang disediakan vendor untuk kami, khususnya diawali dengan percobaan pada chip 5501. Divais ini menyediakan kriptografi simetrik berkinerja tinggi dari yang pernah kami lihat.

    Bluesteelnet telah dibeli oleh Broadcom dan mulai membuat bagian asli. BCM5805 baru mereka mirip, kecuali mereka juga menambahkan mesin asimetrik untuk menjalankan DSA, RSA, sejumlah algoritma lainnya. Dengan perkiraan kinerja awal yang empat kali lebih cepat dari Hifn, diharapkan chip ini akan menjadi umum dalam waktu dekat.

    Broadcom/Bluesteelnet telah menyepakatinya. Mereka memberikan kami dokumentasi lengkap dan contoh kode sumber untuk chip mereka dan sejumlah kartu yang memadai untuk dicoba.

    Akhir 2.8, pengendali ini juga telah dimodofikasi untuk membangkitkan angka acak pada BCM5805 dan versi sejenis, dan memebrikan datanya ke pool entropi kernel.

    Akhir 2.9, dukungan telah ditambahkan pada BCM5820, versi BCM5805 yang umumnya lebih cepat (64bit, kecepatan detak lebih tinggi). Dukungan tak melewati percobaan juga ditambahkan untuk BCM5821 pada akhir 3.0.

    Sebagaimana 3.1, sejumlah besar mesin terdukung, dan operasi RSA/DH/DSA bisa dipercepat.

    Dukungan untuk BCM5801, BCM5802, BCM5821 dan BCM5822 telah ditambahkan sebelum OpenBSD 3.2 (dukungan BCM5821 yang belum melewati percobaan pada 3.1 telah kacau karena beberapa kebutuhan pengelolaan interup yang tidak terdokumentasi).

  • Securealink PCC-ISES
    PCC-ISES adalah sebuah chipset baru dari Belanda. Kami telah menerima contoh peranti keras dan dokumentasinya, dan pekerjaan untuk pengendalinya dalam perkembangan. saat ini, pengendali bisa menyediakan angka acak ke pool entropi kernel.

  • SafeNet SafeXcel 2141
    Kami telah menerima dokumentasi dan contoh peranti keras kartu kripto SafeNet. Pekerjaan untuk mendukung paling tidak kriptografi simetrik diviais ini etlah dimulai.

  • 3com 3cr990
    3com telah memberi kami sebuah pengendali untuk mendukung komponen ethernet chipset ini, dan berdasar itu, kami telah menulis pengendali ethernet kami sendiri. Saat ini pengendali ini belum diintegrasikan karena masih belum bisa mendapatkan lisensi gratis dari. Karena dokumentasi yang Karena dokumentasi yang minim dan kurang kerjasama (sebagian karena tingginya pergantian pegawai 3Com), fungsi IPsec chip ini tidak terdukung... sehingga usaha ini dihentikan paling tidak menjadi latihan yang sepenuhnya berguna.

  • Kartu IPsec Intel
    Seperti kebanyakan Intel membuat komponen divisi jaringan mereka sendiri, dan keseluruhan tidak sama dengan evndor lainnya, Intel dengan cepat tetap menolak memberikan dokumentasi pada kami. Kami harus berbicara dengan sekitar lima orang teknis yang ikut serta dalam pengembangan produk tersebut. Mereka semua ingin agar kami mempunyai dokumentasinya. Mereka berkomentar tentang apa yang etlah kami lakukan. Tetapi tangan mereka dibelenggu yang tidak ingin melihat sebuah keuntungan bagi mereka sendiri dengan menyediakan dokumentasi. Lupakan Intel. (Jika anda ingin membeli peranti keras ethernet gigabit, kami sarankan untuk beli yang lainnya...untuk alasan yang sama: kebanyakan pengendali yang kami punyai untuk peranti keras jaringan Intel dibuat tanpa dokumentasi dari mereka).

  • Intel 82802AB/82802AC Firmware Hub RNG
    Chip FWH 82802 (ditemukan dalam papan utama i810, i820, i840, i850, dan i860) mempunyai sebuah pembangkit angka acak (RNG). IPsec berkinerja tinggi membutuhkan entropi angka yang lebih acak. Pada 10 April 2000, kami mendukung RNG. Kami akan menambahkan dukungan RNG lain yang ditemukan dalam chip kripto.

  • VIA C3 RNG
    Cpu C3 VIA terbaru mempunyai sebuah pembangkit angka acak sebagai sebuah petunjuk. Sebagaimana pada 3.3 pembangkit angka acak ini digunakan dalam kernel untuk mendukung pool entropi.

  • OpenSSL
    Bertahun lalu, kami mempunyai rancangan besar untuk mendukung kartu kriptoang bisa melakukan otomasi RSA/DH/DSA lewat panggilan OpenSSL. Sebagaimana OpenBSD 3.2, dukungan tersebut berjalan, dan kartu apapun yang terdukung dengan sejumlah fungsionalitas akan otomatis menggunakan peranti keras, termasuk OpenSSH dan httpd dalam modus SSL. Tanpa perubahan aplikasi yang diperlukan.

Jika ada yang mau membantu kami dalam menulis pengendali, selamat datang dan bantulah kami.

Dicari Kriptografer Internasional

Tentu, proyek kami membutuhkan orang yang bekerja untuk sistem ini. Jika ada siapa saja kriptografer non-Amerika yang memenuhi syarat pada penjelasan sebelumnya dan ingin membantu kami untuk menyertakan kriptografi dalam OpenBSD, mohon hubungi kami.
