1.0 - Cos'è OpenSSH e Come Posso Procurarmelo?

1.1 - Cos'è OpenSSH e dove posso scaricarlo?

OpenSSH è una versione LIBERA della suite di strumenti di connessione di rete SSH cui si affida un numero sempre crescente di persone su internet. Molte persone che usano telnet, rlogin, ftp e altri programmi simili possono non rendersi conto che la loro password viene trasmessa su internet in chiaro, ma è così. OpenSSH critta tutto il traffico (incluse le password) per eliminare efficacemente l'ascolto passivo, l'hijacking di connessione e altri attacchi a livello di rete.

La suite OpenSSH include il programma ssh(1) che sostituisce rlogin e telnet, e scp(1) che sostituisce rcp(1) e ftp(1). OpenSSH ha inoltre aggiunto sftp(1) e sftp-server(8) che implementano una soluzione più semplice per il trasferimento dei file, basati sul draft secsh-filexfer dell' IETF.

OpenSSH è formato da parecchi programmi.

  • sshd(8) - Programma server eseguito sulla macchina server. Quest'ultima aspetta connessioni dalle macchine client, e quando riceve una connessione, autentica e serve il client. Il suo comportamento è controllato dal file di configurazione sshd_config(5).
  • ssh(1) - Questo è il programma client usato per loggarsi su un'altra macchina o per eseguire comandi sull'altra macchina. slogin è un altro nome per questo programma. Il suo comportamento è controllato dal file di configurazione globale ssh_config(5) e dai file $HOME/.ssh/config dei singoli utenti.
  • scp(1) - Copia i file in maniera sicura da una macchina a un'altra.
  • ssh-keygen(1) - Serve a creare chiavi (chiavi host e chiavi di autenticazione) Pubkey Authentication (RSA o DSA).
  • ssh-agent(1) - Agente di autenticazione. Può essere usato per mantenere le chiavi RSA per l'autenticazione.
  • ssh-add(1) - Serve a registrare nuove chiavi con l'agente.
  • sftp-server(8) - Sottosistema di server SFTP.
  • sftp(1) - Programma di trasferimento sicuro di file.
  • ssh-keyscan(1) - Raccoglie le chiavi pubbliche ssh.
  • ssh-keysign(8) - Programma di appoggio di ssh per l'autenticazione basata sull'host.

Download

OpenSSH è disponibile in due versioni scaricabili: la ditribuzione nativa di OpenBSD e la distribuzione Portabile multipiattaforma. Se si vuole OpenSSH per una versione recente di OpenBSD o per integrarlo in un prodotto, probabilmente serve la distribuzione OpenBSD. Se si vuole OpenSSH per un'altra piattaforma, o una vecchia versione di OpenBSD, probabilmente serve la distribuzione Portabile.

Per scaricare, si prega di utilizzare un mirror vicino.

1.2 - Perché usarlo?

OpenSSH è una serie di strumenti che aiutano a rendere più sicure le connessioni di rete. Ecco un elenco di funzionalità:

  • Autenticazione forte. Chiude parecchi buchi di sicurezza (per esempio IP, routing e DNS spoofing).
  • Maggior privacy. Tutte le comunicazioni sono crittate automaticamente e in maniera trasparente.
  • Sessioni X11 sicure. Il programma imposta automaticamente DISPLAY sulla macchina server, e inoltra tutte le connessioni X11 sul canale sicuro.
  • Porte TCP/IP arbitrarie possono essere redirette attraverso il canale crittato in entrambe le direzioni (per esempio per transazioni e-cash).
  • Non è necessaria nessuna formazione aggiuntiva per i normali utenti.
  • Non si fida mai della rete. Fiducia minima verso l'altra parte della connessione. Minima fiducia verso i server dei nomi. L'autenticazione RSA pura non si fida mai di niente'altro che della chiave privata.
  • Il client autentica, tramite RSA, la macchina server all'inizio di ogni connessione per prevenire cavalli di Troia (tramite routing o spoofing del DNS) e attacchi man-in-the-middle, e il server autentica, tramite RSA, la macchina client prima di accettare l'autenticazione tramite i file .rhosts o /etc/hosts.equiv (per prevenire lo spoofing di DNS, routing, o IP).
  • La distribuzione delle chiavi per l'autenticazione degli host può essere centralizzata dall'amministrazione, automaticamente alla prima connessione verso una macchina.
  • Qualunque utente può creare un numero qualsiasi di chiavi RSA per l'autenticazione dell'utente per suo uso personale.
  • Il programma server rigenera la propria chiave RSA automaticamente ogni ora.
  • Un agente di autenticazione, eseguito sul portatile o la workstation dell'utente, può essere usato per mantenere le chiavi RSA di autenticazione dell'utente.
  • Il software può essere installato e utilizzato (con meno funzionalità) anche senza privilegi di root.
  • Il client è configurabile tramite file di configurazione sia a livello di sistema sia a livello di utente.
  • Compressione opzionale di tutti i dati con gzip (comprese le connessioni X11 forwardate e i dati delle porteTCP/IP), accelerando significativamente le connessioni lente.
  • Sostituzione completa di rlogin, rsh e rcp.

Attualmente, quasi tutte le comunicazioni nelle reti di computer non sono crittate. Di conseguenza, chiunque abbia accesso a qualsiasi macchina connessa alla rete può intercettare qualunque comunicazione. Questo fanno hacker, amministratori curiosi, datori di lavoro, criminali, spie industriali e governi. Alcune reti emanano abbastanza radiazioni elettromagnetiche da rendere i dati catturabili perfino a distanza.

Durante il login, la tua password passa per la rete in chiaro. In questo modo, chiunque stia in ascolto può usare il tuo account per effettuare atti illeciti. Sono numerosi, in tutto il mondo, gli incidenti in cui dei "cracker" lanciano sulle workstation, all'insaputa del proprietario, programmi in ascolto sulla rete per raccogliere le password. Programmi per fare questo sono disponibili su internet, o possono essere sviluppati da un programmatore competente in poche ore.

Le imprese hanno segreti industriali, richieste di brevetti in fase di preparazione, informazioni sui prezzi, sui fornitori, dati dei clienti, del personale, informazioni finanziarie ecc. Attualmente, chiunque abbia accesso alla rete (qualunque macchina sulla rete) può ascoltare tutto il traffico di rete, a dispetto di ogni restrizione di accesso.

Molte compagnie non si rendono conto che le informazioni sono facilmente ricavabili dalla rete. Confidano che i loro dati siano sicuri perché nessuno dovrebbe sapere che ci sono informazioni sensibili sulla rete, o perché motli altri dati viaggiano sulla rete. Questa non è una policy sicura.

1.4 - E per quanto riguarda diritto d'autore, uso e brevetti?

Gli sviluppatori di OpenSSH hanno fatto di tutto per mantenere OpenSSH libero da ogni problema di brevetti o copyright. Per questo motivo, alcune opzioni hanno dovuto essere tolte da OpenSSH. Per la precisione il supporto di algoritmi brevettati.

OpenSSH non supporta alcun algoritmo di trasporto brevettato. In modalitá SSH1, le uniche opzioni disponibili sono 3DES e Blowfish. In modalitá SSH2 la scelta è fra 3DES, Blowfish, CAST128, Arcfour e AES. L'algoritmo brevettato IDEA non è supportato.

OpenSSH supporta entrambi i protocolli SSH1 e SSH2.

Essendo scaduto il brevetto dell'algoritmo RSA, non ci sono restrizioni nell'utilizzo di questo algoritmo per il software, compreso OpenBSD.

2.0 - Domande Generali

2.1 - Perché ssh/scp stabilisce connessioni su porte privilegiate.

Il client OpenSSH usa porte privilegiate per l'autenticazione rhosts e rhosts-rsa perché il server deve potersi fidare dello username fornito dal client. Per evitarlo, puoi aggiungere questo esempio al tuo file ssh_config o ~/.ssh/config.

UsePrivilegedPort no

O puoi specificare ques'opzione dalla riga di comando, usando l'opzione -o del comando ssh(1).

$ ssh -o "UsePrivilegedPort no" host.com

2.2 - Perché il client ssh è setuid root?

Riprendendo la domanda precedente, (2.1) OpenSSH ha bisogno dei privilegi di root per poter fare il binding a porte privilegiate per facilitare l'autenticazion rhosts. Vecchie versioni di SSH richiedono una porta privilegiata anche per l'autenticazione rhosts-rsa.

In aggiunta, sia per l'autenticazione rhosts-rsa (nel protocollo versione 1) sia per l'autenticazione basata sull'host (nel protocollo versione 2), il client ssh deve accedere alla chiave privata dell'host per autenticare la macchina client presso il server. Le versioni di OpenSSH precedenti alla 3.3 richiedevano che il binario ssh fosse setuid root per permettere questo, ma è possibile toglierlo se non si vogliono usare questi metodi di autenticazione.

A partire da OpenSSH 3.3, ssh non è setuid di default. Viene utilizzato ssh-keysign per accedere alle chiavi private dell'host, e ssh non usa porte sorgenti privilegiate di default. Se si desidera usare una porta sorgente privilegiata, occorre settare manualmente il bit setuid su ssh.

2.3 - Perché SSH 2.3 ha problemi d'interoperabilitá con OpenSSH 2.1.1?

SSH 2.3 e le versioni precedenti contengono un errore nella loro implementazione di HMAC. Il loro codice non forniva l'intero blocco di dati risultante dal digest, bensì restituiva sempre 128 bit. Per digest più lunghi, questo impediva ad SSH 2.3 di interoperare con OpenSSH.

OpenSSH 2.2.0 riconosce che SSH 2.3 ha questo difetto. Le versioni recenti di SSH correggeranno questo difetto. In alternativa puoi aggiungere la seguente riga al file sshd2_config di SSH 2.3.

Mac hmac-md5

2.4 - Perché OpenSSH stampa a video: Dispatch protocol error: type 20

Ci sono stati problemi di interoperabilità perché vecchie versioni di OpenSSH non supportavano la rinegoziazione delle chiavi di sessione. Comunque la versione commerciale di SSH 2.3 cerca di negoziare questa funzionalità, e potresti vedere la sessione sospesa o il messaggio di errore "Dispatch protocol error: type 20 ". Per risolvere questo problema, aggiorna la tua versione di OpenSSH o disabilita la rinegoziazione della chiave di sessione, aggiungendo la seguente riga al file ssh2_config o sshd2_config della tua versione commerciale di SSH 2.3.

RekeyIntervalSeconds 0

2.5 - Vecchie versioni commerciali di SSH crittano le chiavi host con IDEA.

Le vecchie versioni di SSH usavano un algoritmo brevettato per crittare il loro file /etc/ssh/ssh_host_key. Questo problema si manifesta quando sshd(8) non riesce a leggere la sua chiave host. Per risolvere questo problema, usa il seguente comando per convertire la tua ssh_host_key in 3DES. NB: Usa il programma ssh-keygen(1) del prodotto commerciale SSH, *NON* OpenSSH per l'esempio seguente.

# ssh-keygen -u -f /etc/ssh/ssh_host_key

2.6 - Cosa sono quei messaggi di allarme sulla lunghezza della chiave?

Il programma ssh-keygen(1) della versione commerciale di SSH conteneva un bug che, a volte, gli faceva generare chiavi Pubkey Authentication (RSA o DSA) con il bit più significativo (Most Significant Bit, MSB) non impostato. Queste chiavi erano passate per chiavi di lunghezza intera, ma di fatto sono, una volta su due, più piccole.

OpenSSH stamperà a video messaggi di allarme incontrando chiavi di questo tipo. Per sbarazzarsi di questi messaggi, apri i tuoi file known_hosts files e sostituisci la lunghezza scorretta della chiave (di solito "1024") con la lunghezza corretta (di solito "1023").

2.7 - Il forwarding dell'X11 e/o dell'agente non funziona.

Verifica i tuoi file ssh_config e sshd_config. I file di configurazione di default disabilitano l'agente di autenticazione e il forwarding dell'X11. Per abilitarli, inserisci la seguente riga in sshd_config:

X11Forwarding yes

e inserisci le seguenti righe in ssh_config:

ForwardAgent yes
ForwardX11 yes

NB: Per utenti con Linux Mandrake 7.2, Mandrake modifica la variabile d'ambiente XAUTHORITY in /etc/skel/.bashrc, e quindi nella directory home di ogni utente che usa la bash. Questa variabile viene impostata da OpenSSH e per far funzionare entrambe le opzioni di cui sopra, devi commentare la riga:

# export XAUTHORITY=$HOME/.Xauthority

2.8 - Dopo l'upgrade di OpenSSH ho perso il supporto per SSH2.

Fra una versione e l'altra possono essere fatte modifiche ai file sshd_config o ssh_config. Dovresti sempre verificare queste modifiche aggiornando OpenSSH. Dopo OpenSSH versione 2.3.0 devi aggiungere le seguenti righe al tuo file sshd_config

HostKey /etc/ssh_host_dsa_key
HostKey /etc/ssh_host_rsa_key

2.9 - sftp/scp non riescono a connettersi, ma ssh funziona.

sftp e/o scp possono fallire durante la connessione se hai dei file di inizializzazione della shell (.profile, .bashrc, .cshrc, etc) che producono un output per le sessioni non interattive. Questo output confonde il client sftp/scp. Puoi verificare se la tua shell lo fa eseguendo:

ssh yourhost /usr/bin/true

Se questo comando produce un output devi modificare i tuoi file di inizializzazione della shell.

2.10 - Aggiungerete la funzione [taldeitali] a scp?

Risposta breve: no.

Risposta lunga: scp non è un comando standard. La specifica che più ci si avvicina è "quello che fa rcp". Siccome lo stesso comando è usato da entrambe le parti della connessione, aggiungere funzionalità o opzioni rischia di impedire l'interoperabilità con altre implementazioni.

Nuove funzionalità sono più probabili in sftp, dato che il protocollo è standardizzato (insomma, un draft standard), e il client e il server sono indipendenti.

2.11 - Come si usa il port forwarding?

Se sul server remoto è in esecuzione sshd(8), è possibile far passare certi servizi in un "tunnel" ssh. Potrebbe essere auspicabile, per esempio, per crittare le connessioni POP o SMTP, anche se il software non supporta direttamente le comunicazioni crittate. Il tunneling utilizza il port forwarding per creare una connessione fra il client e il server. Il software client dev'essere in grado di specificare una porta non standard a cui connettersi perché questo funzioni.

L'idea è che l'utente si colleghi all'host remoto usando ssh, e specifichi quale porta sulla mecchina client debba essere usata per inoltrare le connessioni al server remoto. Dopo questo è possibile avviare il servizio da crittare (per es. fetchmail, irc) sulla macchina client, specificando la stessa porta locale passata ad ssh, e la connessione avverrà in un tunnel ssh.

Le opzioni principali per il tunneling sono -L e -R, che consentono all'utente di inoltrare le connessioni, l'opzione -D, che permette il port forwarding dinamico, l'opzione -f, che indica a ssh di girare in background. Vd. sotto per ulteriori dettagli su queste opzioni.

Questo è un esempio di tunnel di una sessione IRC dalla macchina client "127.0.0.1" (localhost) al server remoto "server.example.com":

ssh -f -L 1234:server.example.com:6667 server.example.com sleep 10
irc -c '#users' -p 1234 pinky 127.0.0.1

Questo stabilisce un tunnel per la connessione al server IRC server.example.com, associandosi al canale "#users", usando il nickname "pinky". La porta locale utilizzata è la 1234. Non importa quale porta sia utilizzata, purché sia superiore alla 1023 (ricorda che solo root può aprire socket su porte privilegiate) e non sia in conflitto con porte già in uso. La connessione viene inoltrata alla porta 6667 sul server remoto, essendo quella la porta standard per i servizi IRC.

Il comando remoto "sleep 10" serve per dare il tempo (10 secondi, nell'esempio) di avviare il servizio che deve passare attraverso il tunnel. Se non avvengono connessioni entro il tempo specificato, ssh uscirà. Se occorre più tempo, il valore di sleep(1) può essere aumentato opportunamente o, in alternativa, l'esempio di cui sopra può essere aggiunto come funzione alla shell dell'utente. Vd. ksh(1) e csh(1) per maggiori dettagli sulle funzioni definite dagli utenti.

ssh ha anche un'opzione -N, utile da abbinare al port forwarding: se specificata, non è necessario specificare un comando remoto ("sleep 10" nell'esempio di cui sopra). Però, usando quest'opzione, ssh rimane in attesa indefinitamente (anziché uscire dopo il completamento del comando remoto), e l'utente deve prendersi cura, in un secondo tempo, di fermare manualmente il processo.

2.12 - La mia connessione ssh si blocca o cade dopo N minuti d'inattività.

Di solito è la conseguenza di un time out, da parte di un firewall o un device di NAT, della tua connessione TCP inattiva. Puoi abilitare ClientAliveInterval nel file sshd_config del server, o abilitare ServerAliveInterval nel file ssh_config del client (quest'ultimo è disponibile in OpenSSH 3.8 o successivo).

Abilitare una delle due opzioni e impostare l'intervallo su un tempo minore di quello di time out della tua sessione assicura che la connessione sia mantenuta attiva nella tavola delle connessioni del device.

2.13 - Come si copia una file con "due punti" nel nome con scp?

scp interpreta la parte prima dei due punti come nome del server remoto e cerca di connettersi ad esso. Per evitare questo, indica il file con il percorso relativo o assoluto, per es:
$ scp ./source:file sshserver:

3.0 - Domande su Portable OpenSSH

3.1 - Falsi messaggi di autenticazione PAM nei log.

La versione portabile di OpenSSH genera falsi messaggi di autenticazione fallita ad ogni login, del tipo:

"authentication failure; (uid=0) -> root for sshd service"

Essi vengono generati perché OpenSSH prima cerca di determinare se un utente deve autenticarsi al login (per es. password vuota). Purtroppo PAM vuole loggare tutti gli eventi di autenticazione, incluso questo.

Se ti dà troppo fastidio, imposta "PermitEmptyPasswords no" in sshd_config. Questo zittirà il messaggio di errore al costo di disabilitare i login ad account senza password. Questo è il default se usi il file sshd_config fornito.

3.2 - Empty passwords not allowed with PAM authentication.

Per abilitare le password vuote in una versione di OpenSSH con PAM, devi aggiungere il flag nullok alla fine del modulo di verifica della password nel file /etc/pam.d/sshd. Ad esempio:

auth required/lib/security/pam_unix.so shadow nodelay nullok

Oltre a questo, occorre impostare "PermitEmptyPasswords yes" nel file sshd_config.

Occorre fare attenzione a una cosa se si usano password vuote con l'autenticazione PAM: PAM accetterà qualunque password autenticando un account con password vuota. Questo ostacola il controllo che sshd(8) effettua per determinare se un account è senza password e consentire l'accesso agli utenti, qualunque sia la policy specificata da PermitEmptyPasswords. Per questo motivo, si consiglia di non aggiungere la direttiva nullok al file di configurazione di PAM, a meno che non si desideri effettivamente consentire le password vuote.

3.3 - ssh(1) impiega tanto tempo per connettersi o loggarsi.

Lunghi ritardi (più di 10 secondi) sono tipicamente causati da un problema di risoluzione dei nomi:

  • Alcune versioni di glibc (notamente glibc 2.1 che fa parte di Red Hat 6.1) può metterci parecchio a risolvere indirizzi "IPv6 o IPv4" dai nomi di dominio. Questo può essere ovviato specificando l'opzione AddressFamily inet in ssh_config.
  • Ci può essere un problema di ricerca DNS, sia dal lato client sia dal lato server. Puoi fare una verifica usando il comando nslookup su client e server per risolvere il nome e indirizzo reciproci. In aggiunta, prova a risolvere sul server il nome ritornato dalle ricerche IP-nome sul client. Puoi disabilitare la maggior parte delle ricerche lato server impostando UseDNS no in sshd_config.

Ritardi inferiori a 10 secondi possono avere altre cause.

  • Le versioni di openSSH precedenti la 3.8 avevano un file moduli con moduli più piccoli di quanto sshd si aspettasse, e di conseguenza, sshd finiva per utilizzare moduli molto più grandi del necessario, con perdita di velocit&agrve;. Sostituire il file moduli risolve questo problema (nella maggior parte dei casi questo file non viene sostituito durante un upgrade e dev'essere copiato a mano).
  • Le versioni di OpenSSH precedenti la 3.8 avevano un difetto in ssh che gli faceva richiedere moduli più grandi del necessario (il che, combinato col punto precedente, causava notevoli rallentamenti). Aggiornare il client alla versione 3.8 o superiore risolve questo problema.
  • Se il client o il server non ha un device per la generazione di numeri casuali (come per esempio Solaris < 9, AIX < 5.2, HP-UX < 11.11) e nessun sostituto è disponibile (per esempio prngd) è possibile che uno dei programmi richiamati da ssh-rand-helper per generare entropia rimanga appeso. Lo si può appurare eseguendolo in modalità debug:
    /usr/local/libexec/ssh-rand-helper -vvv
    Si dovrebbe cercare la causa di ogni ritardo significativo e porvi rimedio, o i comandi corrispondenti dovrebbero essere rimossi da ssh_prng_cmds.

Quanto è lento "lento"?

In condizioni normali, la velocità dei login SSH dipende dalla velocità della CPU del client e del server. Per farsi un' idea, questi sono tipici tempi di connessione per time ssh localhost true con una chiave RSA a 1024-bit su host scariche.OpenSSH e OpenSSL sono state compilate con gcc 3.3.x.

3.4 - "Messagi "Can't locate module net-pf-10" nei log con Linux.

Il kernel di linux sta cercando (con modprobe) la famiglia di protocolli 10 (IPv6). Carica il modulo appropriato e inserisci l'alias corretto in /etc/modules.conf o disabilita IPv6 in /etc/modules.conf.

Per qualche strano motivo il file /etc/modules.conf può anche chiamarsi /etc/conf.modules.

3.5 - L'autenticazione della password non funziona con Slackware 7.0 o Red Hat Linux 6.x

Per Slackware 7.0, occorre linkare OpenSSH a libcrypt.

LIBS=-lcrypt ./configure [options]

Certe versioni di Red Hat hanno degli strumenti per le password che creano password in stile MD5, ma la funzione crypt() di sistema non le capisce. Se l'autenticazione con password fallisce con i nuovi account o account con password cambiate di recente, ma funziona con i vecchi account, probabilmente questa è la causa. (Le password coinvolte avranno stringhe di password in /etc/shadow che cominciano con $1$).

per risolvere questo problema, abilita il supporto MD5 in sshd durante la compilazione:

./configure --with-md5-passwords [options]

3.6 - Configure o sshd(8) si lamentano perché RSA non è supportato

Assicurati che le librerie di OpenSSL siano state compilate includendo il supporto per RSA o DSA internamente o tramite RSAref.

3.7 - errori "scp: command not found"

scp(1) deve essere nel PATH di default sia sul client che sul server. Sul server puoi usare l'opzione --with-default-path per specificare un percorso di ricerca particolare. Quest'opzione sostituisce il percorso di default, per cui devi specificare tutte le directory attualmente nel tuo path e dove hai installato scp. Per esempio:

$ ./configure --with-default-path=/bin:/usr/bin:/usr/local/bin:/path/to/scp

La configurazione dell'amministratore del server ha precedenza sul valore di --with-default-path. Questo include reimpostare il PATH in /etc/profile, in /etc/environment su AIX, o (per le versioni 3.7p1 e superiori) impostrare PATH o SUPATH in /etc/default/login su Solaris o Reliant Unix.

3.8 - Non riesce a leggere la passphrase

Alcuni sistemi operativi impostano /dev/tty con permessi scorretti, causando il fallimento della lettura della password con il seguente errore:

You have no controlling tty. Cannot read passphrase.

La soluzione consiste nell'impostare i permessi di /dev/tty su 0666 e riferire l'errore al venditore del sistema operativo.

3.9 - Manca il file 'configure' o fallisce il make

Se non c'è nessun file 'configure' nel tar.gz scaricato o il make fallisce con errori "missing separator", probabilmente hai scaricato la distribuzione di OpenSSH per OpenBSD e stai cercando di compilarlo per un'altra piattaforma. Consulta la pagina relativa alla versione portabile.

3.10 - La connessione rimane sospesa uscendo da ssh

OpenSSH può rimanere appeso uscendo. Questo capita quando c'è un processo attivo in background. Succede su Linux e HP-UX. Il problema può essere verificato in questo modo:

$ sleep 20 & exit
Prova invece:
$ sleep 20 < /dev/null > /dev/null 2>&1 &

un rimedioper chi usa la bash è inserire "shopt -s huponexit" nel file /etc/bashrc o ~/.bashrc. Altrimenti, cerca, nella man page della tua shell, l'opzione per mandare un segnale HUP ai processi attivi, uscendo. Vedi bug #52 per altre soluzioni.

3.11 - Perché la connessione ssh rimane sospesa uscendo?

Eseguendo

$ ssh host command
ssh deve restare appeso, perché deve aspettare:
  • di essere sicuro che command non abbia bisogno di altro input.
  • di essere sicuro che command non produca altro output.
  • che command esca perché sshd deve dire l'exit status di command a ssh.

3.12 - Ho fatto l'upgrade a OpenSSH 3.1 e il forwarding dell'X11 non funziona più.

A partire da OpenSSH 3.1, il server di forwarding dell'X11 di sshd ascolta su localhost per default; vedi l'opzione X11UseLocalhost di sshd per ripristinare il comportamento precedente se i tuoi vecchi client X11 non funzionano con questa configurazione.

In genere, client X11 che usano X11 R6 dovrebbero funzionare con l'impostazione di default. Alcuni venditori, fra cui HP, vendono client X11 con librerie R6 e R5, così alcuni client funzioneranno, e altri no. Questo vale per HP-UX 11.X.

3.13 - Ho fatto l'upgrade a OpenSSH 3.8 e alcuni programmi X11 non funzionano più.

Come documentato nelle 3.8 release notes, ssh userà ora per default cookies X11 untrusted. Il comportamento di prima può essere ripristinato impostando ForwardX11Trusted yes in ssh_config.

I possibili sintomi includono:
BadWindow (invalid Window parameter)
BadAccess (attempt to access private resource denied)
X Error of failed request: BadAtom (invalid Atom parameter)
Major opcode of failed request: 20 (X_GetProperty)

3.14 - Ho copiato la mia chiave pubblica fra le 'authorized_keys' ma l'autenticazione tramite chiave pubblica non funziona ancora.

Tipicamente questo è dovuto a permessi su $HOME, $HOME/.ssh o $HOME/.ssh/authorized_keys più permissivi di quanto sshd consenta per default.

In questo caso, si può risolvere eseguendo il seguente comando sul server.

$ chmod go-w $HOME $HOME/.ssh
$ chmod 600 $HOME/.ssh/authorized_keys

Se non è possibile per qualche motivo, un'alternativa consiste nell'impostare StrictModes no in sshd_config, anche se è sconsigliato.
