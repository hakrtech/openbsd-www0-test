Comment soumettre un problème

Soumission de problèmes concernant les versions publiées.

Avant de soumettre des bogues/problèmes rencontrés avec les versions publiées d'OpenSSH, prenez le temps de suivre cette liste de contrôle :
  1. Vérifiez les correctifs et notes concernant OpenSSH.
  2. Voyez s'il n'existe pas de version plus récente pour OpenBSD ou pour Portable OpenSSH, selon votre plate-forme.
  3. Consultez FAQ pour les problèmes fréquemment signalés comme étant des bogues mais qui ne le sont pas.
  4. Consultez la liste des bogues corrigés depuis la mise à disposition de la dernière version et les bogues connus signalés depuis la mise à disposition de la dernière version.

Si rien ne semble résoudre le problème, utilisez Bugzilla pour soumettre un rapport de bogues (voir ci-dessous pour soumettre des bogues de sécurité de manière privée).

Vous pouvez joindre les développeurs OpenSSH en envoyant un message à l'une des adresses suivantes :

  • openssh@openssh.com. Ceci est une liste privée uniquement lue par les développeurs OpenSSH.
    Appropriée pour : soumission de problèmes avec la version OpenSSH d'OpenBSD, et les problèmes de sécurité sur n'importe quelle plate-forme.
    Non appropriée pour : demandes du type "comment faire ceci ou cela", soumission de problèmes pour d'autres plates-formes qu'OpenBSD ou avec des correctifs fournis par des tierces parties, discussion générale.
  • openssh-unix-dev@mindrot.org C'est une liste publique ouverte aux envois de messages expédiés par des utilisateurs non inscrits.
    Appropriée pour : soumission de problèmes, questions de type "howto" et discussions autour d'OpenSSH sur n'importe quelle plate-forme (y compris OpenBSD).
    Non appropriée pour : rapports de bogues sécurité, rapports de bogues concernant des binaires non fournis par l'équipe OpenSSH (sauf s'ils peuvent être reproduits sur la version standard d'OpenSSH)

Soumission de problèmes concernant la version de développpement.

  1. Testez le problème au moins deux fois, en mettant à jour vos sources à quelques jours d'intervalle.
  2. Ne soumettez pas des problèmes de compilation de l'arborescence des sources, sauf s'ils persistent. Ils sont pratiquement tout le temps dûs à une mauvaise manipulation de votre part sinon ils doivent entrain d'être réglés quand vous les avez rencontré.
  3. Rappelez vous que les serveurs anoncvs sont mis à jour de manière significative pour reflèter l'arborescence des sources actuelle.
  4. Vérifiez les Changements au niveau d'OpenSSH pour voir si le problème a été résolu.

Envoi de rapports de bogues.

Essayez d'indiquer le plus précisemment possible le problème exact. Ne fournissez jamais de vagues instructions, ou détaillez des problèmes vagues tels que "il crashe." Parlez en à d'autres sur IRC ou utilisez un autre forum tel que les archives des listes de diffusion pour confirmer que le problème est nouveau, qu'il est reproduisible, etc. et assurez vous que ce n'est pas un problème local.

Pour une excellente vue d'ensemble de l'art de soumettre des rapports de bogues, veuillez consultez le document de Simon Tatham intitulé How to Report Bugs Effectively. Gardez à l'esprit que meilleure est l'information contenue dans votre rapport, plus grandes seront les chances pour que votre problème soit résolu.

Les nouvelles fonctionnalités sont acceptées, particulièrement si le code permettant de les implémenter est présent. Si quelqu'un d'autre écrit le code pour implémenter votre nouvelle fonctionnalité, il y a des chances pour une qu'une incompréhension se glisse et que vous ne reconnaissiez pas la dite fonctionnalité.

Les types de rapports de bogues par ordre de souhait :

  1. Les problèmes reproduisibles avec des correctifs pour les sources sont les meilleurs.
  2. Les problèmes reproduisibles non spécifiques à votre agencement matériel/logiciel.
  3. Les problèmes reproduisibles non spécifiques à votre agencement logiciel.
  4. Les problèmes reproduisibles non spécifiques à votre agencement matériel.
