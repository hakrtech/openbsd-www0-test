Portable OpenSSH

De normale ontwikkeling van OpenSSH produceert een zeer kleine, veilige en gemakkelijk onderhoudbare versie voor het OpenBSD project. Het OpenSSH Portabiliteitsteam voegt portabiliteitscode aan die zuivere versie toe zodat OpenSSH op veel besturingssystemen kan draaien (Helaas moeten zij met *veel* verschillen tussen Unix besturingssystemen rekening houden, niet in de laatste plaats omdat OpenSSH een vorm van authenticatie bevat).

De OpenSSH die hier wordt aangeboden, is ontworpen om op de volgende Unix besturingssystemen te draaien:

  • AIX
  • HP-UX
  • Irix
  • Linux
  • NeXT
  • SCO
  • SNI/Reliant Unix
  • Solaris
  • Digital Unix/Tru64/OSF
  • Mac OS X
  • Cygwin
  • ... en er komen steeds meer bij.

De portable OpenSSH volgt de ontwikkeling van de officiële versie, maar ze worden niet tegelijkertijd uitgebracht. Portable versies worden aangeduid met een 'p' (b.v. 4.2p1). De officiële OpenBSD broncode zal nooit het achtervoegsel 'p' gebruiken, maar zal de versienummer verhogen wanneer ze 'stabiele punten' bereiken in hun ontwikkeling.

De volgende bestanden beschrijven de ontwikkelingen van het OpenSSH portabiliteitsontwikkelteam. De bestanden zijn gesigneerd met de PGP publieke sleutel in het bestand DJM-GPG-KEY.asc op de ftp-site. Deze sleutel is ook beschikbaar via het key server netwerk.

  • Portabiliteits ChangeLog
  • Installatie-instructies
  • FAQ
  • Dagelijkse snapshots worden beschikbaar gesteld in de snapshots/ directory op de mirror sites

Ontwikkelaars

Elke nacht worden bleeding-edge snapshots beschikbaar gesteld in de snapshots/ directory op de mirror sites. De portable versie van OpenSSH is ook beschikbaar via anonieme CVS. Om deze te verkrijgen moet u de volgende commando's uitvoeren: 

export CVSROOT=openssh@anoncvs.at.openbsd.org:/cvs
export CVS_RSH=/usr/bin/ssh
cvs get openssh
(Er zit geen wachtwoord op deze opslagplaats. Ja, het is alleen-lezen, probeer dus geen wijzigingen aan te brengen). Deze opslagplaats wordt voorzien en onderhouden door Wim Vandeputte.

Ontwikkelaars worden gevraagd om deze richtlijnen te lezen en toe te passen aan hun code. Patches moeten worden opgestuurd in unified diff formaat.
