Wie man einen Bug in OpenSSH meldet

Probleme mit veröffentlichten Versionen.

Bevor du Bugs/Probleme mit veröffentlichten OpenSSH Versionen meldest, gehe diese Checkliste durch:
  1. Überprüfe, ob es patches und Hinweise bezüglich OpenSSH gibt.
  2. Finde heraus, ob es eine neuere Version für OpenBSD oder für die portable release, abhängig von deiner Plattform, gibt.
  3. Überprüfe die FAQ nach Problemen, die oft als Bugs gemeldet werden, aber keine sind.
  4. Überprüfe die Liste der bereits beseitigten Bugs seit der letzten Version und die bekannten Bugs, die seit der letzten Version gemeldet wurden.

Wenn es scheinbar nichts gibt, was dein Problem beheben könnte, benutze Bugzilla um einen Bug Peport zu verschicken (siehe unten, wenn du die Bugs privat senden willst).

Du kannst die OpenSSH Entwickler erreichen, indem du eine Email an eine der folgenden Adressen sendest:

  • openssh@openssh.com. Dies ist eine private Liste, die nur von den OpenSSH Entwicklern gelesen wird.
    Passend zu: Problemberichte zu OpenBSDs OpenSSH oder Sicherheitsprobleme auf jeglicher Plattform.
    Unpassend für: HOWTO-Fragen, Problemberichte für andere Plattformen als OpenBSD oder mit Patchen dritter Seite, allgemeine Diskussion.
  • openssh-unix-dev@mindrot.org Dies ist eine öffentliche Liste und man kann an diese Liste auch senden, wenn man sie nicht abonniert hat.
    Passend für: Fehlerberichte, HOWTO-Fragen und Diskussionen über OpenSSH auf allen Plattformen (einschließlich OpenBSD).
    Nicht passend für: Berichte von Sicherheitsproblemen, Fehlerberichte für Binärdateien, die nicht vom OpenSSH Team angeboten werden oder erzeugt wurden (es sei denn, man kann die Fehler mit einer Standard-OpenSSH Distribution nachvollziehen).

Probleme mit momentanen Versionen.

  1. Teste das Problem mindestens zweimal mit Quellcode, der vor höchstens ein paar Tagen aktualisiert wurde.
  2. Berichte nicht über Kompilierprobleme, es sei denn sie bleiben über einige Zeit bestehen. Sie sind meistens dein Fehler oder es wurde bereits an ihnen gearbeitet, bevor du sie entdeckt hast.
  3. Denk daran, dass die anoncvs Server deutlich hinter der tatsächlichen Entwicklungsarbeit hinterherhinken.
  4. Checke OpenSSH Änderungen, um zu prüfen, ob sich des Problems bereits jemand annimmt.

Bug Reports einschicken.

Versuche, das exakte Problem festzunageln. Gib niemals vage Instruktionen oder Details wie z.B. ,es stürzt ab.' Sprich über IRC mit anderen oder benutze ein anderes Forum wie etwa Mailinglistenarchive um sicherzustellen, dass es etwas neues, reproduzierbares und kein lokales Problem ist.

Einen exzellenten Überblick über die Kunst Fehler zu melden bietet Simon Tathams How to Report Bugs Effectively. Denke daran, je besser die Informationen in deinem Bericht sind, desto größer ist die Chance, dass dein Problem auch beseitigt wird.

Neue Funktionen werden akzeptiert, insbesondere mit Code, der deine neuen Funktionalitäten implementiert. Wenn jemand anderes den Code für dein neues Feature schreibt, sind die Chancen groß, dass derjenige dich falsch versteht und du das nicht einmal bemerkst.

Bug Report-Typen in der Reihenfolge ihrer Beliebtheit:

  1. Reproduzierbare Probleme mit Source Fixes sind die besten.
  2. Reproduzierbare Probleme, die nicht spezifisch für dein Hardware-/Software-Layout sind.
  3. Reproduzierbare Probleme, die spezifisch für dein Software Layout sind.
  4. Reproduzierbare Probleme, die spezifisch für dein Hardware Layout sind.
