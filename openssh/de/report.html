Wie man einen Bug in OpenSSH meldet

Probleme mit veröffentlichten Versionen.

Bevor du Bugs/Probleme mit veröffentlichten OpenSSH Versionen meldest, gehe diese Checkliste durch:
  1. Zuerst überprüfe, ob es patches und Hinweise bezüglich OpenSSH gibt.
  2. Als nächstes finde heraus, ob es eine neuere Version für OpenBSD oder für die portable release, abhängig von deiner Plattform, gibt.

Wenn es scheinbar nichts gibt, was dein Problem beheben köte, benutze Bugzilla um einen bug report zu verschicken.

Du kannst die OpenSSH Entwickler direkt erreichen, indem du ihnen eine Email an openssh@openssh.com schickst. Wenn dein Problem die portable Version von OpenSSH betrifft und nicht in Bezug auf Sicherheit steht, benutze anstattdessen bitte die openssh-unix-dev@mindrot.org Mailingliste. Diese Liste läßt auch Mails von Benutzern zu, die dort nicht eingetragen, bzw. eingeschrieben sind.

Probleme mit momentanen Versionen.

  1. Teste das Problem mindestens zweimal, mit Quellcode, der vor höchstens ein paar Tagen upgedatet wurde.
  2. Berichte nicht über Kompilierprobleme, es sei denn sie bleiben über einige Zeit bestehen. Sie sind meistens dein Fehler oder es wird bereits an ihnen gearbeitet, bevor du sie entdeckt hast.
  3. Denk daran, dass die anoncvs Server deutlich hinter der tatsächlichen Entwicklungsarbeit hinterherhinken.
  4. Checke OpenSSH Änderungen, um zu prüfen, ob sich des Problems bereits jemand annimmt.

Bug Reports einschicken.

Versuche, das exakte Problem festzunageln. Gib niemals vage Instruktionen oder Details wie z.B. "es stürzt ab." Sprich über IRC mit anderen oder benutze ein anderes Forum wie etwa Mailinglistenarchive um sicherzustellen, dass es etwas neues und reproduzierbares und kein lokales Problem ist.

Neue Features werden akzeptiert, insbesondere mit Code, der deine neuen Features implementiert. Wenn jemand anderes den Code für dein neues Feature schreibt, sind die Chancen gross, dass derjenige dich falsch versteht und du das nicht einmal bemerkst.

Bug Report-Typen in der Reihenfolge ihre Beliebtheit:

  1. Reproduzierbare Probleme mit Source Fixes sind die besten.
  2. Reproduzierbare Probleme, die nicht spezifisch für dein Hardware-/Software-Layout sind.
  3. Reproduzierbare Probleme, die spezifisch für dein Software Layout sind.
  4. Reproduzierbare Probleme, die spezifisch für dein Hardware Layout sind.
