Wie man über ein Problem berichtet

Probleme mit veröffentlichten Versionen.

Bevor du Bugs/Probleme mit veröffentlichten OpenSSH Versionen meldest, gehe diese Checkliste durch:
  1. Zuerst checke ob es patches und Hinweise bezüglich OpenSSH gibt.
  2. Dann finde heraus ob es eine neuere Version gibt.

Wenn anscheinend nichts davon dein Problem löst, kann es Zeit sein man sendbug zu lesen, bevor du einen Bug Report einschickst.

Du kannst die OpenSSH Entwickler direkt erreichen, indem du ihnen eine Email an openssh@openssh.com schickst.

Probleme mit momentanen Versionen.

  1. Teste das Problem mindestens zweimal, mit Source der vor höchstens ein paar Tagen upgedatet wurde.
  2. Berichte nicht über Kompilierprobleme, es sei denn sie bleiben über einige Zeit bestehen. Sie sind meistens dein Fehler oder es wird bereits an ihnen gearbeitet, bevor du sie entdeckt hast.
  3. Denk daran dass die anoncvs Server deutlich hinter der tatsächlichen Entwicklungsarbeit hinterherhinken.
  4. Checke OpenSSH Changes um zu prüfen, ob sich des Problems bereits jemand annimmt.

Bug Reports einschicken.

Versuche das exakte Problem festzunageln. Gib niemals vage Instruktionen oder Details wie z.B. "es crasht." Sprich mit anderen über IRC oder einem anderen Forum um zu bestätigen, dass es neu, reproduzierbar und so weiter ist, und stelle sicher dass es kein lokales Problem ist.

Neue Features werden akzeptiert, insbesondere mit Code der deine neuen Features implementiert. Wenn jemand anderes den Code für dein neues Feature schreibt, sind die Chancen gross, dass derjenige dich falsch versteht und du das nicht einmal bemerkst.

Bug Report-Typen in der Reihenfolge ihre Beliebtheit:

  1. Reproduzierbare Probleme mit Source Fixes sind die besten.
  2. Reproduzierbare Probleme, die nicht spezifisch für dein Hardware-/Software-Layout sind.
  3. Reproduzierbare Probleme, die spezifisch für dein Software Layout sind.
  4. Reproduzierbare Probleme, die spezifisch für dein Hardware Layout sind.
