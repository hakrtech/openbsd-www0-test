Errata & lista de correções para o OpenBSD 3.3:

Para informações sobre a ramificação de correções.
Para atualizações importantes dos pacotes.

Para a errata de certa versão, clique abaixo:
2.1, 2.2, 2.3, 2.4, 2.5, 2.6, 2.7, 2.8, 2.9, 3.0, 3.1, 3.2, 3.4.
Você pode baixar um arquivo tar.gz contendo todas as seguintes correções. Este arquivo é atualizado diariamente.

As correções abaixo estão disponíveis no CVS através da ramificação de correções OPENBSD_3_3.

Para informações mais detalhadas em como instalar as correções no OpenBSD, por favor consulte o FAQ do OpenBSD.

Todas as arquiteturas

  • 019: CORREÇÃO DE SEGURANÇA: 13 de Março de 2004
    Devido a um problema encontrado na interpretação das regras de Allow/Deny do módulo de acesso do httpd(8), a utilização de endereços IP sem netmask em plataformas "big endian" de 64-bit causa que as regras falhem ao serem examinadas. Isso afeta somente a plataforma sparc64.
    Uma correção via código fonte existe e corrige esse problema.

  • 018: CORREÇÃO DE CONFIANÇA: 8 de Março de 2004
    A pilha TCP/IP do OpenBSD não impunha limites em quantos segmentos TCP fora de ordem eram agendados no sistema. Um atacante poderia enviar segmentos TCP fora de ordem e fazer com que o sistema utilizasse todos os buffers de memória existentes.
    Uma correção via código fonte existe e corrige esse problema.

  • 017: CORREÇÃO DE CONFIANÇA: 14 de Fevereiro de 2004
    Vários buffer overflows existem no código que analiza os arquivos font.aliases no XFree86. Graças ao ProPolice, esses erros não podem ser explorados para obter privilétios, porém podem fazer que o servidor X seja abortado.
    Uma correção via código fonte existe e corrige esse problema.

  • 016: CORREÇÃO DE SEGURANÇA: 8 de Fevereiro de 2004
    Um problema de manipulação de MTU IPv6 existe e pode ser usado pelo atacante para causar um ataque de negação de serviço aos equipamentos com portas IPv6 TCP abertas.
    Uma correção via código fonte existe e corrige esse problema.

  • 015: CORREÇÃO DE SEGURANÇA: 5 de Fevereiro de 2004
    Existe um problema na contabilização de referências na chamada de sistema shmat(2) que pode ser usado pelo atacante para escrever na memória do kernel sob certas circunstâncias.
    Uma correção via código fonte existe e corrige esse problema.
  • 014: CORREÇÃO DE SEGURANÇA: 15 de Janeiro de 2004
    Diversas falhas no gerenciamento de mensagens do isakmpd(8) foram reportadas por Thomas Walpuski. Essas falhas permitem ao atacante apagar SAs arbitrários.
    Uma correção via código fonte existe e corrige estes problemas.

  • 013: CORREÇÃO DE CONFIANÇA: 20 de Novembro de 2003
    Uma verificação imprópria de limites torna possível que um usuário local cause uma pane no sistema passando às funções     semctl(2) e semop(2) certos argumentos.
    Uma correção via código fonte existe e corrige esse problema.

  • 012: CORREÇÃO DE CONFIANÇA: 20 de Novembro de 2003
    É possível que um usuário local cause uma pane no sistema via sysctl(3) utilizando certos argumentos.
    Uma correção via código fonte existe e corrige esse problema.

  • 010: CORREÇÃO DE CONFIANÇA: 4 de Novembro de 2003
    É possível a um usuário do sistema, causar um pânico no sistema executando um binário malicioso com cabeçalho inválido.
    Uma correção via código fonte existe e corrige esse problema.

  • 009: CORREÇÃO DE CONFIANÇA: 29 de Outubro de 2003
    Um usuário com permissão de escrita ao httpd.conf ou a um arquivo .htaccess pode danificar o httpd(8) ou potencialmente executar códigos arbitrários como o usuário www (apesar de que é provável que a ProPolice irá prever a execução deste código).
    Uma correção via código fonte existe e corrige esse problema.

  • 008: CORREÇÃO DE CONFIANÇA: 1 de Outubro de 2003
    É possivel que um usuário local cause um pânico no sistema inundando-o com requisições ARP falsas.
    Uma correção via código fonte existe e corrige esse problema.

  • 007: CORREÇÃO DE SEGURANÇA: 1 de Outubro de 2003
    A utilização de certas codificações ASN.1 ou chaves públicas mal formadas pode permitir ao atacante montar um ataque de negação de serviço conta aplicativos ligados com ssl(3). Isso não afeta o OpenSSH.
    Uma correção via código fonte existe e corrige esse problema.

  • 006: CORREÇÃO DE SEGURANÇA: 24 de Setembro de 2003
    Três casos de acesso potencial a memória livre foram encontrados no pf(4). Ao menos um deles pode ser utilizado para causar um pânico remotamente no pf ativado com regras de scrub.
    Uma correção via código fonte existe e corrige esse problema.

  • 005: CORREÇÃO DE SEGURANÇA: 16 de Setembro de 2003
    Um buffer overflow na análise de endereços no sendmail(8) pode permitir ao atacante obter privilégios de root.
    Uma correção via código fonte existe e corrige esse problema.
    NOTA: esta é a segunda revisão do patch que corrige um problema adicional.

  • 004: CORREÇÃO DE SEGURANÇA: 16 de Setembro de 2003
    Todas as versões do sshd do OpenSSH menores que a 3.7 contém um erro de gerenciamento de buffer. Ainda não se sabe se este bug é ou não explorável.
    Uma correção via código fonte existe e corrige esse problema.
    NOTA: esta é a segunda revisão do patch que corrige um problema adicional.

  • 003: CORREÇÃO DE SEGURANÇA: 10 de Setembro de 2003
    O root pode ser capaz de reduzir o nível de segurança utilizando um overflow de números inteiros quando os limites de semáforos são muito grandes.
    Uma correção via código fonte existe e corrige esse problema.

  • 002: CORREÇÃO DE CONFIANÇA: 20 de Agosto de 2003
    Uma verificação imprópria na chamada de sistema semget(2) pode permitir a um usuário local causar um pânico no kernel.
    Uma correção via código fonte existe e corrige esse problema.

  • 001: CORREÇÃO DE SEGURANÇA: 4 de Agosto de 2003
    Um erro do tipo off-by-one existe na função da biblioteca C realpath(3). Pelo motivo de que esse bug resultou em um comprometimento do root no servidor ftp wu-ftpd é possível que este bug possa permitir ao atacante ganhar privilégios escalados no OpenBSD.
    Uma correção via código fonte existe e corrige esse problema.

    Uma correção via código fonte existe e corrige esse problema.
