Errata & lista de correções para o OpenBSD 3.4:

Para informações sobre a ramificação de correções.
Para atualizações importantes dos pacotes.

Para a errata de certa versão, clique abaixo:
2.1, 2.2, 2.3, 2.4, 2.5, 2.6, 2.7, 2.8, 2.9, 3.0, 3.1, 3.2, 3.3, 3.5.
Você pode baixar um arquivo tar.gz contendo todas as seguintes correções. Este arquivo é atualizado diariamente.

As correções abaixo estão disponíveis no CVS através da ramificação de correções OPENBSD_3_4.

Para informações mais detalhadas em como instalar as correções no OpenBSD, por favor consulte o FAQ do OpenBSD.

Todas as arquiteturas

  • 028: CORREÇÃO DE CONFIANÇA: 26 de Agosto de 2004
    Conforme relatado por Vafa Izadinia, uma bridge(4) com processamento IPsec ativado pode ser quebrada remotamente por um único echo ICMP atravessando-a.
    Uma correção via código fonte existe e corrige esse problema.

  • 027: CORREÇÃO DE CONFIANÇA: 25 de Agosto de 2004
    Melhorada a verificação de erros ICMP para que o impacto de ataques ICMP contra o TCP seja minimizado.
    http://www.ietf.org/internet-drafts/draft-gont-icmp-payload-00.txt
    Uma correção via código fonte existe e corrige esse problema.

  • 026: CORREÇÃO DE CONFIANÇA: 25 de Julho de 2004
    Sob certa carga na rede o kernel pode ficar sem espaço de armazenamento. Isso foi encontrado em um ambiente usando CARP em uma interface VLAN. Inicialmente, esse problema se manifestou como uma falha relacionada a FPU no processo de inicialização do sistema.
    Uma correção via código fonte existe e corrige esse problema.

  • 025: CORREÇÃO DE SEGURANÇA: 12 de Junho de 2004
    Múltiplas vulnerabilidades foran encontradas no httpd(8) / mod_ssl. CAN-2003-0020, CAN-2003-0987, CAN-2004-0488, CAN-2004-0492.
    Uma correção via código fonte existe e corrige esse problema.

  • 024: CORREÇÃO DE SEGURANÇA: 10 de Junho de 2004
    Como descrito por Thomas Walpuski, o isakmpd(8) ainda é vulnerável à remoção de SA não autorizados. Um atacante pode remover túneis IPsec de acordo com sua vontade.
    Uma correção via código fonte existe e corrige esse problema.

  • 023: CORREÇÃO DE SEGURANÇA: 9 de Junho de 2004
    Múltiplas vulnerabilidades remotas foram encontradas no servidor cvs(1) que permitem a um atacante destruir o servidor ou possivelmente executar códigos arbitrários com os mesmos privilégios que o programa servidor CVS.
    Uma correção via código fonte existe e corrige esse problema.

  • 022: CORREÇÂO DE SEGURANÇA: 30 de Maio de 2004
    Uma falha no servidor kdc(8) Kerberos pode resultar que um administrador de um realm Kerberos tenha a habilidade de fingir ser qualquer administrador de qualquer outro realm que já tenha estabelecido uma confiaça de realm cruzada com o seu próprio realm. A falha é causada pela verificação inadequada do campo "transited" em uma requisição Kerberos. Para maiores detalhes veja o anúncio do Heimdal.
    Uma correção via código fonte existe e corrige esse problema.

  • 021: CORREÇÂO DE SEGURANÇA: 20 de Maio 2004
    Um overflow de armazenamento de pilha no servidor cvs(1) foi descoberto e pode ser explorado por clientes que enviem requisições mal formadas a ele permitindo assim que esses clientes executem código arbitrário com os mesmos privilégios do programa servidor CVS.
    Uma correção via código fonte existe e corrige esse problema.

  • 020: CORREÇÂO DE SEGURANÇA: 13 de Maio 2004
    Verificação de overflows de inteiros no procfs. A utilização do procfs não é recomendada.
    Uma correção via código fonte existe e corrige esse problema.

  • 019: CORREÇÃO DE CONFIANÇA: 6 de Maio de 2004
    Responda para os SYN em janela com um ACK limitado em freqüencia.
    Uma correção via código fonte existe e corrige esse problema.

  • 018: CORREÇÃO DE CONFIANÇA: 5 de Maio de 2004
    Controladoras gdt(4) de modelos recentes travam em alta carga.
    Uma correção via código fonte existe e corrige esse problema.

  • 017: CORREÇÃO DE SEGURANÇA: 5 de Maio de 2004
    Problemas de validação de caminho (path) foram encontrados no cvs(1), permitindo que clientes maliciosos criem arquivos fora do repositório, permitindo assim que servidores maliciosos sobreescrevam os arquivos fora da árvore CVS local no cliente e permitindo que clientes façam checkout de arquivos fora do repositório CVS.
    Uma correção via código fonte existe e corrige esse problema.

  • 016: CORREÇÃO DE CONFIANÇA: 17 de Março de 2004
    Uma falta de verificação de referenciação de um ponteiro nulo foi encontrada no ssl(3). Um atacante pode se utilizar dessa falha para causar uma pane em um aplicativo OpenSSL; isso pode também levar a uma negação de serviço.
    Uma correção via código fonte existe e corrige esse problema.

  • 015: CORREÇÃO DE CONFIANÇA: 17 de Março de 2004
    Problemas na validação e dados e processamento de funções do isakmpd(8) foram descobertos. Um atacante pode enviar mensagens ISAKMP mal formadas e fazer com que o isakmpd entre em pane ou entre em loop infinitamente. Essa correção corrige esses problemas e remove alguns vazamentos de memória.
    Uma correção via código fonte existe e corrige esse problema.

  • 014: CORREÇÃO DE SEGURANÇA: 13 de Março de 2004
    Devido a um problema encontrado na interpretação das regras de Allow/Deny do módulo de acesso do httpd(8), a utilização de endereços IP sem netmask em plataformas "big endian" de 64-bit causa que as regras falhem ao serem examinadas. Isso afeta somente a plataforma sparc64.
    Uma correção via código fonte existe e corrige esse problema.

  • 013: CORREÇÃO DE CONFIANÇA: 8 de Março de 2004
    A pilha TCP/IP do OpenBSD não impunha limites em quantos segmentos TCP fora de ordem eram agendados no sistema. Um atacante poderia enviar segmentos TCP fora de ordem e fazer com que o sistema utilizasse todos os buffers de memória existentes.
    Uma correção via código fonte existe e corrige esse problema.

  • 012: CORREÇÃO DE CONFIANÇA: 14 de Fevereiro de 2004
    Vários buffer overflows existem no código que analiza os arquivos font.aliases no XFree86. Graças ao ProPolice, esses erros não podem ser explorados para obter privilétios, porém podem fazer que o servidor X seja abortado.
    Uma correção via código fonte existe e corrige esse problema.

  • 011: CORREÇÃO DE SEGURANÇA: 8 de Fevereiro de 2004
    Um problema de manipulação de MTU IPv6 existe e pode ser usado pelo atacante para causar um ataque de negação de serviço aos equipamentos com portas IPv6 TCP abertas.
    Uma correção via código fonte existe e corrige esse problema.

  • 010: CORREÇÃO DE SEGURANÇA: 5 de Fevereiro de 2004
    Existe um problema na contabilização de referências na chamada de sistema shmat(2) que pode ser usado pelo atacante para escrever na memória do kernel sob certas circunstâncias.
    Uma correção via código fonte existe e corrige esse problema.

  • 009: CORREÇÃO DE SEGURANÇA: 13 de Janeiro de 2004
    Diversas falhas no gerenciamento de mensagens do isakmpd(8) foram reportadas por Thomas Walpuski. Essas falhas permitem ao atacante apagar SAs arbitrários. Este patch também inclui uma correção para um vazamento de um filedescriptor que causava problemas quando um cartão criptográfico era instalado.
    Uma correção via código fonte existe e corrige estes problemas.

  • 008: CORREÇÃO DE CONFIANÇA: 20 de Novembro de 2003
    Uma verificação imprópria de limites torna possível que um usuário local cause uma pane no sistema passando às funções semctl(2) e semop(2) certos argumentos.
    Uma correção via código fonte existe e corrige esse problema.

  • 007: CORREÇÃO DE CONFIANÇA: 20 de Novembro de 2003
    É possível que um usuário local cause uma pane no sistema via sysctl(3) utilizando certos argumentos.
    Uma correção via código fonte existe e corrige esse problema.

  • 005: CORREÇÃO DE CONFIANÇA: 4 de Novembro de 2003
    É possível a um usuário do sistema, causar um pânico no sistema executando um binário malicioso com cabeçalho inválido.
    Uma correção via código fonte existe e corrige esse problema.

  • 004: CORREÇÃO DE CONFIANÇA: 1 de Novembro de 2003
    Um usuário com permissão de escrita ao httpd.conf ou a um arquivo .htaccess pode danificar o httpd(8) ou potencialmente executar códigos arbitrários como o usuário www (apesar de que é provável que a ProPolice irá prever a execução deste código).
    Uma correção via código fonte existe e corrige esse problema.

  • 003: CORREÇÃO DE CONFIANÇA: 1 de Novembro de 2003
    É possivel que um usuário local cause um pânico no sistema inundando-o com requisições ARP falsas.
    Uma correção via código fonte existe e corrige esse problema.

  • 002: CORREÇÃO DE SEGURANÇA: 1 de Novembro de 2003
    A utilização de certas codificações ASN.1 ou chaves públicas mal formadas pode permitir ao atacante montar um ataque de negação de serviço conta aplicativos ligados com ssl(3). Isso não afeta o OpenSSH.
    Uma correção via código fonte existe e corrige esse problema.

  • 001: CORREÇÃO NA DOCUMENTAÇÃO: 1 de Novembro de 2003
    A documentação inserida no CD possui um exemplo incorreto de instalação de pacotes.
    Aonde está escrito:

               # pkg_add ftp://ftp.openbsd.org/pub/OpenBSD/3.4/packages/i386

    Deve-se ler:

               # pkg_add ftp://ftp.openbsd.org/pub/OpenBSD/3.4/packages/i386/

    A / no final é importante. Não disponibilizados arquivos de correção para os documentos impressos em papel.

i386

  • 006: CORREÇÃO DE SEGURANÇA: 17 de Novembro de 2003
    É possível que um usuário local ultrapasse os limites de pilha (stack) no compat_ibcs2(8).
    O ProPolice captura isso, tornando uma potencial escalação de privilégios em um ataque de negação de serviço. A emulação iBCS2 não precisa estar ativada via sysctl(8) para que isso aconteça.
    Uma correção via código fonte existe e corrige esse problema.
