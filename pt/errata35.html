Para informações mais detalhadas em como instalar as correções no OpenBSD, por favor consulte o FAQ do OpenBSD.

Todas as arquiteturas

  • 025: CORREÇÃO DE CONFIANÇA, 6 de Janeiro de 2005
    A biblioteca de função getcwd(3) contém um erro de gerenciamento de memória que pode causar falha ao tentar obter o diretório de trabalho atual caso o caminho seja muito longo.
    Uma correção via código fonte existe e corrige esse problema.

  • 024: CORREÇÃO DE SEGURANÇA: 14 de Dezembro de 2004
    Em sistema rodando isakmpd(8) é possível um usuário local causar uma corrupção de memória no kernel e pânico no sistema ajustando credenciais ipsec(4) em um socket.
    Uma correção via código fonte existe e corrige esse problema.

  • 023: CORREÇÃO DE CONFIANÇA, 10 de Novembro de 2004
    Devido a um problema no lynx(1) é possível que páginas como esta façam com que o lynx(1) exauste toda a memória e falhe ao tentar obter tais páginas.
    Uma correção via código fonte existe e corrige esse problema.

  • 022: CORREÇÃO DE CONFIANÇA, 10 de Novembro de 2004
    pppd(8) contém um problema que permite que um atacante quebre a sua própria conexão, porém não pode ser usado para negar serviço a outros usuários.
    Uma correção via código fonte existe e corrige esse problema.

  • 021: CORREÇÃO DE CONFIANÇA, 10 de Novembro de 2004
    O BIND possui um problema que resulta na tentativa de contactar servidores de nomes via IPv6, mesmo em casos de não haver conectividade IPv6. Isso resulta em timeouts e portanto pesquisas DNS lentas.
    Uma correção via código fonte existe e corrige esse problema.

  • 020: CORREÇÃO DE SEGURANÇA, 20 de Setembro de 2004
    Eilko Bos relatou que a autenticação radius, implementada pelo login_radius(8), não estava verificado a "shared secret" utilizada na respostas enviadas pelo servidor radius. Isso pode permitir que um atacante forge uma resposta, concedendo-lhe acesso. Note que o OpenBSD não possui autenticação radius por padrão.
    Uma correção via código fonte existe e corrige esse problema.

  • 019: CORREÇÃO DE SEGURANÇA, 16 de Setembro de 2004
    Chris Evans relatou diversas falhas (overflows de pilha e inteiros) na biblioteca Xpm que analiza arquivos de imagem (CAN-2004-0687, CAN-2004-0688). Alguns desses podem ser exploráveis quando fornecendo arquivos de imagem maliciosos em uma aplicação que lida com imagens XPM, caso eles consigam escapar do ProPolice.
    Uma correção via código fonte existe e corrige esse problema.

  • 018: CORREÇÃO DE SEGURANÇA, 10 de Setembro de 2004
    O módulo mod_rewrite do httpd(8) pode ser forçado a gravar um byte de tamanho zero em uma posição de memória arbitrária fora de um array char causando um DoS ou possivelmente buffer overflows. Isso irá requerer que o suporte a dbm seja ativado no mod_rewrite e que seja utilizado um arquivo dbm malicioso.
    Uma correção via código fonte existe e corrige esse problema.

  • 017: CORREÇÃO DE CONFIANÇA: 29 de Agosto de 2004
    Devido a um mal gerenciamento de erro na zlib, um atacante poderá potencialmente causar um ataque de Negação de Serviço. CAN-2004-0797 .
    Uma correção via código fonte existe e corrige esse problema.

  • 016: CORREÇÃO DE CONFIANÇA: 26 de Agosto de 2004
    Conforme relatado por Vafa Izadinia, uma bridge(4) com processamento IPsec ativado pode ser quebrada remotamente por um único echo ICMP atravessando-a.
    Uma correção via código fonte existe e corrige esse problema.

  • 015: CORREÇÃO DE CONFIANÇA: 25 de Agosto de 2004
    Melhorada a verificação de erros ICMP para que o impacto de ataques ICMP contra o TCP seja minimizado.
    http://www.gont.com.ar/drafts/icmp-attacks-against-tcp.html
    Uma correção via código fonte existe e corrige esse problema.

  • 014: CORREÇÃO DE CONFIANÇA: 25 de Julho de 2004
    Sob certa carga na rede o kernel pode ficar sem espaço de armazenamento. Isso foi encontrado em um ambiente usando CARP em uma interface VLAN. Inicialmente, esse problema se manifestou como uma falha relacionada a FPU no processo de inicialização do sistema.
    Uma correção via código fonte existe e corrige esse problema.

  • 013: CORREÇÃO DE SEGURANÇA: 12 de Junho de 2004
    Múltiplas vulnerabilidades foran encontradas no httpd(8) / mod_ssl. CAN-2003-0020, CAN-2003-0987, CAN-2004-0488, CAN-2004-0492.
    Uma correção via código fonte existe e corrige esse problema.

  • 012: CORREÇÃO de SEGURANÇA: 10 de Junho de 2004
    Como descrito por Thomas Walpuski, o isakmpd(8) ainda é vulnerável à remoção de SA não autorizados. Um atacante pode remover túneis IPsec de acordo com sua vontade.
    Uma correção via código fonte existe e corrige esse problema.

  • 011: CORREÇÃO DE SEGURANÇA: 9 de Junho de 2004
    Múltiplas vulnerabilidades remotas foram encontradas no servidor cvs(1) que permitem a um atacante destruir o servidor ou possivelmente executar códigos arbitrários com os mesmos privilégios que o programa servidor CVS.
    Uma correção via código fonte existe e corrige esse problema.

  • 010: CORREÇÃO DE CONFIANÇA: 9 de Junho de 2004
    Um problema de FIFO foi introduzido no OpenBSD 3.5 que ocorre quando um FIFO é aberto em modo não-bloqueado para escrita quando não existem processos lento este mesmo FIFO. Um programa afetado por isso é o servidor de email qmail que pode entrar em um loop infinito e consumir toda a CPU.
    Uma correção via código fonte existe e corrige esse problema.

  • 009: CORREÇÂO DE SEGURANÇA: 30 de Maio de 2004
    Uma falha no servidor kdc(8) Kerberos pode resultar que um administrador de um realm Kerberos tenha a habilidade de fingir ser qualquer administrador de qualquer outro realm que já tenha estabelecido uma confiaça de realm cruzada com o seu próprio realm. A falha é causada pela verificação inadequada do campo "transited" em uma requisição Kerberos. Para maiores detalhes veja o anúncio do Heimdal.
    Uma correção via código fonte existe e corrige esse problema.

  • 008: CORREÇÂO DE SEGURANÇA: 26 de Maio de 2004
    Com a introdução do código IPv6 no xdm(1), um teste no recurso 'requestPort' foi apagado por acidente. Isso faz com que o xdm crie um socket mesmo se o xdmcp esteja desativado no xdm-config, especificando o requestPort para 0. Veja o XFree86 bugzilla para detalhes.
    Uma correção via código fonte existe e corrige esse problema.

  • 007: CORREÇÂO DE SEGURANÇA: 20 de Maio 2004
    Um overflow de armazenamento de pilha no servidor cvs(1) foi descoberto e pode ser explorado por clientes que enviem requisições mal formadas a ele permitindo assim que esses clientes executem código arbitrário com os mesmos privilégios do programa servidor CVS.
    Uma correção via código fonte existe e corrige esse problema.

  • 006: CORREÇÂO DE SEGURANÇA: 13 de Maio 2004
    Verificação de overflows de inteiros no procfs. A utilização do procfs não é recomendada.
    Uma correção via código fonte existe e corrige esse problema.

  • 005: CORREÇÃO DE CONFIANÇA: 6 de Maio de 2004
    Responda para os SYN em janela com um ACK limitado em freqüencia.
    Uma correção via código fonte existe e corrige esse problema.

  • 004: CORREÇÃO DE CONFIANÇA: 5 de Maio de 2004
    Restaurada a função de negociação de tags/wide/sync em algumas controladoras SCSI ( ex. siop(4), trm(4), iha(4) ).
    Uma correção via código fonte existe e corrige esse problema.

  • 003: CORREÇÃO DE CONFIANÇA: 5 de Maio de 2004
    Controladoras gdt(4) de modelos recentes travam em alta carga.
    Uma correção via código fonte existe e corrige esse problema.

  • 002: CORREÇÃO DE SEGURANÇA: 5 de Maio de 2004
    Problemas de validação de caminho (path) foram encontrados no cvs(1), permitindo que clientes maliciosos criem arquivos fora do repositório, permitindo assim que servidores maliciosos sobreescrevam os arquivos fora da árvore CVS local no cliente e permitindo que clientes façam checkout de arquivos fora do repositório CVS.
    Uma correção via código fonte existe e corrige esse problema.
