2.1, 2.2, 2.3, 2.4, 2.5, 2.6, 2.7, 2.8, 2.9, 3.0, 3.1, 3.2, 3.3, 3.5, 3.6, 3.7.

Alle Architekturen

  • 007: SECURITY FIX: 14. Dezember 2004
    Auf Systemen, auf denen isakmpd(8) läuft, ist es für einen lokalen Anwender möglich, eine Kernelspeicherverletzung und einen Systemfehler zu verursachen, indem ipsec(4)-,credentials' auf einem Socket gesetzt werden.
    Es existiert ein Quellcodepatch, der das Problem behebt.

  • 004: RELIABILITY FIX: 10. November 2004
    Wegen einem Fehler in lynx(1) ist es für Seiten wie dieser möglich dafür zu sorgen, dass lynx(1) den Speicher verbraucht und dann abstürzt, wenn solche Seiten verarbeitet werden.
    Es existiert ein Quellcodepatch, der das Problem behebt.

  • 003: RELIABILITY FIX: 10. November 2004
    pppd(8) beinhaltet einen Fehler, der es einem Angreifer ermöglicht, seine eigene Verbindung abstürzen zu lassen, aber der Fehler kann nicht ausgenutzt werden, um die Dienste, die anderen Benutzern zur Verfügung gestellt werden, zu unterbinden.
    Es existiert ein Quellcodepatch, der das Problem behebt.

  • 002: RELIABILITY FIX: 10. November, 2004< /font>
    BIND beinhaltet einen Fehler, der darin enden kann, dass BIND versucht, Nameserver über IPv6 zu erreichen, selbst wenn die IPv6 Verbindung nicht existiert. Dies resultiert in unnötigen Timeouts und daher langsamen DNS Anfragen.
    Es existiert ein Quellcodepatch, der das Problem behebt.

  • 031: SECURITY FIX: 20. September 2004
    Eilko Bos hat gemeldet, dass die Radius Authentifizierungs, so wie von login_radius(8) implementiert, nicht das Shared Secret benutzt, das für die Antworten verwendet wird, die vom Radius-Server gesendet werden. Das könnte einem Angreifer erlauben, eine Antwort zu fälschen, um sich Zugriff zu verschaffen. In OpenBSD ist die Radius Authentifizierung standardmässig abgeschaltet.
    Es existiert ein Quellcodepatch, der das Problem behebt.

  • 030: SECURITY FIX: 16. September 2004
    Chris Evans hat mehrere Fehler (Stack und Integer Overflow) im Xpm Library Code gemeldet, der Bild-Daten parst. (CAN-2004-0687, CAN-2004-0688). Einige dieser Fehler wären ausnutzbar, wenn bösartige Bilder in einer Anwendung XPM Bilder behandeln, sofern diese Fehler ProPolice entkommen würden.
    Es existiert ein Quellcodepatch, der das Problem behebt.

  • 029: SECURITY FIX: 10. September 2004
    httpd(8) 's mod_rewrite Modul kann dazu gebracht werden ein Null-Byte an eine beliebige Speicher Position ausserhalb eines Arrays zu schreiben, was zu einem DoS oder möglichen Buffer-Overflow führen kann. Dazu müsste man jedoch dbm für mod_rewrite einschalten und zusätzlich eine bösartige dbm Datei benutzen.
    Es existiert ein Quellcodepatch, der das Problem behebt.

  • 016: RELIABILITY FIX: 26. August 2004
    Wie von Vafa Izadinia berichtet kann eine bridge(4) mit eingeschalteter IPsec Verarbeitung durch ein einziges ICMP echo zum Absturz gebracht werden, das die bridge durchquert.
    Es existiert ein Quellcodepatch, der das Problem behebt.

  • 015: RELIABILITY FIX: 25. August 2004
    Verbesserte Verifikation von ICMP Fehlern um die Auswirkungen von ICMP Angriffen gegen TCP zu minimieren.
    http://www.gont.com.ar/drafts/icmp-attacks-against-tcp.html
    Es existiert ein Quellcodepatch, der das Problem behebt.

  • 026: RELIABILITY FIX: 25. Juli 2004
    Unter einer bestimmten Netzwerklast kann dem Kernel der Stack space ausgehen. Das wurde in einer Umgebung mit CARP und VLAN Interfaces festgestellt. Ursprünglich manifestierte sich dieses Problem als FPU bezogener Crash beim Hochfahren.
    Es existiert ein Quellcodepatch, der das Problem behebt.

  • 013: SECURITY FIX: 12. Juni 2004
    Mehrere Schwachstellen wurden in httpd(8) / mod_ssl gefunden. CAN-2003-0020, CAN-2003-0987, CAN-2004-0488, CAN-2004-0492.
    Es existiert ein Quellcodepatch, der das Problem behebt.

  • 012: SECURITY FIX: 10. Juni 2004
    Wie von Thomas Walpuski veröffentlicht, ist isakmpd(8) immer noch gegen das unautorisierte Löschen von SAs verwundbar. Ein Angreifer kann IPsec Tunnel nach Belieben löschen.
    Es existiert ein Quellcodepatch, der das Problem behebt.

  • 023: SECURITY FIX: 9. Juni 2004
    Es wurden mehrere remote vulnerabilities im cvs(1) Server gefunden, die es einem Angreifer erlauben, den Server zum Absturz zu bringen oder möglicherweise beliebigen Code mit den Rechten des CVS Servers aszuführen.
    Es existiert ein Quellcodepatch, der das Problem behebt.

  • 009: SECURITY FIX: 30. Mai 2004
    Ein Fehler im Kerberos V kdc(8) Server könnte sich so auswirken, dass der Administrator eines Kerberos Realms sich als Principal jedes beliebigen anderen Realms ausgeben kann, das einen Cross-Realm-Trust zum eigenen Realm aufgebaut hat. Dieser Fehler ist wegen einer unzureichenden Überprüfung des "transited" Felds in einer Kerberos Anfrage. Für mehr Details lies die Ankündigung von Heimdal.
    Es existiert ein Quellcodepatch, der das Problem behebt.

  • 021: SECURITY FIX: 20. Mai 2004
    Es wurde ein heap overflow im cvs(1) Server entdeckt, der von Clients mittels falsch geformter Anfragen ausgenutzt werden kann, was diese Clients in die Lag versetzt beliebigen Code auf dem Server mit den Rechten, unter denen der Server läft, auszuführen.
    Es existiert einen Quellcode-Patch, der das Problem behebt.

  • 020: SECURITY FIX: 13. Mai 2004
    Überprüfung von Integer Overflow in Procfs. Von der Benutzung von procfs wird abgeraten.
    Es existiert einen Quellcode-Patch, der das Problem behebt.

  • 019: RELIABILITY FIX: 6. Mai 2004
    Antworten auf in-window SYN mit einer begrenzten Anzahl von ACK.
    Es existiert einen Quellcode-Patch, der das Problem behebt.

  • 018: RELIABILITY FIX: 5. Mai 2004
    Unter Last hängen sich "aktuelle Modelle" von gdt(4) auf.
    Es existiert einen Quellcode-Patch, der das Problem behebt.

  • 017: SECURITY FIX: May 5, 2004
    Es wurden Probleme bei der Überprüfung von Pfadnamen in cvs(1) gefunden, die es bösartigen Clients erlauben, Dateien ausserhalb des Repository zu erzeugen, es bösartigen Servern erlauben, Dateien ausserhalb des lokalen CVS Baums auf dem Client zu überschreiben und es ausserdem Clients erlauben, Dateien von ausserhalb des CVS Repository zu laden.
    Es existiert einen Quellcode-Patch, der das Problem behebt.

  • 012: RELIABILITY FIX: 14. Februar 2004
    Es existieren mehrere Buffer Overflows im Code, der die font.aliases Dateien in XFree86 parst. Dank ProPolice können diese nicht dazu ausgenutzt werden, um erweiterte Rechte zu erlangten, aber sie können den X Server dazu bringen, sich zu beenden.
    Es existiert einen Quellcode-Patch, der das Problem behebt.

  • 011: SECURITY FIX: 8. Februar 2004
    Es gibt ein Problem beim Behandeln der IPv6 MTU, das von Angreifern dazu benutzt werden könnte, Denial of Service Angriffe gegen Rechner mit per IPv6 erreichbaren TCP Ports zu fahren.
    Es existiert ein Quellcode-Patch, der das Problem behebt.

  • 010: SECURITY FIX: 5. Februar 2004
    Es gibt einen Bug beim Zählen von Referenzen im shmat(2) Systemaufruf, der von Angreifern unter bestimmten Umständen dazu benutzt werden könnte, Kernel Speicher zu beschreiben.
    Es existiert ein Quellcode-Patch, der das Problem behebt.

  • 009: SECURITY FIX: 13. Januar 2004
    Mehrere Fehler bei der Nachrichtenbehandlung in isakmpd(8) wurden von Thomas Walpuski gefunden. Sie erlauben einem Angreifer, bestimmte SAs zu löschen. Der Patch beinhaltet auch ein Verläßlichkeitsfix für ein Filedeskriptor-Loch, das Probleme verursacht, wenn eine Crypto-Karte installiert ist.
    Es existiert ein Quellcode-Patch, der diese Probleme behebt.

  • 008: RELIABILITY FIX: 20 November 2003
    Durch eine inkorrekte Grenzwertüberprüfung ist es einem lokalen Benutzer möglich durch die Übergabe bestimmter Argumente an die Funktionen semctl(2) und semop(2) das System zum Absturz zu bringen.
    Es gibt einen Source Code Patch, der das Problem behebt.

  • 007: RELIABILITY FIX: 20 November 2003
    Für einen lokelen Benutzer ist es, durch sysctl(3) mit bestimmten Argumenten, möglich einen Absturz zu verursachen.
    Es gibt einen Source Code Patch, der das Problem behebt.

  • 005: RELIABILITY FIX: 4. November 2003
    Es ist für einen lokalen Benutzer möglich eine "system panic" auszulösen, indem er ein spezielles Binärprogramm mit einem ungültigen Header ausführt.
    Es gibt einen Source Code Patch, der das Problem behebt.

  • 004: RELIABILITY FIX: 1. November 2003
    Ein Benutzer mit Schreibrechten auf die httpd.conf oder eine .htaccess Datei kann den httpd(8) zum Absturz bringen oder möglicherweise eigenen Code als Benutzer www laufen lassen (obwohl wir glauben, dass ProPolice die Code-Ausführung verhindern wird).
    Es gibt einen Source Code Patch, der das Problem behebt.

  • 003: RELIABILITY FIX: 1. November 2003
    Einem lokalen Benutzer ist es möglich, eine system panic auszulösen durch das Fluten mit gefälschten ARP-Anfragen.
    Es gibt einen Source Code Patch, der das Problem behebt.

  • 002: SECURITY FIX: 1. November 2003
    Das Benutzen bestimmter ASN.1 Kodierungen oder fehlgeformte Public Keys erlauben es einem Angreifer, einen Denial-of-Service Angriff gegen Anwendungen zu fahren, die gegen ssl(3) gelinkt sind. Dies betrifft die OpenSSH nicht.
    Es gibt einen Source Code Patch, der das Problem behebt.

  • 001: DOCUMENTATION FIX: 1. November 2003
    Die Dokumentation auf dem CD Booklet hat ein falsches Beispiel für eine Standardinstallation.
    Dort, wo geschrieben steht:

               # pkg_add ftp://ftp.openbsd.org/pub/OpenBSD/3.4/packages/i386

    sollte in Wirklichkeit stehen:

               # pkg_add ftp://ftp.openbsd.org/pub/OpenBSD/3.4/packages/i386/

    Der extra / am Ende der Zeile ist wichtig. Wir geben keine Patch files heraus für Sachen, die auf Papier gedruckt sind.

i386

  • 006: SECURITY FIX: 17 November 2003
    Es könnte für einen lokalen Benutzer möglich sein den Stapel von compat_ibcs2(8) zum Überlauf zu bringen.
    Die ProPolicy fängt eine mögliche Eskalation der Privilegien auf und verwandelt diese in einen Denial of Service. Es ist deswegen nicht nötig, die iBCS2 Emulation mit sysctl(8) zu aktivieren.
    Es gibt einen Source Code Patch, der das Problem behebt.
