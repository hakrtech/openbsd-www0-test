Die Dateien dieser Version erhältst du auf einen der folgenden Wege:
  • Bestelle eine CDROM über unser Bestellsystem.
  • Sieh dir die FTP Seite an, um eine Liste der FTP-Mirrors zu betrachten, und den passenden für dich herauszufinden.
  • Gehe in das Verzeichnis pub/OpenBSD/3.3/ auf einer der spiegelnden Maschinen deiner Wahl.
  • Lies den Rest dieser Seite sorgfältig.
  • Sieh dir die 3.3 Errata Seite an, ob es Bugs und/oder Workarounds gibt.
  • Lies dir das detaillierte Log der Änderungen zwischen den Versionen 3.2 und 3.3 durch.



Hinweis: Alle anwendbaren Copyrights und Danksagungen können in den passenden Quellcodedateien in src.tar.gz, sys.tar.gz, XF4.tar.gz, oder den Dateien gefunden werden, die via ports.tar.gz geholt wurden. Die Distributionsdateien, die man braucht, um mit Hilfe der ports.tar.gz Datei packages zu bauen, sind aus Platzgründen nicht mit auf den CDROMs.

Was gibt es Neues?

Dies ist eine unvollständige Liste von neuen Features und Teilsystemen, die in OpenBSD 3.3 integriert sind. Eine vollständige Liste findet sich im changelog zu 3.3.

  • Integration der ProPolice Stack-Schutz-Technologie von Hiroaki Etoh in den Compiler des Systems. Dieser Schutz ist standardmäßig eingeschaltet. Mit dieser Änderung werden Funktionsaufrufe dahingehend modifiziert, dass der Stack geändert wird: ein Zufallswert (canary) wird vor die Rücksprungadresse gesetzt, und Puffer-Variablen werden näher an den "canary" plaziert, so dass reguläre Variablen weiter unten sind, und somit schwieriger zu überschreiben. Der Funktionsrücksprung untersucht dann, ob der "canary" noch intakt ist. Wenn nicht, wird der Prozess beendet. Diese Änderung macht es sehr schwierig für einen Angreifer die Rücksprungadresse zu überschreiben, die beim Verlassen der Funktion genutzt wird.

  • W^X (gesprochen: "W xor X") auf Architekturen, die eine Unterstützung für ein reines execute-Bit in der MMU (sparc, sparc64, alpha, hppa) haben. Dies ist eine feinkörnige Kontrolle von Rechten im Speicher, die sicherstellt, dass Speicher, auf den eine Anwendung schreibend zugreift, nicht gleichzeitig ausführbar sein kann und umgekehrt. Das erschwert potentielle Buffer Overflows und andere Angriffe: Als Resultat kann ein Angreifer also nicht mehr an einen beliebigen Platz im Speicher seinen Code schreiben und dann ausführen. (HINWEIS: i386 und powerpc unterstützen kein W^X in 3.3; dieses Feature wird aber für i386 in 3.3-current bereits unterstützt und unter 3.4 werden beide Prozessoren unterstützt werden.)

  • Eine noch weitergehende Reduzierung der setuid und setgid Binaries, und noch mehr Nutzung von chroot im System. Obwohl noch einige Programme setuid oder setgid sind, holen sich fast alle die notwendige Ressource und lassen ihre Rechte/Privilegien dann fallen.

  • Der X Window Server und xconsole benutzen jetzt privilege separation, um bessere Sicherheit zu erzielen. xterm wurde auch dahingehend modifziert, dass jetzt privilege revocation zu einem möglichst frühen Zeitpunkt durchgeführt wird. xdm läuft unter einem speziellen User und spezieller Gruppe, um weiter zu beschränken, was schieflaufen könnte.

  • Wie immer wurde die Dokumentation verbessert, und zwar besonders die man pages und die Web FAQ. Ein größerer Teil der Webseiten ist jetzt in verschiedenen Sprachen vorhanden.

  • Eine vollständigere und besser getestete Sammlung von "ports". Die Anzahl von setuid/setgid ports wurde ebenso wesentlich verringert. Viele derjenigen, die weiterhin setuid sind, wurden so modifiziert, dass diese Privilegien möglichst früh entzogen werden.

  • Über 2000 vorkompilierte und getestete packages.

  • Signifikante Verbesserungen an der pthread library.

  • Eine erstaunliche Menge von Erweiterungen und Verbesserungen der Stabilität an unserem Paketfilter pf, einschließlich:
    • Queue, ein Bandbreiten-Management System (unter der Oberfläche wird altq verwendet)
    • Anchors, die es erlauben subrulesets zu benutzen, die unabhängig geladen und modifiziert werden
    • Tables, eine sehr effiziente Möglichkeit große Mengen an Listen in den Regeln zu verwenden
    • es gibt jetzt Address Pools und Redirect/NAT zu multiplen Adressen und dadurch Load Balancing
    • die Konfigurationssprache wurde deutlich flexibler gestaltet
    • Unterstützung für TCP Window Scaling
    • Volle CIDR Unterstützung
    • Frühe Rückkehr bei Überprüfung von Checksummen ungültiger Pakete
    • Performance Boost: Große Regelsätze werden bedeutend schneller geladen
    • spamd, ein Spam Verzögerungs-Daemon, an den SMTP Verbindungen weitergeleitet werden können. Dieser daemon verwaltet Verbindungen anhand von Positiv- oder Negativ-Listen, "teert" die Verbindungen (verlangsamt sie) und stellt sicher, daß der Spammer weiss, warum seine Mail nicht akzeptiert wurde.

  • Deutliche verbesserte Unterstützung von sparc64: mehr Modelle werden unterstützt und einige bedeutende Bugs wurden beseitigt.

  • Das System enthält die folgenden großen Komponenten von Drittherstellern/Anbietern:
    • XFree86 4.2.1 (und i386 enthält auch noch 3.3.X Server, daher Unterstützung für alle Chipsets)
    • Gcc 2.95.3 (+ patches)
    • Perl 5.8.0 (+ patches)
    • Apache 1.3.27, mod_ssl 2.8.12, DSO Unterstützung (+ patches)
    • OpenSSL 0.9.7beta3 (+ patches)
    • Groff 1.15
    • Sendmail 8.12.9
    • Bind 9.2.2 (+ patches)
    • Lynx 2.8.2rel.1 mit hinzugefügter HTTPS Unterstützung (+ patches)
    • Sudo 1.6.7
    • Ncurses 5.2
    • Latest KAME IPv6
    • KTH Kerberos 1.1.1
    • Heimdal 0.4e (+ patches)
    • OpenSSH 3.6

  • Viele Verbesserungen in Bezug auf Sicherheit und Zuverlässigkeit (siehe auch die roten Zeilen im kompletten changelog).

  • und vieles mehr.

Wie man es installiert

Es folgen Instruktionen, die du in gedruckter Form haben würdest, wenn du anstatt einer alternativen Installationsmethode eine CDROM gekauft hättest. Die Anleitungen für FTP-Installationen sind recht ähnlich, die Anleitungen für die CDROM-Installationen wurden jedoch intakt gelassen, um zu zeigen, um wieviel einfacher alles mit einer CDROM gewesen wäre.

Bitte beachte auch die folgenden Dateien auf den CDROMs oder den FTP-Servern, die noch weitere Details enthalten, um OpenBSD 3.3 auf deiner Maschine zu installieren:

  • CD1:3.3/i386/INSTALL.i386

  • CD2:3.3/macppc/INSTALL.macppc
  • CD2:3.3/vax/INSTALL.vax

  • CD3:3.3/sparc/INSTALL.sparc
  • CD3:3.3/sparc64/INSTALL.sparc64

  • FTP:.../OpenBSD/3.3/alpha/INSTALL.alpha
  • FTP:.../OpenBSD/3.3/hp300/INSTALL.hp300
  • FTP:.../OpenBSD/3.3/hppa/INSTALL.hppa
  • FTP:.../OpenBSD/3.3/mac68k/INSTALL.mac68k
  • FTP:.../OpenBSD/3.3/mvme68k/INSTALL.mvme68k

Hier gibt es kurze Installationsinformationen für Leute, die mit OpenBSD vertraut sind und den Befehl "disklabel -E" kennen. Wenn du irgendwie unsicher in Bezug auf die Installation bist, lies die entsprechenden INSTALL.* Dateien, die oben angegeben sind!

OpenBSD/i386:

    Spiele mit den BIOS-Optionen herum und versuche, von CD zu booten. Die i386 Version von OpenBSD ist auf der CD1. Wenn dein BIOS kein Booten von CD unterstützt, schreibe CD1:3.3/i386/floppy33.fs auf eine Floppy, und nimm diese als Boot-Medium.

    Benutze CD1:3.3/i386/floppyB33.fs für eine bessere Unterstützung von SCSI-Controllern oder CD1:3.3/i386/floppyC33.fs für bessere Laptop-Unterstützung.

    Wenn du OpenBSD neben anderen Betriebssystemen betreiben willst, solltest du unbedingt das Dokument INSTALL.i386 lesen.

    Benutze das "rawrite" Werkzeug, von CD:/3.3/tools/rawrite.exe um eine Floppy unter MS-DOS zu erstellen. Unter Unix einfach das dd(1) Utility benutzen. Das folgende Beispiel für dd(1) könnte "floppy", "rfd0c", oder "rfd0a" als Device benutzen. 

      # dd if=<file> of=/dev/<device> bs=32k

    Benutze ausschließlich perfekte und sauber formatierte Floppies OHNE DEFEKTE BLÖCKE oder deine Installation wird wahrscheinlich nicht klappen. Mehr Informationen über die Installation von OpenBSD/i386 gibts in der FAQ4.1.

OpenBSD/macppc:

    Lege die CD2 in dein CDROM Laufwerk und drücke die C Taste während des Einschaltens, bis schließlich die Anzeige OpenBSD/macppc boot zeigt.

    Alternativ kannst du am Open Firmware prompt boot cd:,ofwboot /3.3/macppc/bsd.rd eingeben.

OpenBSD/vax:

    Boote über das Netzwerk via mopbooting, so wie es in INSTALL.vax beschrieben ist.

OpenBSD/sparc:

    Version 3.3 von OpenBSD/sparc befindet sich auf der CD3. Um von dieser CD zu booten, kannst du eines von den zwei Kommandos benutzen, die weiter unten aufgeführt sind, je nachdem welche ROM Version du hast. 
      > boot cdrom 3.3/sparc/bsd.rd
or
> b sd(0,6,0)3.3/sparc/bsd.rd

    Wenn deine Sparc kein CD Laufwerk hat, kannst du alternativ dazu auch von einer Floppy booten. Dazu musst du die Datei "CD3:3.3/sparc/floppy33.fs" auf eine Floppy schreiben. Mehr Informationen gibts es in der FAQ4.1. Um von dieser Floppy zu booten, kannst du eines von den zwei Kommandos benutzen, die weiter unten aufgeführt sind, je nachdem welche ROM Version du hast. 

      > boot floppy
or
> boot fd()

    Stelle sicher, dass du eine sauber formatierte Floppy OHNE DEFEKTE BLÖCKE benutzt, ansonsten wird deine Installation mit hoher Wahrscheinlichkeit nicht klappen.

    Wenn deine Sparc weder Floppy noch CD-Laufwerk hat, kannst du entweder ein bootbares Band erzeugen, oder via Netzwerk installieren, indem du den Anweisungen in der Datei INSTALL.sparc folgst.

OpenBSD/sparc64:

    Lege die CD3 in dein CDROM Laufwerk und gib boot cdrom ein.

    Wenn das nicht klappt oder du kein CDROM-Laufwerk hast, kannst du die Datei CD3:3.3/sparc64/floppy33.fs auf eine Floppy schreiben und sie mit boot floppy booten.
    Stelle sicher, dass du eine sauber formatierte Floppy OHNE DEFEKTE BLÖCKE benutzt, ansonsten wird deine Installation mit hoher Wahrscheinlichkeit nicht klappen.

    Du kannst auch CD3:3.3/sparc64/miniroot33.fs auf die Swap-Partition auf der Platte schreiben und mit boot disk:b booten.

    Wenn nichts davon klappt, kannst du auch vom Netz booten, wie es in INSTALL.sparc64 beschrieben ist.

OpenBSD/alpha:

    Schreibe FTP:3.3/alpha/floppy33.fs oder FTP:3.3/alpha/floppyB33.fs (abhängig von deiner Maschine) auf eine Diskette und gib boot dva0 ein. INSTALL.alpha enthält dazu weitere Details.

    Stelle sicher, dass du eine sauber formatierte Floppy OHNE DEFEKTE BLÖCKE benutzt, ansonsten wird deine Installation mit hoher Wahrscheinlichkeit nicht klappen.

OpenBSD/hp300:

    Boote mittels der Anweisungen in INSTALL.hp300 über dein Netzwerk.

OpenBSD/hppa:

    Boote vom Netzwerk, indem du die Anweisungen in INSTALL.hppa oder von der hppa Plattform Seite befolgst.

OpenBSD/mac68k:

    Boote MacOS ganz normal und partitioniere deine Festplatte mit den normalen A/UX Konfigurationen. Dann entpacke die Macside Utilities aus FTP:3.3/mac68k/utils auf deine Festplatte. Benutze Mkfs, um deine Dateisysteme auf den A/UX Partitionen, die du gerade erzeugt hast, zu erstellen. Dann benutze den "BSD/Mac68k Installer", um alle Sets in FTP:3.3/mac68k/ auf deine Partitionen zu kopieren. Letzlich bist du nun soweit, dass du den "BSD/Mac68k Booter" mit dem Ort, an dem dein Kernel liegt, konfigurieren kannst, und dann das System booten kannst.

OpenBSD/mvme68k:

    Du kannst ein bootbares Installationsband erzeugen, oder über Netzwerk booten.
    Der Netzwerk Boot benötigt eine MVME68K BUG Version dass die NIOT und NBO Debugger Kommandos unterstützt. Die Anweisungen in INSTALL.mvme68k beinhalten weitere Details.

Hinweise zum Source Code:

    src.tar.gz enthält ein Source-Code-Archiv, das in /usr/src beginnt. Diese Datei enthält bis auf die Kernel-Sourcen, die sich in einem anderen Archiv befinden, alles, was du brauchst. Um es zu entpacken: 

      # mkdir -p /usr/src
# cd /usr/src
# tar xvfz /tmp/src.tar.gz

    sys.tar.gz enthält ein Quellcode-Archiv, das in /usr/src/sys beginnt. Diese Datei enthält alle Kernel-Quellen, die man benötigt, um einen neuen Kernel zu erzeugen. Um sie zu entpacken: 

      # mkdir -p /usr/src/sys
# cd /usr/src
# tar xvfz /tmp/sys.tar.gz

    Beide Archive enthalten einen regulären CVS checkout. Wenn man diese Dateibäume benutzt, ist es möglich, die anoncvs-Server wie hier beschrieben zu benutzen. Wenn man diese Dateien benutzt, hat man ein viel schnelleres erstes Update, als wenn man einen frischen, eigenen OpenBSD Quellbaum (source tree) mit Hilfe von CVS holt.

Ports Tree

Mitgeliefert wird auch ein ports tree Archiv. Um es zu entpacken: 

    # cd /usr
# tar xvfz /tmp/ports.tar.gz
# cd ports

Das ports/ Unterverzeichnis ist ein checkout für den OpenBSD ports tree. Lies am besten die Ports Seite, wenn du noch nichts über Ports weisst. Dieser Text ist keine Anleitung, wie man die Ports benutzt, sondern nur dazu da, um den Leuten den Einstieg in das OpenBSD ports System zu ermöglichen.

Natürlich ist das OpenBSD ports System nicht vollständig. Es ist zu bezweifeln, dass es das jemals sein wird. Trotzdem wächst es sehr schnell und wird stabiler. Fast alle ports, die ausgeliefert werden, sollten ohne Probleme auf den meisten Architekturen kompilierbar sein (auf z.B. i386 konnten über 2000 packages ohne Probleme erzeugt werden).

Das ports/ Verzeichnis stellt ein CVS (siehe in der manpage für cvs(1) nach, wenn du mit CVS nicht vertraut bist) checkout unserer ports dar. Wie unser kompletter Source tree ist auch auch unser Ports Tree via anoncvs verfügbar. Um ihn also auf dem Laufenden zu halten, musst du den ports/ tree auf einem schreibbaren Medium haben und den tree mit diesem Kommando updaten: 

    # cd [portsdir]/; cvs -d anoncvsserver.openbsd.org:/cvs update -Pd -rOPENBSD_3_3

[Natürlich musst du die Namen für Server und [portsdir] mit deinen lokalen Verzeichnis, bzw. dem Namen eines anoncvs-Servers in der Nähe austauschen.]

Denk daran, dass die meisten ports via ftp als fertige packages erhältlich sind. Upgedatete packages für 3.3 werden verfügbar gemacht, sobald Probleme bekannt sind.

Falls du gerne einen weiteren port sehen willst, mithelfen willst, oder einfach mehr erfahren willst, solltest du die Mailingliste ports@openbsd.org kennenlernen.
