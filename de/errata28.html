  • Alle Architekturen

    • 013: SECURITY FIX: 21. August, 2001
      Es gibt ein Sicherheitsloch in       sendmail(8) , dass einem lokalen Angreifer auf der Maschine erlaubt sich root Privilegien anzueignen, indem er debug Paramter angibt, die "out-of-bound" sind, also ausserhalb der Grenzen.
      Ein Source Code Patch behebt das Problem

    • 007: SECURITY FIX: 15. Juni 2001
      Es gibt eine race condition in der Kernel execve(2) Implementation, die ein kleines Fenster für einen nicht-privilegierten Angreifer öffnet, der dann ptrace(2) an einen suid/sgid Prozess attachen kann.
      Ein Source Code Patch behebt das Problem.

    • 029: SECURITY FIX: 30. Mai 2001
      Programme, die die fts(3) Routinen benutzen (wie etwa rm, find, und die meisten Programme, die das -R Flag akzeptieren) können dazu gebracht werden in das falsche Verzeichnis zu wechseln, wenn das parent dir von einem Unterverzeichnis heraus verändert wird. Das ist so ähnlich wie der alte fts Bug, passiert aber, wenn man Verzeichnisse verlässt, anstatt in sie hineinzugehen. Es gibt einen Source-Code Patch, der das Problem beseitigt.. Dies ist die zweite Version des Patches.

    • 028: SECURITY FIX: 29. Mai 2001
      Die signal handler in sendmail(8) enthalten Code, der im Umfeld der signal handler unsicher ist. Das führt zu einer potentiell gefährlichen race condition. Momentan ist das nur eine theoretische Attacke, die auf dem local host exploited werden kann (falls überhaupt).
      Es gibt einen Source Code Patch, der das Problem dadurch beseitigt, dass er sendmail auf Version 8.11.4 updatet.

    • 027: SECURITY FIX: 23. Apr 2001
      IPF hat ein echtes Problem mit fragment cacheing, der Bug tritt in Erscheinung, wenn man die ipf(5) Syntax "keep state" benutzt.
      Es gibt einen Source-Code Patch, der das Problem beseitigt.

    • 026: SECURITY FIX: 23. Apr 2001
      ftpd(8) hat einen potentiellen DoS bezogen auf glob(3). Dieser Patch führt ein GLOB_LIMIT ein, und eliminiert damit den DoS. Du musst vorher den 025_glob patch installiert haben, bevor du diesen Patch installierst.
      Es gibt einen Source-Code Patch, der das Problem beseitigt.

    • 025: SECURITY FIX: 10. Apr 2001
      glob(3) enthält mehrere buffer overflows.
      Es gibt einen Source-Code Patch, der das Problem beseitigt.

    • 024: SECURITY FIX: 18. März 2001
      Die readline library, die mit OpenBSD ausgeliefert wurde, erlaubt das Erzeugen von history files mit einer 'permissive' umask(2). Das kann zu einem Leck an sensitiven Informationen in Applikationen führen, die Passwörter und ähnliches während der Userinteraktion benutzen. (ein Beispiel ist mysql).
      Es gibt einen Source-Code Patch, der das Problem beseitigt.

    • 023: SECURITY FIX: 2. März 2001
      Unzureichende Checks im IPSEC AH IPv4 option handling Code können zu einem 'buffer overrun' und zu einem 'remote DoS' führen. Diese Option ist standardmässig deaktiviert.
      Es gibt einen Source-Code Patch, der das Problem beseitigt.

    • 021: SECURITY FIX: 22. Feb 2001
      Es gibt einen exploitbaren Buffer Overflow in sudo.
      Es gibt einen source code Patch, der das Problem beseitigt.

    • 020: IMPLEMENTATION FIX: 15. Feb. 2001
      Das Client seitige ident Protokoll in libwrap war defekt, was alles betraf, was libwrap benutzt, inklusive tcpd. Der Effekt war, dass libwrap nie ident Werte von der Gegeseite gefordert und gelogt hat.
      Es gibt einen source code Patch, der das Problem beseitigt.

    • 019: IMPLEMENTATION FIX: 31. Jan. 2001
      Die Memory allocation im PCI LANCE Treiber in Ordnung gebracht, le. Ein Nebeneffekt davon ist, dass OpenBSD jetzt auch wieder unter VMWare funktioniert.
      Es gibt einen source code Patch, der das Problem beseitigt.

    • 018: SECURITY FIX: 29. Jan. 2001
      Bringe named mit ISC BIND 4.9.8-REL zusammen, was einige 'buffer vulnerabilities' beseitigt (tatsächlich sieht es so aus, als ob diese bereits vorher nicht auszunutzen waren).
      Es gibt einen source code Patch, der das Problem beseitigt.

    • 017: SECURITY FIX: 22. Jan. 2001
      Das rnd(4) Gerät benutzt nicht alle Eingabedaten, die zu ihm geschrieben werden.
      Es gibt einen source code Patch, der das Problem beseitigt.

    • 016: RELIABILITY FIX: Jan 4, 2001
      Erlaube es den ThunderLAN Karten, sich friedlich Interrupts zu teilen.
      Es gibt einen source code Patch, der das Problem beseitigt.

    • 014: SECURITY FIX: 22. Dez 2000
      Verbesserung von xlock(1)'s Authentifikation durch eine Authentifikation durch eine pipe in einem früh ge"fork"tem Prozess. Es gibt keine bekannte Möglichkeit zum Ausnutzen, das ist einfach nur ein vorrausschauender Patch.
      Ein Source-Code Patch beseitigt das Problem.

      Zusätzlich zum Source-Code Patch wurden neue new xlock binaries für alle unten aufgeführten Architekturen erzeugt. Installiere sie unter /usr/X11R6/bin/xlock und führe ein chmod 4755 /usr/X11R6/bin/xlock aus.

      • Xlock - i386
      • Xlock - sparc
      • Xlock - mac68k
      • Xlock - powerpc
      • Xlock - amiga
      • Xlock - hp300
      • Xlock - mvme68k

    • 013: SECURITY FIX: 18. Dez 2000
      Es gab im Procfs verschiedene overflows, die einem Eindringling zu root verhelfen könnten. Procfs ist standardmässig NICHT in OpenBSD eingeschaltet.
      Es gibt einen source code Patch, der das Problem beseitigt.

    • 011: RELIABILITY FIX: 13. Dez. 2000
      Das Crypto Subsystem könnte unkorrekterweise dabei versagen einige Software-Chiffren auszuführen, wenn im System eine Hardwarekarte existiert..
      Es gibt einen Source-Code Patch, der das Problem beseitigt.

    • 010: RELIABILITY FIX: Dec 11, 2000
      Es kann ein Crash während des 'fast routing' auftreten, wenn IPSEC eingeschaltet ist.
      Es gibt einen Source-Code Patch, der das Problem beseitigt.

    • 009: SECURITY FIX: 10. Dez 2000
      Es gibt noch ein weiteres Problem in den Kerberos libraries.
      Es gibt einen Source-Code Patch, der das Problem beseitigt.

    • 008: SECURITY FIX: 7. Dez. 2000
      Zwei Probleme wurden kürzlich im KerberosIV Code entdeckt.

      1. Ein symlink Problem wurde in den KerberosIV Passwort Check Routinen /usr/bin/su und /usr/bin/login entdeckt, das es einem lokalen User möglich macht, jede Datei auf der lokalen Maschine zu überschreiben.

      2. Es ist in telnet möglich spezielle Umgebungsvariablen zu definieren, die dem remote host übergeben werden. Das macht es möglich die Umgebungsvariablen auf der anderen Seite zu setzen, inklusive solcher,die eine spezielle Bedeutung für den Server haben. Es ist bisher jetzt nicht klar, welche Auswirkungen das genau hat, aber wir raten jedem seine Maschinen sofort upzugraden, bzw. zu patchen.

      Es gibt einen Source-Code Patch, der das Problem beseitigt.

    • 005: SECURITY FIX: 4. Dez 2000
      OpenBSD 2.8's ftpd enthält einen 'one-byte overflow' in der replydirname() Funktion.
      Es gibt einen Source-Code Patch, der das Problem beseitigt.
      Den passenden OpenBSD-Verhaltens-Hinweis findest du hier.

    • 004: RELIABILITY FIX: 17. Nov 2000
      Zuerst waren AES (rijndael) für Ver- und Entschlüsselung bei IPSec und Swap Verschlüsselung kaputt.
      Danach arbeitete der AES Code nicht sauber auf "Big endian" Maschinen.
      Es gibt eine zweite Version eines Source-Code Patch, der das Problem beseitigt.

    • 002: IMPLEMENTATION FIX: 10. Nov 2000
      Die skey Unterstützung in ssh(1) für das SSH1 Protokoll war kaputt. Manche Leute mögen das als wichtig einschätzen.
      Es gibt einen Source-Code Patch, der das Problem beseitigt.

  • i386

    • 022: SECURITY FIX: 2. März 2001
      Die USER_LDT Kernel Option erlaubt einem Angreifer Zugriff zu privilegierten Bereichen des kernel memory zu erlangen. Diese Option ist standardmässig deaktiviert. Es gibt einen Source-Code Patch, der das Problem beseitigt.

    • 015: STABILITY FIX: 22. Dez 2000
      Manche Maschinen hängen sich beim Versuch auf, die Maus im Konsolen-Modus zu benutzen. Dieser Patch löst das Problem.
      Es gibt einen Source-Code Patch, der das Problem beseitigt.

    • 006: STABILITY FIX: 4. Dez. 2000
      Auf manchen Maschinen führt ein "PCIBIOS device driver interrupt allocation bug" mölicherweise zu einem Hängen des Kernels w&aul;hrend des PCI-Geräte Prüfens. Wenn du dieses Sympton bei deinem Rechner feststellst, kannst du das PCIBIOS vorübergehend deaktivieren. Dazu
      • Gehe in die "User Kernel Configuration", in dem mit der "boot -c" Option bootest.
      • Am UKC> Prompt gib folgendes ein:
        
       UKC> disable pcibios
       UKC> quit
      • Sieh dir die FAQ 5.6 nach einem erfolgreichen Booten an, damit du deinen Kernel dauerhaft ändern kannst.
