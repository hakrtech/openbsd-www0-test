  • 目標

    OpenBSD 相信牢固的安全是必要的。我們的願景是在業界成為安全領域的桂冠(如果 目前還不是的話)。我們的開放軟體發展模式，使得我們不會像 Sun, SGI, IBM, HP 及其它廠商，可以在安全方面採用更不妥協，有所堅持的觀點。我們可以做到這些 廠商所無法達成的改進。並且由於 OpenBSD 是伴隨密碼系統 輸出的，我們可以利用密碼方法來解決所面對的安全問題。

  • 完全公開

    如同許多 BUGTRAQ 電子 郵件論壇的讀者，我們堅信所發現的安全問題需要完全公開。在作業系統的領 域，我們大概是第一個採用這項觀點的。許多的廠商、甚至包括自由軟體廠商，仍 然試著對他們的使用者隱藏已知的安全問題。

    安全資訊在 cracker 圈子裡是散播得很快的。從另一方面來說，我們的經驗也發 現，撰寫程式碼並發佈適當的安全修正一般來說只需一個小時的工作時間 -- 要非 常快速的得到修正是可達成的。因此我們認為，完全公開的方針可以幫助那些真正 在乎安全問題的人。

  • 稽核程序

    我們的安全稽核團隊通常會有六到十二位成員，他們會持續的搜索並修正所發現的 安全漏洞。我們的稽核工作始自一九九六年。我們所遵循的稽核的程序，是很單純 的針對每個檔案做全面性的分析檢查每一個軟體元件。我們並不是特意的尋找安全 漏洞，而是更專注在尋找基本的軟體暇疵上，因此，假如多年以後，某人又發現了 這個暇疵可能造成的安全漏洞，我們早在修正軟體暇疵時免於這個安全漏洞的威脅。 系統上的各個層面都曾經被發現有漏洞。我們的稽核程序中，也發現了一種全新安 全問題，而且稽核過的原始程式碼通常也會因為新種類的安全問題需要重新稽核。 程式碼常常都會由擁有不同稽核技術的不同專家，重覆的進行稽核檢查。

    我們的安全稽核成員中有些人來自於 Secure Networks 公司，這家公司寫出了業 界第一的網路安全掃描軟體 Ballista(Secure Networks 已經被 Network Associates 併購，而 Ballista 也被更名為 Cybercop Scanner ...)。這家公司在安全領域的 研究上投入了許多，也因此與 OpenBSD 是採相同的立場。OpenBSD 自發起的第 一天就能完全通過所有 Ballista 執行的測試。

    另一方面來說，我們的安全稽核也是相當積極的。在大部份的情況中，我們認為 重點不在一個漏洞是否能被破解。在持續的稽核程序中，我們發現了許多軟體暇 疵，即使未知一個漏洞是否能被破解，我們仍努力的修正這些問題。我們已經修 正了許多簡單而明顯大意的程式設計疏失，並且常常在數個月後發現這些漏洞 事實上是可以破解的。(因此，就會像是某人在 BUGTRAQ 發布會對其他作業系統造成安全弱點某一項 "新發現的安全問題"，在 OpenBSD 的上一個正式發行版本中，已經修正相關的安全弱點)。在其他的情況中，我們 已經修正在一個需要環環相扣的複雜步驟進行破解的漏洞中的其中一步，而避免 了所可能造成的威脅。一個明顯彰顯稽核成功的案例就是 Secure Networks 發 佈的 lpd 安全通報。

  • 報償

    我們積極的稽核程序真的有代價。諸如 "這個安全問題在六個月之前就被修正了" 的聲明常常出現在 BUGTRAQ 。 我們最密集的安全稽核在 OpenBSD 2.0 正式發行版之前、以及在 2.0 版到 2.1 版之間發生在一九九六年第三季以及一九九七年的上半年。在這將近一年的期間裡， 我們迅速的修正了數千(沒錯，真的是數千)個安全問題；從一般的 buffer overflows ，到協定實作的缺陷，information gathering, filesystem races 都包括在內。 我們所遇到的大部份安全問題都在 2.1 版正式發行前修正完，另外需要修正的 一小部份在 2.2 版正式發行前修正完畢。自此以後，我們再也沒有找到這個數 目更多的安全問題，問題的數量很理所當然的遞減下來。最近我們所發現並修正 的通常是更難以發覺得、也更複雜的安全問題。基於某些理由，我們仍然會持續進 行安全稽核的工作:

    • 喔! 我們偶而會發現先前遺漏的簡單的問題。
    • 安全問題就像軍備競賽一樣；最強的攻擊者會持續的搜尋更精巧的破解方法， 相對的我們也需要更好的安全防護。
    • 在一個複雜的軟體當中找尋並修正難解的安全漏洞是相當有趣的。

    這個稽核程序尚未結束，而如你所見的，我們會持續找尋並修正新的安全漏洞。

  • "安全的預設安裝"

    為了讓 OpenBSD 的初學者免於需要在確保安全前，就學習龐雜的安全知識(雖然一 般的廠商預設立場使用者都已知這些安全知識)，我們在發行的作業系統上是採用 安全的預設安裝。所有非必要的服務都被暫時停用。當使用者/系統管理者對整個 系統較熟以後，他將會發現如何啟動各種系統提供的服務。在學習如何啟動新服務 的過程中，初學者將更有機會學習到啟動服務所需考量的安全問題。

    這完全跟漸成風潮預設啟動各種 NFS, mountd, 網頁伺服器和其他各式各樣服務 ，讓使用者在剛安裝完後就伴隨安全威脅的作業系統有著天壤之別。

  • 檢視更新記錄

    既然我們對安全問題採取了積極的態度，我們會持續的找尋並修正新的安全問題。 並非所有的安全問題都會被廣泛的通報，因為(如同前面所說的)裡面有許多並不確 定是能被破解的；許多我們修正的簡單軟體暇疵最終都被證明是我們無法預知的安 全漏洞。我們並沒有時間來讓這些更新記錄如以上安全通報的形式呈現給大家。

    因此，通常會有一些微不足道的安全問題修正，出現在相對於之前一個正式發行版 本的 current 版本原始程式碼中，我們可作出有限度的保證是，這些被修正的問題 是影響不大並且尚未被證明是可破解的。假如我們發現一個絕對會對安全造成影響 的暇疵，我們會以極快的速度撰寫出相關 patches.

    真正關心安全問題的人可以做下列的事:

    • 假如你瞭解安全問題，請查看我們的原始碼電子郵件論壇 並且留意和安全問題相關的討論。是否可破解是無法證明的，畢竟我們做出的修正 並不會寫著"這是安全問題的修正"的相關 commit 訊息。假如一個問題被證明並且 是一個嚴重的安全問題，我們會在極短的時間內做出相關的 patch 檔案。
    • 除了觀看原始程式碼的更新記錄，你也可以注意 公佈安全問題的電子郵件論壇，其中會通知你每一個 OpenBSD 團隊成員認為可能造成威脅的安全問題，並且也會包括如何使用 patch 檔案修正該安全問題的資訊。
    • 跟隨 current 版本的原始程式碼更新，並學會不時的重新建造整個系統(請仔細 地閱讀 /usr/src/Makefile 檔案)。使用者可以假設任何的 current 版本原始 程式碼目錄，都會比之前的正式發行版本更加安全。然而，要由原始程式碼重新 建造整個系統可能是相當困難的；所有的原始程式碼佔了 300MB 的容量空間， 在每個正式發行版的轉換中，常常也會有問題產生。
    • 為你的硬體平台安裝一份時常更新的搶鮮版。比如我們平均每個星期會有一份 搶鮮版的 i386 平台供你安裝使用。
