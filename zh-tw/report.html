如何回報一個問題

正式釋出版本的問題回報

在回報有關正式釋出版的臭蟲/問題之前, 請依下列檢查項目確認一遍:
  1. 首先檢查有關此正式釋出版的更新 及紀錄.
  2. 接著看看是否有 更新的正式釋出版.
  3. 最後檢查 OpenBSD 版本間的更動部分.

如果沒有類似您要提出的問題, 在您送出一個臭蟲回報之前, 請熟讀 sendbug(1) .

更進一步往下閱讀所需的臭蟲回報型別.

目前開發版本的問題回報

如果您的問題是有關 current 這份原始碼而不是 release 或 stable ,
  1. 最少兩次以上的測試, 分別經過數天更新過的原始碼進行測試.
  2. 請不要回報原始碼目錄編譯問題, 除非一直存在者. 當您遇到這類問題時, 幾乎很有可能都是您的錯誤或是它們正在 被解決中. 為這個計畫做事的成員至少每天一次會 做 make build, 通常一天是好幾次, 為了不同的硬體架構而做.
  3. 記住, anoncvs 伺服器更新的程度是落後於真正的工作原始碼目錄.
  4. 看看OpenBSD 版本間的 更動部份 是否已經有同樣的問題 被提出.
  5. 當建立搶鮮版時我們會特別謹慎. 有時錯誤還是會發生, 我們在此致上歉意. 利用電子郵件來回報會比用錯誤回報 來的更恰當.

如何建立一個問題回報

盡可能提供所知的資訊. 試著指出問題真正的所在. 千萬不要給不明確的指示, 或詳細卻模糊不清的問題, 如"它掛了" 或 "我碰到了奇怪的中斷 問題." 和 IRC 或一些其他論壇的人討論確認這是否 是個新的, 可重複產生的問題等等, 並確認這不是 您個人的問題.

請不要馬上開始解決需要重要工作的問題, 除非您確定您真的瞭解這些問題, 特別是當 我們正在釋出的期間, 我們不能大幅修改程式碼. 如果您正在撰寫重大部分的程式碼, 請檢查各式不同的 論壇以免有人正在解決同樣的問題 (節省重複的努力).

下列項目應該要包含在每一個臭蟲回報中:

  1. 從頭開始重製問題的確實步驟. 這應該要非常完整; 只有指令而沒有參數及其他您提供的資料是不夠的. 如果臭蟲需要一連串特定的事件, 請列出來. 鼓勵您簡短您的範例, 但這不是絕對必須的. 如果臭蟲是可重複產生的, 我們總 會找到方法的.

  2. 您所得到的輸出結果. 請不要說它"不能動" 或 "失敗了". 如果有錯誤訊息, 把它寫出來, 即便您不瞭解 這個訊息. 如果 OpenBSD 因特定錯誤當機, 請寫出來. 如果沒有任何事情發生, 請寫出來. 甚至測試結果 是程式當掉或很明顯的不會發生在我們的測試. 最簡單的就是把終端機輸出 複製下來, 如果可以的話.

    注意: 如果是無可挽回的錯誤, 錯誤訊息可能並沒有包含所有的資訊. 這種情況下, 看看系統記錄檔, 如那些放在 /var/log 下的. 另, 如果您處理的是 有自己記錄檔的程式, 例如 httpd, 檢查 存放記錄檔的地方是否有錯誤 (以 httpd 來說, 是 在 /var/www/logs).

  3. OpenBSD 核心輸出. 您可以從 dmesg 指令得到, 但有可能您的 dmesg 輸出並沒有包含所有 在 /var/run/dmesg.boot 的資訊. 如果這種情況, 把上述兩個資訊都包含進來. 請在所有的臭蟲回報中包含 這個資訊.

  4. 如果您的臭蟲是跟執行第三者軟體有關, 把 所有這個軟體會有的破壞列出. 如果您是用 CVS 或 FTP 的搶鮮版, 把版本的時間日期列出來.

  5. 系統核心當機的追溯. 如果您的核心當掉了, 而且您 是在 ddb> 的提示下, 請提供當機訊息, 及 traceback 和 ps 指令的輸出在您的臭蟲回報中.
    假如, 有某些原因, 無法看到當機訊息, 您可以再次取得 透過 x/s *panicstr 指令.
    這是必要的. 當機回報卻沒有當機訊息, traceback 和 ps 的輸出是沒用的.
    show registers 的輸出也許也有重要性. 您也許可以用boot dump 重新開機, 如此一個核心 映像檔會被用savecore(8)存下來 以供更進一步的事後除錯.

  6. 如果您是在使用 XFree86 的平台上要回報有關 X window 的問題, 除了 dmesg.boot 的資訊外, 請將 /var/log/XFree86.0.log 檔案放在您的回報中.

不用擔心您的臭蟲回報會有點冗長. 這是嚴酷的 事實. 把所有事情一次報告出來總比要我們從您身上 擠出事實要好多了. 另一方面, 如果您的提供的資訊 很多, 才好真正的提問是否有人有興趣往下追查.

最後, 當編寫臭蟲回報時, 請選擇不會被誤解的用詞.

送出臭蟲回報

可能的話, 使用 sendbug(1) 命令從我們的追蹤系統中取得臭蟲的資訊. 您可以密切注意追蹤系統, 在這個網頁. Sendbug 需要您的系統可以正常地送出網路郵件. 如果您 沒辦法在特殊用途的 OpenBSD 機器上使用 sendbug, 請將您的臭蟲回報寄 到 bugs@openbsd.org.

也許您要提的也可以是功能需求, 而不是非要是一個臭蟲. 新功能是可以被接受, 尤其是附上您所建議的新功能的程式碼. 如果是其他人寫出您提的新功能, 有可能會被誤解而您無法認得.

為了某些問題的除錯, 我們必須要有特定的 硬體. 請記得這個計畫的資源是有限的. 您可以捐贈一些硬體設備.

臭蟲回報的種類, 根據可取的順序排列

  1. 可重複的問題, 有解決方案最好.
  2. 可重複的問題, 不侷限在您的硬體/軟體 架構.
  3. 可重複的問題, 限於您的軟體架構.
  4. 可重複的問題, 限於您的硬體架構.
  5. 無法重複產生的問題 -- 或無法預期去重複產生的問題.
