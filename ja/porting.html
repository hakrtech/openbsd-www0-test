OpenBSD の ports のビルド

気に入ったソフトウェアを OpenBSD マシン上でコンパイルしたとしましょう。 そして努力の結果を標準の ports として共有したくなりました。 さて、どうすれば良いのでしょう ?

そのとき最も重要なのは会話することです。まず ports@openbsd.org の人たちに、同じ ports で作業していないかどうか聞いてみましょう。次に、ソフトウェアの 作者に伝えましょう。その際、あなたが見つけた問題点も含めるように します。ライセンス情報が正しくないようなら、伝えましょう。直さないと ports がビルドできないような問題があれば、修正方法を知らせます。 もしも作者たちが Linux 上でのみ開発していて、その他の Unix の世界を 無視しているようなら、彼らの見方を変えるように努力してみましょう。

会話こそ、成功する ports と 徐々に廃れていく ports の違いなのです。

まず、このページにある移植のための情報に目を通してください。そして、 参照ドキュメント、特に OpenBSD への移植に関する チェックリストもチェックしてください。

テストしてください。次に、テストをしてください。最後に、もう一度テストしてください !

そして ports を投稿します。まずディレクトリの gzip 圧縮 tar ボールを 作成しましょう。それから、FTP か HTTP の公開サーバに置いてその場所を ports@openbsd.org 宛に送るか、 あるいは ports を MIME エンコードして同じアドレスに送るかしてください。 どちらでも、都合のよい方を選んでください。

OpenBSD の移植ポリシー

  • OpenBSD では /usr/local/etc/rc.d を使用しません。 /usr/local は、しばしば NFS を利用していくつかの マシン間で共有されていることがあるため、各マシン専用の 設定ファイルを置くことができず、 /etc がマシンごとの設定ファイルすべての置場となっています。 さらに、OpenBSD のポリシーでは、/etc の下のファイルは 決して自動更新されません。起動のために特別な設定を要する ports は、 勝手にファイルをインストールするのではなく、設定手順を管理者に 提案するべきなのです。
  • OpenBSD は man ページを圧縮しません。
  • OpenBSD は -lcrypt を必要としません。
    DES による暗号化は標準 libc の一部となっています。
  • OpenBSD では、ports によって生成されるユーザおよびグループの名前空間を分けています。 詳細は、/usr/ports/infrastructure/db/user.list を参照してください。
  • OpenBSD は強力なセキュリティ指向に立脚しています。このページの セキュリティの章を読んで理解しておいてください。
  • Makefile 中に $OpenBSD$ CVS タグを入れるのを 忘れないようにしてください。また、他のシステムの ports を持ってくる 場合には、そのシステムのタグも残しておくようにしてください。
  • 目標は、すべての移植済アプリケーションを OpenBSD に対応させることです。 これを達成するため、アプリケーションのメンテナに OpenBSD 用のパッチを フィードバックする必要があります。

セキュリティ上の推奨

気を付けるべきセキュリティ上の問題が多数あります。これらについて、 大丈夫だという絶対の確信が持てない場合は、どうぞ ports メーリングリストに援助を求めてください。
  • ports を用意する際に、アルファやベータバージョンのコードは使用しないで ください。最新の通常バージョンかパッチバージョンを使用するようにしてください。
  • サーバとしてインストールされるいかなるソフトウェアに対しても、 バッファオーバーフロー、特に strcat/strcpy/strcmp/sprintf の安全でない使用法について検査を行うべきです。また、一般的に、 sprintf は snprintf に置き換えるべきです。
  • 安易にファイル名を使用して本当のセキュリティを犠牲にしないことです。正当に制御 されていない部分には数多くの競合状態が発生し得ます。たとえば、あなたのマシン上 のユーザ権限を既に手に入れた攻撃者は、/tmp の下のファイルを、 /etc/master.passwd のような、より重要なファイルへの シンボリックリンクに置き換えてしまうかも知れません。
  • たとえば、fopen および freopen のどちらも、書き込み時に 新しいファイルを作成するか既存のファイルをオープンするものです。 攻撃者は、/etc/master.passwd へのシンボリックリンクを /tmp/addrpool_dump として作成するかも知れません。これをオープンした その瞬間に、パスワードファイルはきれいさっぱり消えてしまいます。 そうです、直前に unlink しても同じことです。それは危険性を 小さくするに過ぎません。代わりに、open を O_CREAT|O_EXCL 付きで使って fdopen してください。
  • もうひとつ非常によくある問題として、mktemp 関数があります。 これを使っているときの BSD リンカの警告に注意してください。 これは修正しなければならないものなのです。 s/mktemp/mkstemp/g という単純なものではありません。
    詳しい指示については、 mktemp(3) を参照してください。 mkstemp を使用する正しいコードの例として、 ed や mail のソースがあります。 mktemp を正しく使用する数少ないコードの実例は、 rsync の ports に見ることができます。
  • 読めるからといって、読んでいいということにはなりません。この種の良く知られた穴は startx 問題です。setuid プログラムでありながら、startx は起動時に どのようなファイルでもスクリプトとして受け取ることができました。そのファイルが 正しいシェルスクリプトでなかった時にはシンタクスエラーメッセージが出たのですが、 そこでパーミッションチェックをせずにファイルの最初の行が表示されてしまっていました。 シャドウパスワードファイルは最初の行がしばしば root ユーザのエントリですから、 そこを盗み出せるというのは攻撃者にとって実に便利なことなのです。ファイルをオープン してしまった後で「オープンしても良かったのか」と確認するためディスクリプタを fstat するのではなく (それでは攻撃者が /dev/rst0 でいたずらをすれば テープが巻き戻されてしまいます)、正しい uid/gid/grouplist を使用してオープンしましょう。
  • 特権を落とす前の setuid プログラムからは、いかなるシェルも フォークしてはいけません。これには、popen や system も含まれます。代わりに、fork、 pipe や execve を使用するようにしましょう。
  • 競合状態を避けるため、他のプログラムにはファイル名ではなくオープンした ディスクリプタをわたすようにしましょう。プログラムがファイルディスクリプタを 受け取ることができない場合でも、/dev/fd/0 なら使用できるはずです。
  • アクセス権限はファイルディスクリプタに割り付けられます。ファイルをオープン するために setuid の権限が必要なら、ファイルをオープンした後に特権を落としてください。 オープン済のディスクリプタに対してはまだアクセス可能ですが、不安材料は減ります。 しかし、これは両刃の剣です。特権を落とした後でも、これらの ディスクリプタに注意を払っておく必要があるのです。
  • できる限り、root の setuid は避けてください。基本的に root は何でもできますが、 おそらく、1024 未満の番号のソケットポートを生成する場合を除けば、root 権限が 必要となることは非常に稀なことなのです。そのようなことは、inetd の制御の下に置いておくのがおそらくより良い方法であり、これは単に 関連するエントリを inetd.conf に追加するだけです。 そのためには、デーモンを書くための適正な術を知っておく必要があります。 もし、そのようにするための方法を御存知ないのであれば、setuid プログラムを 書くのはあなたの仕事ではないと言えるかも知れません。
  • setuid の代わりに setgid を使用してください。setgid プログラムが必要とする特定の ファイルを除けば、ほとんどのファイルはグループ書き込み可能にはなっていません。 ですので、setgid プログラムのセキュリティの問題は、あまりシステムの脅威には ならないはずです。ただし、グループの権限でも、最悪の場合には、侵入者はゲームの スコアテーブルをハックしたり、まぁその程度のことができる場合があります。 そのような変更の実例としては、ports の xkobo を参照してください。
  • グループ書き込み可能なファイルを信用してはいけません。 監査されているとしても、setgid プログラムは重大な潜在的セキュリティホールとは 考えられていないのです。よって、そうしたプログラムで改変することのできる情報を 確実なものとみなすべきではありません。
  • 環境変数を信用してはいけません ! これは PATH のような単純なものでも 関係のあることで (決して精査されてない名前で system を使用してはいけませんし、 execvp の使用は避けましょう)、locale、timezone、termcap などのような微妙なもの についても同様です。プログラムが他のプログラムを実行することもあるという点に注意しましょう。 あなた自身が直接呼び出すときにどれほど細心の注意を払ったとしても、そのプログラムが他の プログラムを呼び出すときに注意を払ってくれるとは限りません。決して、 特権を持つプログラム上で system を呼び出してはいけません。コマンドラインを作成し、 制御可能な環境変数を用意した上で、直接 execve を呼び出します。 perlsec の man ページは、このような問題に対する良い手引書です。
  • setuid されたシェルスクリプトは決して使用してはいけません。このようなものは本質的 に安全ではありません。それらは正当な環境変数を保証するための C コードでラップして 使用しましょう。これに対して、OpenBSD には安全な Perl スクリプトがあります。
  • ダイナミックローダにも注意が必要です。setuid プログラムを実行しているなら、 ldconfig がスキャンした信用できるライブラリのみを使用してくれます。 しかし、setgid にはそのような仕組みはありません。
  • ダイナミックライブラリはトリッキーです。C++ のコードにも同様の問題があります。 基本的に言ってライブラリは、メインプログラム自身が setuid をチェックする前 でさえ、その環境に基づいてある種の動きをするかも知れません。 OpenBSD の issetugid は、ライブラリを書く側の観点から、この問題を 解決してくれます。この問題を完全に理解できるまで、ライブラリは 移植しようとしない方が良いでしょう。

一般的な移植のヒント

  • 使うとしても、__OpenBSD__ は慎重に使うべきです。 次のような構文、 
                #if defined(__NetBSD__) || defined(__FreeBSD__)
    は、しばしば適切ではありません。ですので、何も考えずに __OpenBSD__ を追加してはいけません。その代わりに、それが何をしようとしているのか、実際に何が 必要なのかの理解に努めてください。多くの場合、マニュアルページが役立ちます。 特殊な機能がいつ BSD に組み込まれたかなどの記述や歴史的なコメントを含んでいるからです。 知られているリリース番号と BSD の数値を比較する のが正しいことも多いでしょう。より詳しい情報は、 NetBSD パッケージガイド を参照してください。
  • BSD を定義するのは良い考えではありません。sys/param.h をインクルードしてみてください。そこには BSD が定義されているだけ ではなく、適正な値も与えられています。正しいコード例の一部を以下に示します。 
    
           #if (defined(__unix__) || defined(unix)) && !defined(USG)
           #include <sys/param.h>
           #endif
  • OS の種類ではなく、機能の有無をテストしてください。長い目で見れば、 tcgetattr が動作するかどうかをテストするほうが、BSD 4.3 以降 や SystemV R4 で実行しているかどうかをテストするより、とても有用です。 後者のテストは、ただ問題を複雑にするだけです。行くべき道は、たとえば、 ある特定のシステムをテストするために HAVE_TCGETATTR のような 定義を多数用意して、それから次のシステムのテストに進むというものです。 この技法は特定の OS から機能テストを分離することができるものです。 そして容易に別の移植者が -DHAVE_XXX の全セットを Makefile 中に 加えることもできるようになります。また、ある人は、configure スクリプトに その機能をチェックし、そして自動的にその機能を追加するためのスクリプトを 追加したり書き起こしたりすることもできるのです。真似してはいけない例として nethack 3.2.2 のソースをチェックしてみてください。システムのタイプに基づいて、 ロードするものを仮定するようになっています。そして、このような仮定のほとんどは 既に時代遅れで実態を反映していません。POSIX 関数の有無は、昔の BSD と SystemV との違い以上に役立つものであり、伝統的な BSD 関数は今や互換ライブラリを 通してサポートされるだけと言えるほどなのです。
  • 他のインクルードファイルをインクルードするインクルードファイルをインクルードする... というのは避けてください。まず、これは非効率的ですし、やがてこのプロジェクトは、 何もかもをインクルードするファイルをインクルードするようになってしまうでしょう。また、 解決が困難な問題を起こしたり、ある場所で特定のファイルをインクルードしない ようにするのが困難になってしまったりします。
  • マクロを使用した奇抜なトリックは避けてください。ヘッダファイルで定義されたマクロを 未定義にするのは良い考えではありません。また、インクルードファイルの特定の挙動の ためだけにマクロを定義することも良い考えではありません。コンパイルモードは グローバルであるべきなのです。POSIX の挙動にしたい場合にはそう宣言しましょう。 ただし、そのようにしたいと思った時だけではなく、プロジェクト全体を通して #define POSIX_C_SOURCE を定義してください。
  • システム関数のプロトタイプ宣言は絶対に書かないでください。OpenBSD を含む、現代的な すべてのシステムでは、このようなプロトタイプ宣言を標準的な場所に持っているからです。 これらはたとえば、unistd.h、fcntl.h や termios.h といった場所のことです。 man ページには、このようなプロトタイプ宣言がどこにあるのか繰り返し書かれています。 また、これとは別に、正しいファイルを得るための HAVE_XXX マクロが大量に 必要になるかも知れません。インクルードファイルは、あらゆる厄介ごとを防止する 仕組みを持っていますので、同じファイルを二度インクルードしても心配はありません。
    ある種の壊れたシステム用にプロトタイプ宣言を書かなければならないかも知れませんが、 他のシステムには見せないようにしてください。手作りのプロトタイプ宣言は (size_t の 代わりの unsigned long のように)、あなたのシステム上では等価な型を使用している かも知れませんが、他のシステムには移植できないかも知れませんし、いくつかの const のステータスが間違っているかも知れないなど、いずれ破綻することになるでしょう。 また、OpenBSD に附属する gcc のようなコンパイラでは、正しいヘッダ ファイルが使用されていれば、strlen のような、いくつかの非常に頻繁に 使用される関数で、より良い性能を発揮したりします。
  • 標準システム関数の名前を他のどのようなものにも使用してはいけません。 独自の関数が必要なら、独自の名前を使用して、それを各所で呼び出すように しましょう。そうしておけば、デフォルトのシステム関数に戻したくなった場合には、 独自関数のコードをコメントアウトした上で、その独自の名前をシステム関数に #define で定義し直すことが必要になるだけです。これ以外の方法では行わないで ください。このコードの例は、以下のようなものになるでしょう。 
    
       /* prototype part */
       #ifdef USE_OWN_GCVT
       char *foo_gcvt(double number, size_t ndigit, char *buf);
       #else
       /* include correct file */
       #include <stdlib.h>
       /* use system function */
       #define foo_gcvt  gcvt
       #endif

       /* definition part */
       #ifdef USE_OWN_GCVT
       char *foo_gcvt(double number, size_t ndigit, char *buf)
          {
          /* proper definition */
          }

       /* typical use */
       s = foo_gcvt(n, 15, b);

その他の役に立つヒント

  • 最近のバージョンの bsd.port.mk はインストーラのパスを セットします。特に、/usr/bin と /bin が /usr/local/bin や /usr/X11R6/bin よりも 先に検索されるようになっています。
  • ${NO_SHARED_LIBS} が yes に設定されている場合には、共有ライブラリを 生成しないでください (これは bsd.port.mk のインクルード後 にのみ定義可能であることに注意してください)。GNU configure を持つ ports の場合は、 単に Makefile に CONFIGURE_ARGS += ${CONFIGURE_SHARED} という行を追加するだけです。
  • ユーザは bsd.port.mk を含めて ports ツリー全体を 更新するでしょうから、bsd.port.mk の最近の バージョンで出た機能に依存しても構いません。 NEED_VERSION は現在は使用されなくなりました。
  • パックリストを生成・更新する時には、手動でやるのではなく update-plist を使いましょう。 入れたくないものはコメントアウトすることができます。 ほとんどの場合、update-plist はファイルの種類を検出して、 必要な情報を正しくコピーしてくれます。
  • 動作がおかしいスクリプトや makefile などを検出するために、 /etc/mk.conf に USE_SYSTRACE=Yes を追加しましょう。
  • OpenBSD では、curses.h/libcurses/libtermlib が ``new curses'' ですので、以下のように変更してください。
    ncurses.h ==> curses.h
    curses.h をインクルードする前に、(通常は Makefile 中で) _USE_OLD_CURSES_ を定義し、-locurses としてリンクを行うことによって、 ``old (BSD) curses'' を利用することもできます。
  • OpenBSD では、ターミナルディシプリンが、以前の BSD sgtty から、 新しい POSIX tcgetattr ファミリにアップグレードされています。 新しいコードでは、古いスタイルを避けてください。いくつかのコードでは、 古い sgtty の同義語として tcgetattr を定義している かも知れませんが、これは OpenBSD では、せいぜい間に合わせの手段でしかありません。 xterm のソースコードは、してはいけないことを非常に良く示す 例です。システムの機能性を正しい状態にするよう努力しましょう。そのためには ターミナルの状態を記憶しておくための型 (おそらく typedef) が必要ですし、 現在の状態を取得するための関数や、新しく状態を設定するための 関数も必要になるでしょう。 ターミナルの状態を変更する関数の実装はとりわけ困難です。そうした 関数はシステムによって異なるものだからです。また、ターミナルに 接続していない場面についても扱わなくてはならないことや、終了 だけではなくバックグラウンドのシグナル (SIGTSTP) も扱う必要があるということを忘れないでください。いつ終了するに しても、ターミナルを健全な状態に戻すべきです。sh のように、 プログラムが終了しても一切ターミナルを初期化しないような 古いシェルでテストを行ってください。
  • OpenBSD に含まれている新しい termcap/terminfo および curses は、 カラーターミナルを制御するための標準的なシーケンスを含んでいます。 可能ならこれらを使用すべきであり、他のシステムでは標準の ANSI カラー シーケンスに戻してください。しかし、いくつかのターミナルの記述はまだ 更新されていませんので、これらのシーケンスを手動で指定する必要がある かも知れません。これは vim に実装されている方法でもあります。これは、 厳密には必要なことではありません。特権を持つプログラムを除いて、一般的には TERMCAP 環境変数を通して termcap の定義を上書きすることが 可能であり、それによって正しい動きをさせることが可能になります。
  • ややこしいことに、シグナルの用法はシステムによって違います。 sigaction およびその man ページで参照されている他の システムコールを使って、確実に意図通りのシグナルになるようにしてください。
