2.1 - Páginas Web Interessantes

A página web oficial do projeto OpenBSD está localizada em: http://www.OpenBSD.org.

Muitas informações valiosas, que abordam todos os aspectos do projeto OpenBSD, podem ser encontradas aqui.

O Jornal do OpenBSD é um site de notícias e opiniões focado no OpenBSD.

O site OpenBSDsupport.org é uma coleção de documentação "mantida por usuários", de qualidade variada, que cobre tópicos que não estão neste FAQ ou outra documentação oficial.

Muitos usuários criam sites e páginas com informações específicas sobre o OpenBSD. Como tudo na Internet, uma boa ferramenta de pesquisas irá facilitar sua vida, mas com uma dose saudável de cepticismo. Como sempre, nunca digite comandos cegamente (sem saber o que eles fazem) no seu computador.

2.2 - Listas de Discussão

O projeto OpenBSD mantém muitas listas populares de discussão, nas quais usuários podem inscrever-se e segui-la. Para se inscrever numa lista de discussão, mande uma mensagem de e-mail para majordomo@openbsd.org. Esse endereço é um serviço de inscrição automático. No corpo da sua mensagem, em uma única linha, você deve incluir um comando de inscrição para a lista que você deseja se juntar. Por exemplo: 

subscribe announce

O processador de e-mail da lista vai te responder, perguntando pela confirmação da sua intenção de se juntar à lista, desse modo outros não podem inscrever você e te inundar de e-mails não desejados. A mensagem incluirá instruções de vários jeitos para confirmar, incluindo uma página web com links de servidores de listas. Responda a mensagem de confirmação ou responda para majordomo@openbsd.org. Use qualquer método que é conveniente para você. Você notará que todas as três técnicas envolvem um número de identificação único e com tempo limitado, tal como A56D-70D4-52C3, para ter novamente a certeza que você é realmente a pessoa que está requisitando essa inscrição na lista de discussão (esse é um "opt-in" real).

Após ter confirmado sua intenção de se inscrever, você será imediatamente adicionado à lista, e o processador de e-mail irá lhe notificar que você foi adicionado com sucesso.

Para se desinscrever da lista, você deve novamente enviar uma mensagem de e-mail para majordomo@openbsd.org. Com algo parecido com isto: 

unsubscribe announce

Se você tiver qualquer dificuldade com o sistema de lista de discussão, por favor, primeiro leia o arquivo de ajuda que pode ser obtido enviando uma mensagem de e-mail para majordomo@openbsd.org com a mensagem de corpo "help".

Sua inscrição nas listas de discussão do OpenBSD também pode ser mantida através da interface web em http://lists.openbsd.org

Algumas das mais populares listas de discussão do OpenBSD são:

  • announce - Anúncios importantes. Essa é uma lista de tamanho pequeno.
  • security-announce - Anúncios de problemas de segurança. Essa é uma lista de tamanho pequeno.
  • misc - Perguntas e respostas gerais dos usuários. Essa é a lista mais ativa, e deve ser a lista "padrão" para a maioria das questões.
  • bugs - Erros recebidos via sendbug(1) e discussões sobre eles.
  • source-changes - Lista que recebe automaticamente as mudanças na árvore de fontes do CVS. Toda vez que um desenvolvedor modifica ("commit") a árvore de fontes do OpenBSD, o CVS enviará uma cópia (geralmente curta) de sua mensagem "commit" a essa lista.
  • ports - Discussão da árvore de ports do OpenBSD.
  • ports-changes - Lista que recebe automaticamente as mudanças na árvore de fontes CVS específica do ports.
  • advocacy - Discussão sobre a promoção do OpenBSD e os tópicos que são muito "off-topic" para misc.

Antes de postar uma pergunta na misc ou qualquer outra lista de discussão, por favor, verifique os arquivos, as questões mais comuns já foram respondidas repetidamente. Enquanto provavelmente será a primeira vez que você encontrou um problema ou uma pergunta, outros nas listas de discussão podem ter visto a mesma pergunta várias vezes na última semana, e não gostam de vê-la novamente. Se você fazer uma pergunta possivelmente referente a hardware, sempre inclua um dmesg(8)!

Você pode encontrar vários arquivos, outros guias de listas de discussão e mais informações na página de listas de discussão.

Uma lista de discussão não-oficial que pode ser de interesse para novos usuários de OpenBSD e Unix é a lista Iniciantes no OpenBSD.

2.3 - Páginas de Manual

O OpenBSD vem com uma extensa documentação em forma de páginas de manual, como também em longos documentos relativos às aplicações específicas. Um esforço enorme é feito para ter certeza que as páginas de manual estão atualizadas e precisas. Em todos os casos, as páginas de manual são consideradas fontes de informação confiável do OpenBSD.

Para acessar as páginas de manual e outras documentações, tenha certeza que você instalou os conjuntos de arquivos man44.tgz e misc44.tgz.

Esta é uma lista de algumas das mais úteis páginas de manual para novos usuários:

Começando

  • afterboot(8) - coisas para checar depois da primeira inicialização completa.
  • help(1) - ajuda para novos usuários e administradores.
  • hier(7) - layout do sistema de arquivos.
  • man(1) - mostra as páginas de manual on-line.
  • intro(1) - introdução aos comandos gerais, também veja as intros para as outras seções do manual: intro(2), intro(3), intro(4) (nota: intro(4) é específica de plataforma), intro(5), intro(6), intro(7), intro(8), e intro(9).
  • adduser(8) - comando para adicionar novos usuários.
  • vipw(8) - edita o arquivo de senhas mestre.
  • disklabel(8) - leitura e escrita do disklabel.
  • reboot, halt(8) - pára e reinicia o sistema.
  • shutdown(8) - desliga o sistema em um dado tempo.
  • dmesg(8) - mostra novamente as mensagens do momento de boot do kernel.
  • sudo(8) - não loga como root, mas executa comando como root.
  • mg(1) - editor de texto compatível com o emacs.
Para usuários mais avançados
  • boot(8) - procedimentos de "bootstrapping" do sistema.
  • boot_config(8) - como mudar a configuração do kernel no boot.
  • gcc_local(1) - modificações específicas do OpenBSD no gcc(1)
  • ifconfig(8) - configura os parâmetros da interface de rede.
  • login.conf(5) - formato do arquivo de configuração da classe de login.
  • netstat(1) - mostra o status da rede.
  • release(8) - compilação de uma release do OpenBSD.
  • sendbug(1) - envia um relato de problema (PR - "problem report") sobre o OpenBSD para o site de suporte central.
  • style(9) - guia de estilo para o código fonte do kernel do OpenBSD.
  • sysctl(8) - obter ou configurar parâmetros do kernel.

Você pode achar todas as páginas de manual do OpenBSD na web em http://www.openbsd.org/cgi-bin/man.cgi e também no seu computador, se você instalou o conjunto de arquivos man44.tgz.

Geralmente, se você conhece o nome de um comando ou página de manual, você pode ler a mesma executando "man comando". Por exemplo: "man vi" para ler sobre o editor vi. Se você não sabe o nome do comando, ou se "man comando" não encontra o página de manual, você pode procurá-la na base de dados, executando "apropos qualquer_coisa" ou "man -k qualquer_coisa", onde "qualquer_coisa" é uma palavra comum que talvez apareça no título da página de manual que você está procurando. Por exemplo: 

# apropos "time zone"
tzfile (5) - time zone information
zdump (8) - time zone dumper
zic (8) - time zone compiler

Os números entre parênteses indicam a seção do manual onde a página se encontra. Em alguns casos, você pode encontrar páginas de manual com nomes idênticos vivendo em seções separadas do manual. Por exemplo, assuma que você quer saber o formato dos arquivos de configuração do daemon cron. Uma vez que você sabe a seção do manual da página que você quer, você vai executar "man n comando", onde n é o número da seção do manual. 

# man -k cron
cron (8) - clock daemon
crontab (1) - maintain crontab files for individual users
crontab (5) - tables for driving cron
# man 5 crontab

Em adição às páginas de manual UNIX, existe um conjunto de documentos imprimíveis (incluídos no conjunto de arquivos misc44.tgz). Eles residem no diretório /usr/share/doc. Você pode formatar cada documento desse conjunto com um "make" no subdiretório apropriado. O subdiretório psd contém a documentação "Programmer's Supplementary Documents" (Documentos Suplementares para o Programador). O subdiretório smm é o do "System Manager's Manual" (Manual de Gerenciamento do Sistema). O subdiretório usd contém a documentação"UNIX User's Supplementary Documents" (Documentos Suplementares do Usuário de UNIX). Você pode dar seu "make" na árvore dos subdiretórios, ou você pode selecionar uma seção específica do conjunto e rodar o `make' em seu subdiretório.

Alguns dos subdiretórios estão vazios. Por padrão, formatar os documentos resultará na saída de PostScript, adequado para a impressão. A saída de PostSript pode ser um pouco grande -- você deve supor um aumento de 250-300% no tamanho. Se você não tem acesso a uma impressora ou leitor de PostScript, você pode também formatar os documentos para leitura no terminal. Cada subdiretório tem um alvo para criar cópias ASCII desses papéis (chamado `paper.txt') que podem ser gerados com o make(1). Por exemplo: 

# cd /usr/share/doc/usd/04.csh
# make paper.txt
# more paper.txt

Note que privilégios do superusuário podem ser requeridos para construir os documentos nesses diretórios, e que o make clean vai remover quaisquer papéis gerados por um make anterior. Veja /usr/share/doc/README para mais detalhes sobre os documentos em /usr/share/doc/.

As páginas de manual UNIX são geralmente mais atuais e confiáveis que os documentos imprimíveis. Os documentos imprimíveis algumas vezes explicam aplicações complicadas em mais detalhes do que as páginas de manual fazem.

Para muitos, ter uma cópia em papel da página de manual pode ser útil. Aqui estão algumas dicas para fazer uma cópia imprimível de uma página de manual.

Como eu vejo uma página de manual a partir do arquivo fonte (i.e. um nome de arquivo terminado em um número, como tcpdump.8)?

Essas páginas são encontradas na árvore src. As páginas de manual são encontradas na árvore sem um formato, e muitas vezes, através do uso do CVS, elas serão atualizadas. Para ver essas páginas, simplesmente digite: 

# nroff -Tascii -mandoc <arquivo> | more

Como eu obtenho uma página de manual plana, sem caracteres de formatação ou controle?

Isto é útil para se obter uma página de manual correta, sem caracteres não-imprimíveis.
Exemplo: 

# man <comando> | col -b

Como eu posso obter uma cópia PostScript, pronta para imprimir, de uma página de manual?

Note que <arquivo> precisa ser o arquivo fonte da página de manual (provavelmente um arquivo que termina em um número, e.g. tcpdump.8). A versão em PostScript da página de manual tem uma aparência muito boa. Eles podem ser imprimidos ou visualizados na tela com um programa como gv (GhostView). O GhostView pode ser encontrado em nossa coleção de pacotes. Use as seguintes opções do comando nroff(1) para obter a versão PostScript de uma página de manual do OpenBSD: 

# nroff -Tps -mandoc <arquivo> > outfile.ps

Como eu gero cópias comprimidas das páginas de manual?

Para as pessoas que estão construindo seu sistema a partir do código fonte, existem várias opções relativas ao modo que as páginas de manual são construídas. Essas opções podem ser colocadas em /etc/mk.conf (pode ser necessário criar esse arquivo) e são usadas durante a construção do sistema. Uma opção especialmente útil é gerar páginas de manual comprimidas para economizar espaço em disco. Estas podem ser visualizadas normalmente, usando o comando man. Para configurar isso, adicione em /etc/mk.conf o seguinte: 

MANZ=yes
Outra opção útil permite gerar páginas de manual no formato PostScript, como também em texto ASCII. Isso é feito definindo a opção MANPS=yes no /etc/mk.conf. Veja mk.conf(5) para mais detalhes.

O que são arquivos info?

Algumas das documentações do OpenBSD vêm na forma de arquivos info, tipicamente armazenados em /usr/share/info. Essa é uma forma alternativa de documentação fornecida pelo projeto GNU. Muitos desses arquivos são mais atualizados que as páginas de manual fornecidas pelo projeto GNU, e podem ser acessados com o comando info(1). Por exemplo, para ver informações sobre o compilador GNU, gcc(1), digite: 

# info gcc
Depois de usar o info, você irá realmente apreciar nossas páginas de manual!

Como eu obtenho páginas de manual coloridas no XTerm?

O arquivo de configuração padrão para o xterm(1) não mostra páginas de manual coloridas. Para obter uma saída colorida, copie o arquivo /etc/X11/app-defaults/XTerm-color para o seu diretório home, e o renomeie para ".Xdefaults". Tome cuidado para não sobrescrever qualquer configuração atual no ".Xdefaults". Esse arquivo contém todas as configurações que você precisa para ativar as cores no XTerm. No entanto, três linhas precisam ser descomentadas para isso funcionar: 

!*VT100*colorULMode: on
!*VT100*underLine: off
!*VT100*colorBDMode: on
O resto desse arquivo permite que você escolha cores para várias configurações. As únicas relevantes às páginas de manual são: 
*VT100*colorUL: yellow
*VT100*colorBD: white
Isso gera uma aparência terrível nas páginas de manual, então customize o quanto for necessário: nós podemos sugestionar vermelho para "colorUL" e magenta para "colorBD"? Existe também um visualizador de páginas de manual para o X11, xman(1), o qual providencia uma interface alternativa (gráfica) para as páginas de manual. Veja as páginas de manual do xterm e xman para mais informações.

Como eu escrevo minha própia página de manual?

Se você deseja escrever sua própia página de manual para um aplicação que você escreveu, um tutorial é providenciado em mdoc.samples(7). Existe também um guia de referência útil providenciado em mdoc(7).

2.4 - Relatando Erros

Antes de gritar "Erro!", por favor tenha certeza que é realmente o caso. Caso contrário, você não está entendendo como algumas coisas são feitas no OpenBSD ou como ele funciona, e não procurou resolver o problema usando as páginas de manual ou o website do OpenBSD. Use as listas de discussão (normalmente a misc@openbsd.org) para pedir ajuda. Se essa é sua primeira experiência com OpenBSD, seja realista: você provavelmente não descobriu um erro desconhecido. Também note que hardware defeituoso pode parecer um erro de software, por favor verifique a condição atual do seu hardware antes de decidir que você encontrou um "bug".

Finalmente, antes de submeter qualquer relato de erro, por favor leia http://www.openbsd.org/report.html.

Uma informação sobre um erro é uma das mais importantes responsabilidades dos usuários finais. Uma informação detalhada é requerida para diagnosticar a maioria dos erros sérios. Desenvolvedores frequentemente recebem relatos de erros via e-mail tal como este: 

From: joeuser@example.com
To: bugs@openbsd.org
Subject: HELP!!!

I have a PC and it won't boot!!!!! It's a 486!!!!!

Felizmente a maioria das pessoas entendem que tais relatos são sumariamente deletados. Todos os relatos de erros devem conter informação detalhada. Se Joe User estivesse realmente esperando alguém para ajudá-lo a encontrar esse erro, ele ou ela deveria fornecer mais informações... algo como isto:

Veja report.html para mais informações sobre criação e submissão de relatos de erro. Informações detalhadas sobre seu hardware são necessárias se você acha que o erro pode estar de algum modo relacionado com seu hardware ou a configuração dele. Normalmente, a saída do dmesg(8) é suficiente nesse respeito. Uma descrição detalhada do seu problema é necessária. Você irá notar que o dmesg descreve o hardware, o texto explica o porquê que Smart User pensa que o seu sistema não está defeituoso, (3.2 funciona corretamente), como esse crash foi causado (iniciando o X), e a saída dos comandos "ps" e "trace" do debugador. Nesse caso, Smart User providenciou a saída capturada em um console serial; se você não pode fazer isso, você vai ter que usar papel e caneta para escrever o crash. (Esse foi um problema real, a informação no relato acima ajudou a levar à correção desse problema que afetava sistemas Sun4c.)

Se Smart User tivesse um sistema OpenBSD funcionando, a partir do qual ele submeteria um relato de erro, ele deveria usar o utilitário sendbug(1) para submeter esse relato de erro ao GNATS, o sistema de rastreamento de erros. Obviamente, você não pode usar o sendbug(1) quando seu sistema não inicia, mas você deve usar ele sempre que for possível. Você vai precisar incluir informação detalhada sobre o que aconteceu, a configuração exata do seu sistema, e como reproduzir o problema. O comando sendbug(1) requer que seu sistema esteja configurado para enviar correio eletrônico corretamente na internet. Note que o servidor de mensagem usa spamd(8) baseado em "greylisting", então pode levar meia ou uma hora até que o servidor de mensagem aceite seu relato de erro, então por favor seja paciente.

Depois de submeter um relato de bug via sendbug(1), você será notificado por e-mail sobre o status do seu relato. Você pode ser contatado por desenvolvedores para informação adicional ou com patches que precisam de testes. Você pode também monitorar os arquivos da lista de discussão bugs@openbsd.org, detalhes na página de listas de discussão, ou consultar a base de dados dos relatos de erros no Sistema de Rastreamento de Erro on-line.

Mais sobre como obter informações úteis para os desenvolvedores

Aqui estão algumas dicas adicionais:

Perdeu a "Panic message"?
Sob certas circunstâncias, você pode perder a primeira mensagem de pânico, determinando a razão para o pânico. Essa é uma mensagem muito importante, então você precisa relatá-la também. Você pode obter isso novamente, usando o comando "show panic" no ddb>, como isto: 

ddb> show panic
0:      kernel: page fault trap, code=0
ddb>

Nesse caso, a frase de pânico foi "Kernel: page fault trap, code=0"

Nota especial para sistemas SMP:
Você deve obter um "trace" de cada processador como parte do seu relato de erro: 

ddb{0}> trace
