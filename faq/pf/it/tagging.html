Introduzione

Il tagging di un pacchetto è un modo per marcare i pacchetti con un identificatore interno che può essere successivamente usato nelle regole di filtraggio e traslazione. Con il tagging è possibile fare determinate cose come creare interfaccie fidate e determinare se i pacchetti sono stati processati da regole di traslazione. E' anche possibile passare da un filtraggio basato sulle regole a un filtraggio basato sulle policy.

Assegnazione dei tag ai pacchetti

Per aggiungere un Tag a un pacchetto, usare la keyword tag:
pass in on $int_if all tag INTERNAL_NET keep state

La tag INTERNAL_NET sarà aggiunta a ogni pacchetto che soddisfa la regola precedente. Da notare l'uso del keep state; keep state (o modulate state/synproxy state) devono essere usate in regole pass che effettuano il tag dei pacchetti.

Una tag può anche essere assegnata usando una macro. Per esempio:

name = "INTERNAL_NET"
pass in on $int_if all tag $name keep state

Ci sono un set di macro predefinite che possono anche essere usate.

  • $if - L'interfaccia
  • $srcaddr - Indirizzo IP sorgente
  • $dstaddr - Indirizzo IP destinazione
  • $srcport - Porta sorgente
  • $dstport - Porta destinazione
  • $proto - Il protocollo
  • $nr - Il numero di regola

Queste macro sono espanse nell'instante in cui vengono caricate le regole di configurazione e non in tempo reale.

Il tagging segue queste regole:

  • I tag sono "incollati". Dall'istante in cui un tag viene applicato a un pacchetto da una corrispondenza di regola, non può più essere rimosso. Tuttavia può essere sostituito con un tag differente.
  • A causa dell'"attaccamento" del tag, un pacchetto può avere un tag anche se l'ultima regola che mostra una corrispondenza non usa la keyword tag.
  • A un pacchetto è assegnato un massimo di un tag per volta.
  • I tag sono identificatori interni. I tag non vengono inviati su cavo.

Consideriamo le seguenti regole come esempio:

(1) pass in on $int_if tag INT_NET keep state
(2) pass in quick on $int_if proto tcp to port 80 tag \
        INT_NET_HTTP keep state
(3) pass in quick on $int_if from 192.168.1.5 keep state

  • Ai pacchetti provenienti da $int_if sarà assegnato il tag INT_NET dalla regola #1.
  • Ai pacchetti TCP provenienti da $int_if e destinati alla porta 80 sarà per prima cosa assegnato un tag INT_NET dalla regola #1. Quel tag sarà sostituito con un tag INT_NET_HTTP dalla regola #2.
  • I pacchetti provenienti dal'interfaccia $int_if dall'indirizzo 192.168.1.5 saranno fatti passare attraverso la regola #3, cioè l'ultima regola. Comunque questi pacchetti saranno taggati con INT_NET_HTTP se erano destinati alla porta TCP 80, altrimenti saranno taggati con INT_NET.

Oltre ad applicare tag con le regole di filtraggio, le regole di traslazione nat, rdr, e binat possono applicare tag a pacchetti usando la keyword tag.

Verifica dei tag applicati

Per verificare le precedenti tag applicate usare la keyword tagged:
pass out on $ext_if tagged INT_NET keep state

I pacchetti in uscita sulla $ext_if devono essere contrassegnati con la tag INT_NET per avere corrispondenza con la regola precedente. Corrispondenze inverse si possono anche avere usando l'operatore !:

pass out on $ext_if tagged ! WIFI_NET keep state

Le regole di traslazione (nat/rdr/binat) possono anche usare la keyword tagged per avere corrispondeza di pacchetti.

Policy di filtraggio

Le policy di filtraggio usano un approcio differente per scrivere le regole di configurazione. Una policy decide quali set di regole per le quali il traffico può passare o venire bloccato. I pacchetti sono quindi classificati in policy basate su criteri tradizionali di indirizzo IP sorgente/destinazione, porta, protocollo ecc. Per esempio esaminiamo la seguente policy del firewall:
  • Il traffico dalla LAN interna a Internet è permesso (LAN_INET) e deve essere traslato (LAN_INET_NAT)
  • Il traffico dalla LAN interna alla DMZ è permesso (LAN_DMZ)
  • Il traffico da Internet ai server nella DMZ è permesso (INET_DMZ)
  • Il traffico da Internet che è stato reindirizzato a spamd(8) è consentito (SPAMD)
  • Tutto il resto del traffico viene bloccato

Da notare che la policy copre tutto il traffico che sarà fatto passare attraverso il firewall. Il parametro tra parentesi indica il tag utilizzato per quella policy.

Regole di filtraggio e traslazione ora hanno bisogno di essere scritte per classificare i pacchetti nella policy.

rdr on $ext_if proto tcp from <spamd> to port smtp \
   tag SPAMD -> 127.0.0.1 port 8025
nat on $ext_if tagged LAN_INET tag LAN_INET_NAT -> ($ext_if)

block all
pass in on $int_if from $int_net tag LAN_INET keep state
pass in on $int_if from $int_net to $dmz_net tag LAN_DMZ keep state
pass in on $ext_if proto tcp to $www_server port 80 tag INET_DMZ keep state

Ora le regole che caratterizzano la policy sono definite.

pass in  quick on $ext_if tagged SPAMD keep state
pass out quick on $ext_if tagged LAN_INET_NAT keep state
pass out quick on $dmz_if tagged LAN_DMZ keep state
pass out quick on $dmz_if tagged INET_DMZ keep state

Ora che tutte le regole sono configurate, i cambi possono avvenire modificando le regole di classificazione. Per esempio, se un server POP3/SMTP viene aggiunto alla DMZ, sarà necessario aggiungere regole di classificazione per il traffico POP3 3 SMTP, come le seguenti:

mail_server = "192.168.0.10"
...
pass in on $ext_if proto tcp to $mail_server port { smtp, pop3 } \
   tag INET_DMZ keep state

Il traffico Email ora sarà fatto passare dalla policy INET_DMZ.

Il set di regole completo sarà: 
# macro
int_if  = "dc0"
dmz_if  = "dc1"
ext_if  = "ep0"
int_net = "10.0.0.0/24"
dmz_net = "192.168.0.0/24"
www_server = "192.168.0.5"
mail_server = "192.168.0.10"

table <spamd> persist file "/etc/spammers"

# classificazione -- classifica i pacchetti in base alla policy di 
# firewall definita.
rdr on $ext_if proto tcp from <spamd> to port smtp \
    tag SPAMD -> 127.0.0.1 port 8025
nat on $ext_if tagged LAN_INET tag LAN_INET_NAT -> ($ext_if)

block all
pass in on $int_if from $int_net tag LAN_INET keep state
pass in on $int_if from $int_net to $dmz_net tag LAN_DMZ keep state
pass in on $ext_if proto tcp to $www_server port 80 tag INET_DMZ keep state 
pass in on $ext_if proto tcp to $mail_server port { smtp, pop3 } \
    tag INET_DMZ keep state 

# rafforzamento della policy -- pass/block basata sulla policy di 
# firewall definita.
pass in  quick on $ext_if tagged SPAMD keep state
pass out quick on $ext_if tagged LAN_INET_NAT keep state
pass out quick on $dmz_if tagged LAN_DMZ keep state
pass out quick on $dmz_if tagged INET_DMZ keep state

Tag di frame Ethernet

Il tag può essere effettuato a livello Ethernet se la macchina che effettua il tag/filtraggio si comporta anche come bridge(4). Creando delle regole di filtraggio bridge(4) che usano la keyword tag keyword, PF può essere utilizzato per filtrare in base all'indirizzo MAC sorgente/destinazione. Le regole Bridge(4) sono create usando il comando brconfig(8). Esempio:
# brconfig bridge0 rule pass in on fxp0 src 0:de:ad:be:ef:0 \
   tag USER1

E quindi in pf.conf:

pass in on fxp0 tagged USER1
