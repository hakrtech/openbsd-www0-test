2.1 - Páginas de interés en Internet

El sitio oficial en Internet del Proyecto OpenBSD está ubicado en:
http://www.OpenBSD.org/es/

Aquí se puede encontrar mucha información útil relacionada con todos los aspectos del Proyecto OpenBSD.

También se puede encontrar información dirigida a los usuarios de portátiles en:
http://www.monkey.org/openbsd-mobile/

2.2 - Listas de correo

El Proyecto OpenBSD mantiene varias listas de correo a las que los usuarios deberían suscribirse. Para darse de alta en una lista de correo, se debe enviar un mensaje a majordomo@openbsd.org. Esta dirección es un servicio de suscripción automática. En el contenido del mensaje, en una sola línea, debe incluirse la orden subscribe seguida del nombre de la lista a la que se desee realizar la suscripción. Por ejemplo: 

    subscribe announce

El procesador de la lista enviará una contestación requiriendo la confirmación de la petición de suscripción a la lista. La confirmación que se envíe de vuelta al procesador de la lista se incluirá en la respuesta que éste envíe. El procesador de la lista enviará una opción para usar un enlace de hipertexto o para enviar una respuesta al mensaje para confirmar el intento. El mensaje de respuesta debe contener algo parecido a lo siguiente: 

    auth 90855167 subscribe announce you@example.com
Una vez que se haya enviado la confirmación de esta autorización de vuelta al procesador y que éste haya confirmado el alta, la dirección será añadida a la lista de inmediato, y el procesador enviará una notificación confirmando el alta en la lista.

Para darse de baja de una lista hay que enviar un mensaje a majordomo@openbsd.org con la orden unsubscribe seguida del nombre de la lista. Algo así: 

    unsubscribe announce

Si se encuentran dificultades con el sistema de suscripción a las listas de correo, se debe leer la información de las instrucciones. Para obtener las instrucciones hay que enviar un mensaje a majordomo@openbsd.org con la orden help en el contexto del mensaje. Algunas de las listas de distribución de OpenBSD más conocidas son:

  • announce - Anuncios importantes. Lista con poco tráfico.
  • security-announce - Anuncios sobre temas de seguridad.
  • misc - Preguntas y respuestas generales. Ésta es la lista más activa, y la mayoría de preguntas deberían ser enviadas a esta lista.
  • tech - Temas técnicos para los desarrolladores y usuarios avanzados de OpenBSD. Por favor, las dudas de «nuevo usuario» y las relacionadas con la instalación se deben enviar a misc, no a tech. No deben enviarse mensajes cruzados a misc y tech.
  • bugs - Errores recibidos por sendbug(1) y discusiones sobre estos errores.
  • source-changes - Lista de distribución de correo automatizada de los cambios en el árbol de fuentes del CVS.
  • ports - Discusión sobre el árbol de portes de OpenBSD.
  • advocacy - Difusión de OpenBSD.
  • ports-changes - Lista de distribución de correo automatizada de los cambios específicos a los portes en el árbol de fuentes del CVS.

Antes de enviar una pregunta a misc o a cualquier otra lista es conveniente buscar entre los archivos, ya que la mayoría de preguntas comunes habrán sido formuladas en repetidas ocasiones. A pesar de que sea la primera vez que un usuario se encuentre con el problema o la duda, es muy probable que otros suscriptores de las listas de correo lo hayan tratado ya varias veces en semanas anteriores, y pueden encontrar molesto volverlo a ver repetido. En la página web sobre listas de correo se puede encontrar enlaces a varios archivos y otras recomendaciones sobre el uso de las listas de correo: Se puede ver los archivos de los mensajes de correo de las listas de OpenBSD visitando la página de las listas de correo: http://www.openbsd.org/es/mail.html.

Otra lista de correo que puede ser de interés es openbsd-mobile@monkey.org. Esta lista trata sobre el uso de OpenBSD en portátiles.

Para suscribirse a esta lista:

    'echo subscribe | mail "openbsd-mobile-request@monkey.org"'

Los archivos de esta lista se pueden encontrar en: http://www.monkey.org/openbsd-mobile/archive/

2.3 - Páginas del manual

OpenBSD viene con una extensa documentación en forma de páginas del manual (man pages), así como de documentos más extensos sobre aplicaciones específicas. Para acceder a las páginas del manual y demás documentación, es necesario tener instaladas las distribuciones de man y misc.

He aquí una lista de las páginas del manual más útiles para los usuarios noveles:

  • help(1) - ayuda para usuarios y administradores nuevos.
  • afterboot(8) - primeras comprobaciones después de iniciar el sistema por primera vez.
  • boot(8) - procedimientos de arranque del sistema.
  • login.conf(5) - formato del fichero de configuración de la clase de ingreso.
  • adduser(8) - orden para añadir nuevos usuarios.
  • vipw(8) - edición del fichero de contraseñas.
  • man(1) - uso de las páginas del manual.
  • sendbug(1) - enviar un informe sobre un problema (PR) en OpenBSD al sitio central de asistencia.
  • disklabel(8) - leer y escribir las clasificaciones del disco.
  • ifconfig(8) - configurar los parámetros de la interfaz de red.
  • route(8) - manipulación manual de las tablas de enrutamiento.
  • netstat(1) - estado de la red.
  • reboot, halt(8) - parar y reiniciar el sistema.
  • shutdown(8) - cerrar el sistema en un tiempo especificado.
  • boot_config(8) - cambiar la configuración del núcleo del sistema ("kernel") durante el arranque.

Si no se ha instalado el paquete man31.tgz durante la instalación del sistema, se pueden ver todas las páginas del manual de OpenBSD en formato de hipertexto en http://www.openbsd.org/cgi-bin/man.cgi.

Por regla general, si se conoce el nombre de una orden o de una página del manual, se puede leer ejecutando man <orden> (en donde <orden> es el nombre de la orden o función sobre la que desea obtener información). Por ejemplo, man vi mostraría la página del manual del editor vi. Si no se sabe el nombre de la orden o función, o si man <orden> no encuentra la página del manual que esté buscando, es posible realizar una búsqueda entre la base de datos de las páginas del manual mediante la orden apropos <clave> o man -k <clave> (en donde <clave> es una palabra que puede aparecer en el título de la página del manual que se desee consultar). Por ejemplo: 

    bsd# apropos "time zone"
tzfile (5) - time zone information
zdump (8) - time zone dumper
zic (8) - time zone compiler

Los números entre paréntesis indican la sección del manual en la que se encuentra esa página. En algunos casos puede ocurrir que para una misma orden o función exista más de una página del manual con el mismo nombre y en secciones separadas del manual. Pongamos como ejemplo que necesitáramos conocer el formato de los ficheros de configuración para el dæmon cron. Una vez que supiéramos la sección del manual para la página que deseamos consultar, ejecutaríamos man <n> <orden> (en donde <n> sería el número de la sección del manual). 

    bsd# man -k cron
cron (8) - daemon to execute scheduled commands (Vixie Cron)
crontab (1) - maintain crontab files for individual users (V3)
crontab (5) - tables for driving cron
bsd# man 5 crontab

Además de las páginas del manual de UNIX, existe una documentación a la que se puede dar formato para la impresión y que está incluida en la distribución misc. Esta documentación se puede encontrar en el directorio /usr/share/doc. Si se ha instalado la distribución text, se puede dar formato a cada grupo de documentos con la orden make desde dentro del subdirectorio apropiado. El subdirectorio psd es el de la distribución Programmer's Supplementary Documents (Documentos Suplementarios para el Programador); el subdirectorio smm es el del System Manager's Manual (Manual de Gestión del Sistema); y el subdirectorio usd es el del UNIX User's Supplementary Documents (Documentos Suplementarios del Usuario de UNIX). Es posible ejecutar 'make' sobre las tres distribuciones, o puede hacerse sobre cada una de ellas por separado desde el correspondiente subdirectorio. Algunos de los subdirectorios están vacíos. El formato del documento predeterminado será Postscript, un formato ideal para la impresión de los documentos. El resultado de la salida en Postscript suele ser bastante voluminoso; el incremento en volumen será de entre 250% y 300%. Si no se dispone de acceso a una impresora con capacidad para Postscript o a un programa de visualización para este formato, también se puede dar formato a los documentos en texto simple. Para ello hay que añadir el indicador -Tascii en cada fichero Makefile por cada una de las órdenes de groff(1) (o ejecutarlo a mano). Algunos documentos usan macros de formato ms, y otros macros me. El fichero Makefile que se encuentra en cada uno de los subdirectorios de los documentos (v.g. /usr/share/doc/usd/04.csh/Makefile) indica cuál se puede utilizar. Por ejemplo: 

    bsd# cd /usr/share/doc/usd/04.csh
bsd# groff -Tascii -ms tabs csh.1 csh.2 csh.3 csh.4 csh.a csh.g > csh.txt 
bsd# more csh.txt

Las páginas del manual de UNIX están, por lo general, más actualizadas y son por tanto más fiables que los documentos formateables. Sin embargo, los documentos formateables suelen ofrecer aclaraciones sobre las aplicaciones complicadas con más detalle que las páginas del manual.

Para muchos usuarios puede ser de gran utilidad el tener una copia impresa de las páginas del manual. A continuación se explica cómo obtener un copia imprimible de una página del manual.

¿Cómo se puede ver el fichero fuente de una página del manual? (o sea, aquél cuyo nombre acabe en número, como tcpdump.8).

Esto es lo que ocurre en el árbol de fuentes. Las páginas del manual se encuentran dentro del árbol en formato fuente, y muchas veces se actualizan mediante el uso de CVS. Para poder ver esas páginas: 

    
# nroff -mdoc <fichero> | more

¿Cómo se puede obtener una página del manual sin caracteres de control o de formato?

Esto es útil para la obtención de una salida directa, sin caracteres no imprimibles: 

    
# man <orden> | col -b

¿Cómo se puede obtener una copia en Postscript de una página del manual lista para la impresión?

Nótese que [man_src_file] debe ser el fichero fuente de la página del manual (probablemente un fichero que termine con un número, como tcpdump.8). Las versiones en formato Postscript de las páginas del manual son muy vistosas. Se pueden imprimir o se pueden visualizar en pantalla con un programa como gv (GhostView). Se puede obtener GhostView desde nuestro árbol de portes. Use las siguientes opciones para la orden groff(1) a fin de obtener una versión en formato PostScript de un página del manual del sistema OpenBSD: 

    
# groff -mdoc -Tps [man_src_file] > outfile.ps

La línea de órdenes anterior sólo funcionará con páginas del manual cuyo formato sea el del paquete de macros mdoc(7), utilizado por las páginas del manual de BSD. Para obtener una versión en formato PostScript de la página del manual de un programa de un tercero (ya sea compilado por uno mismo o instalado desde los portes [ports(7)] o paquetes [packages(7)]), hay que hacer lo siguiente: 

    
# groff -Tps -mandoc [man_src_file] > outfile.ps

2.4 - Cómo se envía un informe sobre un error

Antes de enviar un informe sobre cualquier error es necesario leer http://www.openbsd.org/es/report.html

Un informe sobre un error bien escrito es una de las responsabilidades más importantes de los usuarios finales. Para poder diagnosticar los errores más serios se requiere una información muy detallada. Los desarrolladores suelen recibir informes por correo electrónico como el siguiente: 

    From: joeuser@example.com
To: bugs@openbsd.org
Subject: HELP!!!

¡¡¡Tengo un PC que no arranca!!!
¡¡¡¡Es un 486!!!!

Espero que comprensible el porqué este tipo de «informes» son eliminados sin mediar palabra. Todos los informes sobre errores deben contener información detallada. Si el usuario del mensaje anterior hubiera querido realmente que el error sobre el que informaba hubiera sido investigado, habría dado más información; algo así:

(Nota: véase report.html para más información sobre cómo crear y entregar informes sobre errores. Básicamente, la información detallada sobre el hardware utilizado es necesaria si se sospecha que el error puede estar relacionado de alguna manera con el hardware o con la configuración del hardware. Por lo general, la salida de la orden dmesg(8) es suficiente para esto. Además, es necesaria una descripción detallada del problema)

Si se dispone de un sistema OpenBSD desde el que se pueda enviar el informe sobre el error, es conveniente usar la utilidad sendbug(1) para enviar el informe a GNATS, el sistema de seguimiento de errores. Lógicamente, no se puede usar sendbug(1) si el sistema no arranca, pero en caso contrario hay que usarlo siempre que sea posible. Se debe incluir información detallada sobre el problema que se ha tenido, la configuración exacta del sistema, y cómo podemos reproducir el problema. Para el uso de la orden sendbug(1) se requiere que el sistema pueda enviar correo electrónico por Internet.

Para comprobar el estado de investigación en el que se encuentra el informe que se haya enviado, las mejores formas son:

  • Visitar el sistema de seguimiento de GNATS: http://cvs.openbsd.org/cgi-bin/wwwgnats.pl.
  • Mirar en la lista: bugs@openbsd.org (archivos de las listas: http://www.openbsd.org/es/mail.html).
