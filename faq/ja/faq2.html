2.2 - メーリングリスト

OpenBSD プロジェクトではいくつかのメーリングリストを用意しており、 ユーザに参加を推奨しています。メーリングリストに参加するには、 majordomo@openbsd.org 宛に電子メールをお送りください。このアドレスは参加を自動処理する プログラムのためのものです。メール本文には、参加したいメーリング リスト用の参加コマンドを一行で記述します。たとえば、次のような 感じです。 

subscribe announce

そうすると自動参加処理プログラムから、そのメーリングリストに参加する意志を 確認するメールが返送されてきます。これは、他人があなたを勝手に参加させて、 不要なメールで溢れさせたりすることがないようにするためなのです。 このメッセージには、確認メッセージに反応するか、majordomo@openbsd.org に応答を返すための、メーリングリストサーバの web ページへのリンクを含む、参加の意思確認のためのいくつかの異なる方法について 操作手順が書かれていますので、お好きな方法を選択してください。 これらの全部で 3 つの確認方法は、A56D-70D4-52C3 のような、 ユニークで時間限定の識別番号を必要とすることに注意してください。 これは、あなたが本当にメーリングリストへの参加を希望した本人かどうかを 確認するためなのです (これが本当の "選択" なのです)。

参加の意志確認が終わるとすぐにメーリングリストへの 参加処理が行われ、自動参加処理プログラムから正常に参加処理が 行われた旨がメールで通知されます。

メーリングリストから退会する場合にも、 majordomo@openbsd.org へメールを送ります。 その場合、本文は次のようになります。 

unsubscribe announce

メーリングリストシステムに関して何か困ったことがありましたら、まずは 本文に "help" と書いたメールを majordomo@openbsd.org に送ると送られてくるヘルプファイルに目をとおしてください。

あなたの OpenBSD のメーリングリストへの参加状況は、 http://lists.openbsd.org の web インターフェイスでメンテナンスすることもできます。

次にあげるのは、一般向けの OpenBSD メーリングリストの いくつかです。

  • announce - 重要なアナウンス。 配送頻度は低めです。
  • security-announce - セキュリティ関連の アナウンス。配送頻度は低めです。
  • misc - ユーザの一般的な質問とその答え。 最も活発なメーリングリストであり、 大部分の質問は普通ここで行われます。
  • bugs - sendbug(1) 経由で送られるバグ情報と それに関する議論。
  • source-changes - CVS ソースツリーの変更点が 自動的に送られるメーリングリスト。開発者がソースツリーの変更点を commit するごとに、CVS は (通常は短かい) commit 時のメッセージの写しを このメーリングリストに送信します。
  • ports - OpenBSD Ports ツリーに関する議論。
  • ports-changes - CVS ソースツリーの Ports 関連の変更点が自動的に送られるメーリングリスト。
  • advocacy - OpenBSD の普及活動に関する議論、 および misc からずれ過ぎてしまった話題。

多くの質問は何度も繰り返し行われていますので、 misc やその他のメーリングリストに質問を送る前に、 メールアーカイブを調べましょう。 その質問や問題があなたにとって初めてのものであったとしても、 メーリングリストに参加している他の人たちは先週も同じ質問を何度も受けていて、 もう見たくないと思っているかも知れません。

メーリングリストのアーカイブやメーリングリストに関するガイドライン、さらに詳しい情報は、 メーリングリストのページに書かれています。

もうひとつ、面白いメーリングリストに openbsd-mobile@monkey.org があります。 ここでは OpenBSD を移動端末で利用することに関する議論が行われます。

このメーリングリストに参加するには、次のようにします。

'echo subscribe | mail "openbsd-mobile-request@monkey.org"'

メーリングリストのアーカイブは次の場所にあります。 http://www.monkey.org/openbsd-mobile/archive/

2.3 - マニュアルページ

OpenBSD は特定のアプリケーションに関する長い文書の他、さまざまな文書を マニュアルページという形で提供しています。マニュアルページや 他の文書を読むには、man と misc の配布セットをインストール していなければなりませんのでご注意ください。

次のリストは、初心者向けに有用なマニュアルページをまとめたものです。

  • help(1) - 初心者ユーザ、管理者向けのヘルプ
  • afterboot(8) - 最初に起動させた後に チェックしなければならないこと
  • boot(8) - システムの起動過程に関する解説
  • login.conf(5) - ログインクラス設定ファイルの書式
  • adduser(8) - 新しいユーザを追加するコマンド
  • vipw(8) - マスターパスワードファイルの編集
  • man(1) - オンラインマニュアルページの表示
  • sendbug(1) - OpenBSD の障害報告 (PR; problem report) を担当サイトに送るコマンド
  • disklabel(8) - ディスクパックラベルの読み書き
  • ifconfig(8) - ネットワークインターフェイスのパラメータ設定
  • route(8) - 経路テーブルの手動操作
  • netstat(1) - ネットワーク利用状況の表示
  • reboot, halt(8) - システムの停止と再起動
  • shutdown(8) - 指定時間にシステムを停止させる
  • boot_config(8) - カーネルの設定を起動時に変更する方法

man33.tgz のファイル一式をインストールしていない場合でも、 http://www.openbsd.org/cgi-bin/man.cgi を使用して、web 上ですべての OpenBSD のマニュアルページを読むことができます。

一般的に、コマンドやマニュアルページの名前が分かっていれば "man コマンド名" とすることでマニュアルページを読むことができます。たとえば "man vi" とすれば、vi エディタのマニュアルページが読めます。コマンド名がわからなかったり、 "man コマンド名" としたのにマニュアルページが見つからないという場合は、 "apropos なんとか" もしくは "man -k なんとか" とすることで、 マニュアルページデータベースから検索を行うことができます。 「なんとか」の部分には、あなたが探しているマニュアルページの タイトルに出てきそうな言葉を入れてください。たとえば、次のような感じです。 

# apropos "time zone"
tzfile (5) - time zone information
zdump (8) - time zone dumper
zic (8) - time zone compiler

カッコ内の数字は、マニュアルページが置かれているセクションを 示しています。お気づきかと思いますが、いくつかのマニュアルページは 同じ名前で複数のセクションに置かれています。たとえば cron デーモンの設定ファイルの書式を知りたいとしましょう。 求めるマニュアルページのセクション番号が分かったら、 "man n command" (n はセクション番号) とします。 

# man -k cron
cron (8) - clock daemon
crontab (1) - maintain crontab files for individual users
crontab (5) - tables for driving cron
# man 5 crontab

UNIX マニュアルページの他に、(misc に含まれる) 組版可能な文書群があります。 それらは /usr/share/doc ディレクトリに置かれており、このテキストを インストールしていればそれぞれのサブディレクトリで "make" を実行することで組版することが可能です。 psd サブディレクトリは Programmer's Supplementary Documents, smm サブディレクトリは System Manager's Manual, usd サブディレクトリは UNIX User's Supplementary Documents です。 この 3 つのサブディレクトリで "make" を実行するか、 あるいはそのディレクトリの下にあるセクションをひとつ選んで `make' を実行することができますが、サブディレクトリには、空のものもあります。 文書は標準で、印刷に適した PostScript 形式に組版されます。 この PostScript 形式は非常に大きなもの -- 250-300% のサイズ -- になります。 PostScript プリンタや PostScript ディスプレイが使用できない環境にいる場合は、 端末のディスプレイで表示できる形式に組版することも可能です。 そのためには、各 Makefile にある groff(1) の部分に "-Tascii" オプションフラグを追加 (もしくはそのコマンドを手動で実行) します。 文書には、ms マクロを使用しているものと me マクロを使用しているものがあります。 各文書のディレクトリにある Makefile (/usr/share/doc/usd/04.csh/Makefile など) には、どれを使用すれば良いのかが書かれています。 たとえば、次のような感じです。 

# cd /usr/share/doc/usd/04.csh
# groff -Tascii -ms tabs csh.1 csh.2 csh.3 csh.4 csh.a csh.g > csh.txt 
# more csh.txt

通常、これらの組版可能な文書よりも UNIX マニュアルページの方が新しく、 信頼できるものになっています。一方で組版可能な文書は、マニュアルページよりも 複雑な応用例について解説してあることがあります。

ほとんどの場合、マニュアルページを印刷すると便利でしょう。 マニュアルページの印刷手順を以下に説明します。

マニュアルページのソースファイル (tcpdump.8 のように数字で終わっているファイル) を表示する方法は?

このファイルはソースツリーにあります。 ソースツリーにあるマニュアルページは組版されておらず、 CVS を使用して何度も更新されています。 これらのマニュアルページを表示するには、次のようにします。 

# nroff -mandoc <ファイル名> | more

組版や制御文字を含まない、 素のままのマニュアルページを作成するには ?

制御文字を含まないマニュアルページを 作成するには、たとえば次のようにすると 良いでしょう。 

# man <コマンド名> | col -b

印刷用に、マニュアルページの PostScript 版を作成するには?

[man_src_file] の部分には、マニュアルページのソースファイル名 (おそらく tcpdump.8 のように数字で終わっています) を入れてください。PortScript 版のマニュアルページは非常にきれいな仕上がりになります。これは gv (GhostView) のような プログラムで印刷やディスプレイ上での表示を行うことが可能です。 GhostView は Ports ツリーにあります。 OpenBSD システムのマニュアルページの PostScript 版を作成するには、次のような groff(1) コマンドオプションを使用してください。 

# groff -mandoc -Tps [マニュアルページのソースファイル] > outfile.ps

2.4 - バグ報告

バグ報告を提出する前に、 http://www.openbsd.org/report.html に目をとおしてください。

バグ報告を適切に行うことは、エンドユーザの最も重要な 役割のひとつです。深刻なバグの診断には、非常に詳細な 情報が必要になります。開発者の元には、よく次のような バグ報告の電子メールが届きます。 

From: joeuser@example.com
To: bugs@openbsd.org
Subject: 助けて!!!

私の PC が起動しないのです !!!!!  486 マシンなのですが !!!!!

大部分のみなさんには、このような報告がすぐに削除される理由を 理解していただけると思います。バグ報告には、詳細な情報が含まれていなければなりません。 もし本当にこの人が誰かにこのバグを見つけて欲しいと思っていたなら、きっと次のように、 より多くの情報を提供していたはずです (訳注: 実際のメールは英語で提出してください)。 

From: smartuser@example.com
To: bugs@openbsd.org

バグ報告の作成と提出についてのより詳しいことは、 report.html を参照してください。もし、 バグがハードウェアかハードウェアの構成と何らかの関係があると考えられる場合は、 ハードウェアについての詳細な情報が必要です。通常、ここで期待されるているものは dmesg(8) の出力で十分でしょう。もちろん、あなたが遭遇した問題についての 詳細な記述は必須です。あなたは、ハードウェアを記述した dmesg や、賢明なユーザが どうしてシステムが壊れているわけではない (3.2 が正しく動いていた) と考えるのか、 ということについて記述された文章、(X の起動時に) どうしてこのようなクラッシュが 発生してしまうのかということ、そして、デバッガの "ps" および "trace" コマンドの出力に注意を払うことでしょう。このような場合、賢明なユーザは シリアルコンソールで記録された出力を用意しますが、もし、 あなたにはこれができない場合、クラッシュの記録を取るために、紙と鉛筆を使用することも できるのです (実は、これが本当の問題であり、このような報告に記載された情報が、Sun4c システムに影響を与えているこの問題の解決につながるのです)。

もし、バグ報告を動作中の OpenBSD システムから行いたければ、 sendbug(1) ユーティリティを使用して GNATS 障害報告追跡システムに バグ報告を送ることができます。 もちろん、システムが起動していなければ sendbug(1) を使用することができませんが、使用できる場合には、できるだけこれを使用すべきです。 その際にも、問題の詳細、システムの設定、 再現方法に関する情報を含めることを忘れないでください。 sendbug(1) コマンドを動作させるには、 インターネットに向けて電子メールを送信できるように システムが設定されている必要があります。

sendbug(1) を使用してバグの報告を提出すると、報告の処理状況は電子メールで通知を 受けることになるでしょう。 付加情報やテストの必要なパッチを開発者に送付することもできます。また、 メーリングリストのページにその詳細が 記述されていますが、bugs@openbsd.org メーリングリストのアーカイブを 監視することもできます。あるいは、オンラインで バグ追跡システム のバグ報告データベースのステータスの問い合わせを行うことも可能です。
