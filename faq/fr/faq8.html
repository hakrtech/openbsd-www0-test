8.1 - Que signifient ces erreurs Kerberos lorsque je me connecte la première fois ?

Remarque: Cette section est obsolète si vous avez mis à jour votre système en OpenBSD 2.8-stable ou supérieur. (le correctif se trouve à l'adresse : ftp://ftp.openbsd.org/pub/OpenBSD/patches/2.8/common/008_kerberos.patch).

Lorsque vous avez installé votre système pour la première fois, vous avez très probablement observé un message d'erreur qui ressemble au message suivant :

    Warning: no Kerberos tickets issued.

CE MESSAGE D'ERREUR N'EST PAS DU TOUT PERTINENT ET DEVRAIT ETRE SUPPRIME SI CELA EST VRAIMENT NECESSAIRE

Vous n'avez probablement pas configuré Kerberos sur votre système. Il est donc normal que vous n'obteniez pas de ticket Kerberos. Mais si Kerberos est configuré, vous devriez vous pencher sur le problème. Consultez la FAQ 10, Mise en place de Kerberos. Si vous ne pouvez plus supporter de voir ce message d'erreur et si vous ne comptez pas mettre en oeuvre Kerberos, suivez la procédure suivante pour vous en débarrasser une bonne fois pour toutes :

  • Assurez-vous d'avoir le code source.
  • Editez le fichier /usr/share/mk/bsd.own.mk et fixez la valeur de KERBEROS à 'no', ou créez un fichier /etc/mk.conf pour prendre en compte cette modification.
  • cd /usr/src/usr.bin/login; make clean; make; make install

8.2 - Comment puis-je changer de terminal virtuel ?

Appuyez simplement sur les touches [ctrl] - [alt] - [une des touches de fonction] (i386)

i386 est la seule architecture incluant le support des terminaux virtuels. Vous pouvez également accéder à des terminaux virtuels lorsque vous utilisez X. Par exemple, si vous démarrez X dans le terminal 1, puis vous appuyez sur [ctrl]-[alt]-[F2] pour accéder au terminal 2, X ne réapparaîtra pas si vous changez à nouveau de terminal pour revenir vers le terminal 1. Pour retrouver votre affichage X, vous devez utilisez la combinaison [ctrl]-[alt]-[F5].

8.3 - J'ai oublié mon mot de passe root... Que faire ?

Respectez les étapes suivantes:

  1. Démarrez en mode "single user". Pour les architectures i386, saisissez "boot -s" au prompt de démarrage
  2. Montez la partition racine :
      bsd# fsck -p / && mount -u /
  3. Si /usr n'est pas sur la même partition que / (recommandé !), alors vous aurez également besoin de la monter :
      bsd# fsck -p /usr && mount /usr
  4. Exécutez passwd(1)
  5. Démarrez en mode multi-utilisateur...et rappelez-vous de votre mot de passe !

8.4 - X ne veut pas démarrer, et j'ai de nombreux messages d'erreur

Si vous êtes sûr que X est proprement paramétré et que votre fichier XF86Config est fonctionnel, alors le problème vient très probablement de machdep.allowaperture. Vous devez aussi vous assurez que votre fichier de configuration du noyau contient les deux options suivantes:

    option XSERVER
option APERTURE

[Ces deux options sont incluses dans le noyau GENERIC]

Editez le fichier /etc/sysctl.conf et positionnez machdep.allowaperture à 1. Ceci permettra à X d'accéder au pilote dit d'ouverture (aperture). Ce paramétre est positionné à la bonne valeur si vous avez correctement répondu à la question sur l'utilisation de X durant la phase d'installation du système. L'accès à ce pilote est nécessaire car il contrôle les entrées/sorties sur les cartes graphiques.

Pour les autres problèmes relatifs à X sur plateforme i386, consultez la documentation en ligne de XFree86 disponible à l'adresse http://www.xfree86.org/support.html.

8.5 - Qu'est-ce que CVS, et comment puis-je l'utiliser ?

CVS est le système utilisé par le projet OpenBSD pour contrôler les modifications apportées au code source. CVS veut dire Concurrent Versions System. Vous pouvez en savoir plus sur CVS à l'adresse http://www.cvshome.org/. CVS peut être employé par l'utilisateur final pour maintenir à jour le code source et l'arborescence des ports. CVS rend le téléchargement du code source via un des miroirs CVS du projet extrêmement simple.

Mise en place initiale de votre environnement CVS

Il y a différentes méthodes pour configurer votre environnement CVS. Pour commencer, vous aurez besoin d'une copie CVS initiale des sources. Si vous avez acheté le CD, alors vous êtes chanceux car il contient une copie CVS pour la version concernée. Vous pouvez extraire cette copie à partir du CD en :

1) Copiant l'arborescence (nous supposerons que le CD est monté sur /mnt) : 

    # mkdir /usr/src
# cd /mnt; cp -Rp CVS Makefile bin distrib etc games gnu \
include kerberosIV lib libexec lkm regress sbin share \
sys usr.bin usr.sbin /usr/src

2) En utilisant le CD en montage de type union sous un répertoire accessible en écriture. Cependant, il faut savoir que le code du système de fichiers union n'est pas infaillible : 

    # mkdir /usr/src
# mount -t union -o -b /mnt /usr/src

/usr/src sera dès lors une zone de travail où toutes les commandes cvs(1) fonctionneront correctement.

Si vous n'avez pas de CD OpenBSD, vous allez devoir récupérer les sources à partir d'un des serveurs CVS anonyme OpenBSD. La liste de ces serveurs est disponible sur http://www.openbsd.org/fr/anoncvs.html. Une fois un serveur sélectionné, vous devez choisir le module à récupérer. Il y a trois modules principaux disponibles via CVS :

  • src - Le module src constitue la totalité du code source de OpenBSD. Ceci inclut aussi bien les sources du noyau que les sources des programmes de l'espace utilisateur.
  • ports - Le module ports contient tout ce dont vous avez besoin pour avoir l'arborescence des ports de OpenBSD. Pour en savoir plus sur cette arborescence, consultez FAQ 8, Qu'est-ce que l'arborescence des ports ?
  • www - Le module www contient toutes les pages web de OpenBSD, y compris la présente FAQ.

Maintenant que vous avez choisi le module à récupérer, il vous reste à effectuer une étape supplémentaire avant de pouvoir effectivement récupérer ce module. Vous devez choisir la méthode à utiliser. Par défaut, CVS utilise rsh(1) mais la plupart des serveurs CVS anonyme n'autorisent pas ce protocole. Il est donc plus judicieux d'utiliser ssh(1). Pour ceux qui sont derrière un pare-feu, il est possible d'utiliser pserver ou ssh sur le port 2022 sur certains serveurs CVS anonyme. Lisez http://www.openbsd.org/fr/anoncvs.html pour savoir quels serveurs acceptent quels protocoles. Une fois cette étape terminée, voici un exemple de récupération du code source. Dans cet exemple, le serveur CVS anonyme utilisé est situé en Belgique. Il est important de noter que si vous êtes en dehors de la Belgique, vous devriez utiliser un serveur plus proche de vous. Les serveurs CVS anonyme sont situés un peu partout dans le monde. Cet exemple utilise ssh comme protocole de récupération. 

    ericj@oshibana:~> export CVS_RSH=/usr/bin/ssh
ericj@oshibana:~> echo $CVS_RSH
/usr/bin/ssh
ericj@oshibana:~> export CVSROOT=CVS anonyme@CVS anonyme.be.openbsd.org:/cvs
ericj@oshibana:~> cvs get src
Warning: Remote host denied X11 forwarding, perhaps xauth program could not be run on the server side.
cvs checkout: in directory src:
cvs checkout: cannot open CVS/Entries for reading: No such file or directory
cvs server: Updating src
U src/Makefile
[snip]

Vous remarquerez que la variable d'environnement CVSROOT est positionnée. Cette variable indique à cvs(1) le serveur CVS anonyme à utiliser. Nous pourrions aussi spécifier ce serveur en utilisant l'option -d. Par exemple :

    ericj@oshibana:~>cvs -d CVS anonyme@CVS anonyme.be.openbsd.org:/cvs get src

Ces commandes doivent être exécutées sous /usr. Les répertoires /usr/src, /usr/ports, et /usr/www seront créés en conséquence si vous récupérez les modules correspondants. Vous pouvez bien sûr mettre ces modules n'importe où dans votre arborescence, mais si vous souhaitez travailler avec (pour faire un make build par exemple), ils doivent être sous /usr.

Mise à jour de votre arborescence CVS

Une fois la mise en place initiale de votre arborescence CVS terminée, la mise à jour est facile. Vous pouvez mettre à jour votre arborescence n'importe quand. Certains serveurs CVS anonyme disposent des mises à jour plus rapidement que d'autres. Ceci est indiqué dans http://www.openbsd.org/fr/anoncvs.html. Dans l'exemple suivant, une mise à jour du module www sera effectuée à partir du serveur CVS anoncvs.be.openbsd.org. Notez l'utilisation de l'option -q qui indique à cvs d'être moins verbeux. 

    ericj@oshibana:~> echo $CVSROOT                                 
CVS anoncvs@CVS anoncvs.be.openbsd.org:/cvs
ericj@oshibana:~> cvs -q up -Pd www
Warning: Remote host denied X11 forwarding, perhaps xauth program could not be run on the server side.
U www/want.html
M www/faq/faq8.html
ericj@oshibana:~>

Autres options cvs

Pour quelques personnes, la bande passante et le temps sont des facteurs importants lors de la mise à jour des sources. CVS dispose d'une option -z[1-9] qui utilise gzip pour compresser les données. Pour l'utiliser, faites -z[niveau-de-compression]. Par exemple, -z3 pour un niveau de compression de 3.

8.6 - Qu'est-ce que l'arborescence des ports ?

L'arborescence des ports est un ensemble de fichiers Makefile qui téléchargent, appliquent des correctifs, configurent et installent des programmes pour que vous puissiez les exécuter dans votre environnement OpenBSD sans avoir à effectuer toutes les opérations précitées manuellement. Vous pouvez récupérer l'arborescence des ports à partir de /pub/OpenBSD/3.0/ports.tar.gz sur n'importe quel serveur ftp OpenBSD. L'arborescence des ports la plus récente est disponible via le module cvs ports ou à partir de /pub/OpenBSD/snapshots/ports.tar.gz. Cependant, la plupart d'entre vous préféreront utiliser les packages. Ces derniers sont crées à partir de l'arborescence des ports et sont déjà compilés et prêts à être utilisés. Pour en savoir plus sur les packages, consultez FAQ 8, Qu'est-ce que les packages?

Obtenir la dernière version de l'arborescence des ports

Si vous comptez utiliser la dernière version de l'arborescence des ports, vous devriez avoir la dernière version de OpenBSD. En effet, cela est nécessaire vu que l'arborescence des ports est constamment corrigée et modifiée pour faciliter l'interaction avec le reste de votre système OpenBSD. Si vous ne tenez pas compte de ce conseil, attendez vous à quelques problèmes mineurs lors des mises à jour.

La meilleure façon de maintenir l'arborescence des ports de OpenBSD à jour est d'utiliser cvs(1). OpenBSD possède un groupe entier de serveurs CVS anonyme publiquement accessibles. Pour en savoir plus sur cvs, consultez www.openbsd.org/anoncvs.html et Qu'est-ce que CVS, et comment puis-je l'utiliser ?

Si vous n'avez pas du tout de copie de l'arborescence des ports en local, vous pouvez la télécharger à partir de n'importe quel serveur ftp OpenBSD. www.openbsd.org/fr/ftp.html contient la liste des serveurs. Une fois connecté à un serveur, téléchargez /pub/OpenBSD/3.0/ports.tar.gz et ouvrez l'archive dans /usr/ports. Par exemple : 

    ericj@oshibana> ftp ftp://ftp.fr.openbsd.org/pub/OpenBSD/3.0/ports.tar.gz
ericj@oshibana> sudo cp ports.tar.gz /usr
ericj@oshibana> cd /usr; sudo tar xfz ports.tar.gz

Une fois ces opérations achevées, vous pouvez utiliser cvs(1) pour mettre à jour votre arborescence des ports. Si vous avez commencé par l'arborescence des ports de OpenBSD 3.0, vous devez effectuer quelques changements. /usr/share/mk, qui contient des fichiers utilisés par make(1) doit être mis à jour. Si vous avez le module src, vous pouvez le mettre à jour simplement de la manière suivante : 

    ericj@oshibana> cd /usr/src/share/mk; sudo make install

Après avoir fait cela, vous aurez probablement besoin de la dernière version du binaire make(1). Si vous n'avez pas le module src, vous pouvez récupérer les sources de make(1) par cvs(1) et les compiler. Par exemple: 

    ericj@oshibana> export CVS_RSH=/usr/bin/ssh
ericj@oshibana> cvs -dCVS anonyme@any.CVS anonyme.server:/cvs get src/usr.bin/make
cvs server: Updating src/usr.bin/make
U src/usr.bin/make/Makefile
U src/usr.bin/make/Makefile.boot
U src/usr.bin/make/arch.c
U src/usr.bin/make/bit.h
U src/usr.bin/make/buf.c
U src/usr.bin/make/buf.h
U src/usr.bin/make/compat.c
[SNIP] 
ericj@oshibana> cd src/usr.bin/make; sudo make install

Si vous avez déjà le module src sur votre machine, mettez à jour make et installez-le. C'est pratiquement tout ce dont vous aurez besoin pour avoir une arborescence des ports fonctionnelle.

Une snapshot de l'arborescence des ports est créée quotidiennement et peut être téléchargé à partir de /pub/OpenBSD/snapshots/ports.tar.gz d'un des serveurs ftp OpenBSD.

8.7 - Qu'est-ce que les packages?

Les packages sont les binaires précompilés de quelques programmes très utilisés. Ils sont prêts à l'utilisation sur un système OpenBSD. Comme les ports, les packages sont très faciles à maintenir et à mettre à jour. Des packages sont ajoutés de manière constante alors regardez pour chaque version les packages additionnels.

Voici une liste d'outils utilisés pour gérer les packages :

  • pkg_add(1) - un utilitaire pour installer les packages
  • pkg_create(1) - un utilitaire pour créer les packages
  • pkg_delete(1) - un utilitaire pour supprimer des packages précédemment installés
  • pkg_info(1) - un utilitaire pour afficher des informations sur les packages

Où trouver les packages

Si vous êtes un utilisateur intelligent et que vous avez acheté un CD OpenBSD, alors les packages se trouvent sur un des CDs selon votre architecture. Si vous ne possédez pas un CD OpenBSD, vous pouvez télécharger à partir de n'importe quel miroir ftp. La page http://www.openbsd.org/fr/ftp.html contient la liste des miroirs. Les packages se trouvent sous /pub/OpenBSD/3.0/packages. Les sous-répertoires sont nommés d'après les différentes architectures supportées par OpenBSD.

8.8 - Y a t'il un moyen d'utiliser mon lecteur de disquettes alors qu'il n'était pas connecté durant la phase de démarrage ?

Bien sûr. Vous devez rajoutez "flags 0x20" à la fin de la ligne fd* et recompiler votre noyau. En définitif, voici à quoi ressemble cette ligne : 

      fd*     at fdc? drive ? flags 0x20

Vous devriez être capable maintenant d'utiliser votre lecteur de disquettes quand vous le souhaitez. Cela n'aura plus d'importance si vous l'avez connecté après le démarrage.

8.9 - Les options de démarrage - Utilisation du chargeur de démarrage OpenBSD

Au démarrage de votre système OpenBSD, vous avez probablement remarqué le prompt de boot :

    boot>

La plupart des gens n'ont rien besoin de faire à ce niveau. Le système démarrera automatiquement si aucune commande n'est donnée. Cependant, des problèmes surviennent quelques fois lorsque des fonctions spéciales sont nécessaires. C'est à cet endroit que les options de démarrage interviennent. Pour commencer, vous devriez lire la page du manuel boot(8). Nous allons couvrir ici les commandes les plus fréquemment utilisées avec le chargeur de démarrage.

Si aucune commande n'est saisie, le chargeur de démarrage essaiera automatiquement de démarrer /bsd. Si cette tentative échoue, il essaiera de charger /obsd et ainsi de suite jusqu'à ce qu'il trouve un noyau adéquat. Vous pouvez spécifier le noyau à charger à l'aide de la commande suivante :

    boot> boot wd0a:/bsd

ou
    boot> b /bsd

Cette commande fonctionnera si wd0a est votre partition racine.

Voici une liste brève des options que vous pouvez utiliser avec le noyau OpenBSD :

  • -a : Cette option vous permettra de spécifier une partition racine alternative après chargement du noyau.
  • -c : Cette option vous permettra d'entrer en mode configuration. Consultez la section Configuration au démarrage de la FAQ.
  • -s : Cette option vous permet de démarrer en mode "single user".
  • -d : Cette option est employée pour basculer le noyau dans le débogueur ddb. Il faut que DDB soit inclus dans votre noyau.

Ces options sont saisies dans le format boot [ image [-acds]]

Pour plus d'informations, vous pouvez lire la page de manuel boot_i386(8)

8.10 - Utilisation de S/Key sur votre système OpenBSD

S/Key est un système de mots de passe à utilisation unique, ce qui permet d'utiliser ces mots de passe au sein de canaux non sécurisés. C'est une fonctionnalité fort utile pour les personnes ne pouvant pas utiliser ssh(1) ou un autre canal chiffré. L'implémentation OpenBSD de S/Key peut utiliser différents d'algorithmes comme fonction de hachage à sens unique. Voici une liste des algorithmes disponibles :

  • md4
  • md5
  • sha1
  • rmd160.

Mise en place de S/Key - Les premières étapes

Pour commencer, le fichier /etc/skeykeys doit exister. Si ce fichier n'existe pas, demandez au super-utilisateur de le créer : 

    # touch /etc/skeykeys

Une fois ce fichier créé, vous pouvez initialiser S/Key. Pour cela, vous devez utiliser skeyinit(1). Avec skeyinit(1), votre mot de passe d'accès au système vous sera d'abord demandé. C'est le même mot de passe que celui que vous utilisez pour vous connecter au système. L'exécution de skeyinit(1) dans un canal non sécurisé n'est pas recommandé. Vous devriez donc effectuer cette opération à travers un canal sécurisé (tel que ssh) ou la console. Une fois que votre mot de passe est validé, skeyinit(1) vous demandera un second mot de passe. Ce dernier est le mot de passe secret et NON votre mot de passe système. Le mot de passe secret n'est pas limité à 8 caractères comme les mots de passe système; en fait, il doit avoir une longueur minimale de 10 caractères. Quelques phrases sont suggérées. Voici un exemple d'ajout d'un utilisateur : 

    oshibana:ericj> skeyinit ericj
[Adding ericj]
Reminder - Only use this method if you are directly connected
   or have an encrypted channel.  If you are using telnet
   or rlogin, exit with no password and use skeyinit -s.
Enter secret password:
Again secret password:

ID ericj skey is otp-md5 99 oshi45820             
Next login password: HAUL BUS JAKE DING HOT HOG

La ligne ID ericj skey is otp-md5 99 oshi45820 est particulièrement intéressante. Elle fournit beaucoup d'informations à l'utilisateur comme expliqué ci-après :

  • otp-md5 - Ceci indique quelle fonction à sens unique a été utilisée pour créer votre mot de passe à usage unique (One-Time Password ou otp).
  • 99 - C'est votre numéro de séquence. Ce numéro appartient à l'intervalle 1-100. Une fois le numéro 1 atteint, un autre mot de passe secret doit être généré.
  • oshi45820 - Ceci est votre clé.

Mais le plus important est votre mot de passe. Il consiste en 6 petits mots. Combinés, ils constituent votre mot de passe, espaces compris.

Utilisation de S/Key pour se connecter

Maintenant, S/Key est initialisé et vous avez votre mot de passe. Vous êtes prêt à vous connecter. Voici un exemple d'une session utilisant S/Key pour se connecter :

    oshibana:ericj> ftp localhost
Connected to localhost.
220 oshibana.shin.ms FTP server (Version 6.4/OpenBSD) ready.
Name (localhost:ericj): ericj
331 Password [ otp-md5 96 oshi45820 ] for ericj required.
Password:
230- OpenBSD 2.5-current (OSHIBANA) #8: Tue Jun 22 19:20:16 EDT 1999
230-
230- Welcome to OpenBSD: The proactively secure Unix-like operating system.
230-
230- Please use the sendbug(1) utility to report bugs in the system.
230- Before reporting a bug, please try to reproduce it with the latest
230- version of the code.  With bug reports, please try to ensure that
230- enough information to reproduce the problem is enclosed, and if a
230- known fix for it exists, include that as well.
230-
230 User ericj logged in.
Remote system type is UNIX.
Using binary mode to transfer files.
ftp> quit
221 Goodbye.

Vous avez peut-être remarqué que le numéro de séquence a changé : otp-md5 96 oshi45820. C'est normal puisque nous avons utilisé S/Key pour vous connecter plusieurs fois. Mais comment avoir votre nouveau mot de passe après une connexion ? Il vous faudra connaître votre clé et votre numéro de séquence. Vous êtes peut-être en train de chercher un moyen de vous rappeler de votre numéro de séquence. En fait, vous n'en avez pas vraiment besoin. Il vous suffit d'utiliser la commande skeyinfo(1). Par exemple, supposons que nous ayons besoin de générer un nouveau mot de passe pour une connexion future (nous le faisons toujours à travers un canal sûr).

    oshibana:ericj> skeyinfo
95 oshi45820

Maintenant, nous pouvons créer le mot de passe pour la prochaine connexion. Nous utilisons pour cela skey(1) avec la sortie de la commande précédente.

    oshibana:ericj> skey 95 oshi45820
Reminder - Do not use this program while logged in via telnet or rlogin.
Enter secret password:
NOOK CHUB HOYT SAC DOLE FUME

Il est plus que probable que beaucoup de personnes n'ont pas toujours accès à un canal sûr pour générer ces mots de passe, et leur création à travers un canal non sécurisé n'est pas faisable. Comment donc créer plusieurs mots de passe en une seule fois ? Vous pouvez fournir à skey(1) le nombre de mots de passe que vous voulez créer. Vous pourriez alors les imprimer et les emporter avec vous.

    oshibana:ericj> skey -n 5 95 oshi45820
Reminder - Do not use this program while logged in via telnet or rlogin.
Enter secret password:
91: SHIM SET LEST HANS SMUG BOOT
92: SUE ARTY YAW SEED KURD BAND
93: JOEY SOOT PHI KYLE CURT REEK
94: WIRE BOGY MESS JUDE RUNT ADD
95: NOOK CHUB HOYT SAC DOLE FUME

Il est à noter que le dernier mot de passe de la liste doit être le premier utilisé vu que nous comptons de manière décroissante à partir de 100.

Utilisation de S/Key avec telnet(1), ssh(1), et rlogin(1)

L'utilisation de S/Key avec telnet(1), ssh(1), et rlogin(1) se fait à peu près de la même manière qu'avec ftp; la seule différence notable étant l'utilisation du mot de passe "s/key" en premier. Exemple :

8.11 - Pourquoi mon Macintosh est si lent ?

C'est à cause d'un bogue matériel. OpenBSD utilise des interruptions d'horloge pour calculer la date courante, mais ces interruptions ont la plus faible priorité sur architecture Apple. Ainsi, durant des charges importantes (telles celles causées par des activités disque ou réseau), des interruptions d'horloge sont perdues et l'horloge Unix n'avance pas comme elle le devrait.

MacOS contourne ce problème de temps en accédant toujours à l'horloge matérielle. OpenBSD lit cette horloge matérielle uniquement durant la phase de démarrage puis l'ignore par après. Vous constaterez peut-être, durant la phase d'arrêt, que le noyau n'est pas suffisamment sûr du temps Unix pour écrire sur l'horloge matérielle car ce problème de temps est bien connu.

La meilleure solution consiste à utiliser ntpd (que vous pouvez trouver dans la collection des ports) et gérer les désynchronisations occasionnelles. Parfois, ces dernières sont tellement importantes que ntpd a peur d'avancer dans le temps. Dans ce cas, ajoutez l'option -g à ntpd dans le fichier /etc/rc.securelevel pour forcer la synchronisation.

Une autre méthode plus simple mais moins précise est d'utiliser rdate(8) de manière régulière, à travers une entrée crontab par exemple, avec l'option -a de préférence pour éviter les "sauts" temporels. Le fichier /etc/ppp/ppp.linkup constitue un autre bon endroit à partir duquel nous pourrions lancer rdate(8) si vous n'êtes pas connecté en permanence et que vous utilisez PPP.

Voir aussi : http://www.macbsd.com/macbsd/macbsd-docs/faq/faq-3.html#ss3.17

8.12 - Est-ce qu'OpenBSD supporte plusieurs processeurs ?

Pas en ce moment. Le travail avance et il y a un projet OpenBSD SMP (Symmetrical Multi-Processor) bien qu'il n'y ait rien d'utilisable à l'heure actuelle et aucun agenda n'a été fixé pour le support SMP.

Sur la plupart des plateformes, OpenBSD peut tourner sur un système SMP mais n'utilisera qu'un processeur. L'exception est la plateforme SPARC -- OpenBSD/SPARC ne pourra être exécuté sur certains systèmes multi-processeurs SPARC ( en règle générale : un module MBus marche toujours. Plusieurs modules MBus peuvent marcher tant qu'ils sont uni-processeurs et qu'ils soient, bien entendu, supportés). Les systèmes multi-processeurs SPARC64 marchent à condition que la machine soit supportée.

Comment puis-je aider ?

  • Ecrivez du code. Il y a beaucoup de travail à faire.
  • Donnez du matériel. Le but du projet SMP OpenBSD est de supporter le plus de plateformes possibles, SPARC, SPARC64, Alpha, MacPPC inclus et bien sûr, i386. Les donations du matériel désiré sont encouragées !
  • Ne vous proposez pas pour "tester". Le projet SMP ne fait pas encore partie du noyau du projet OpenBSD pour une raison : il n'est pas encore prêt pour être utilisé en production ou en test. Informer les développeurs que ça n'a pas marché sans le code nécessaire pour le faire fonctionner n'est pas utile maintenant. Envoyez votre dmesg. Si un développeur voit quelque chose d'intéressant dans votre configuration matérielle, il vous contactera pour tester.

8.13 - Parfois, j'ai des erreurs d'entrée/sortie lorsque j'essaie d'utiliser mes périphériques tty

Avec la version 2.6 (et antérieur) de OpenBSD, vous devez utiliser /dev/cuaXX pour les connexions initiées à partir du système OpenBSD. Les périphériques /dev/ttyXX doivent seulement être employées pour les terminaux et les connexions d'appel entrant (dial-in). Auparavant, il était possible d'utiliser les périphériques tty mais le noyau n'est plus compatible avec ce type d'utilisation.

Extrait de cua(4):

Pour les ports terminaux matériels, l'appel sortant (dial-out) est supporté à travers des noeuds périphériques correspondants dénommés unités d'appel. Par exemple, le terminal /dev/tty03 aurait une unité d'appel correspondante dénommée /dev/cua03. Ces deux périphériques sont différenciés en créant un périphérique pour l'unité d'appel avec un nombre mineur supérieur de 128 par rapport au noeud de périphérique d'appel entrant. Alors que le périphérique d'appel entrant (tty) requiert habituellement un signal matériel pour indiquer au système qu'il est actif, le périphérique d'appel sortant (dial-out) n'en pas besoin ; il peut donc communiquer sans entrave avec un équipement type modem. Ce qui veut dire qu'un programme comme getty(8) se mettra en attente par rapport à un dispositif d'appel entrant jusqu'à établissement d'une connexion. En même temps, une connexion d'appel sortant peut être établie à partir du périphérique d'appel sortant (sur le même port terminal matériel) sans gêner quoi que ce soit dans le système. Le programme getty(8) ne remarque même pas que quelque chose a lieu sur le port terminal. Si un appel de connexion arrive après que la connexion d'appel sortant soit terminée, le programme getty(8) le gérera sans problème et sans remarquer l'action générée par la connexion d'appel sortant.

8.14 - Où puis-je trouver un binaire Netscape pour OpenBSD ?

Il n'y a pas de binaire Netscape pour OpenBSD. Cependant, et puisque OpenBSD supporte l'émulation binaire de plusieurs autres systèmes d'exploitation, vous pouvez utiliser d'autres binaires.

  • Sur les systèmes I386, il est préférable d'utiliser le binaire BSDI.
  • Sur les systèmes SPARC, il est préférable d'utiliser le binaire SUN-OS.

En utilisant la ports collection, vous pouvez installer ces binaires sans problème. Les ports Navigator et Communicator sont situés sous /usr/ports/www/netscape si vous avez installé l'arborescence des ports.

8.15 - Comment s'utilise l'éditeur mg ? (faisant partie de OpenBSD post-2.6 !!)

mg est un micro éditeur texte ressemblant à Emacs. Micro veut dire qu'il est très petit (alors qu'Emacs est énorme !). Pour les notions de base, consultez la page du manuel mg(1) et le guide d'utilisation, inclus avec le code source. Si vous avez d'autres questions (comme par exemple : "Je n'ai pas de touche Meta !"), consultez la FAQ Emacs.

Vu que mg est une implémentation d'Emacs de petite taille, très similaire aux fonctionnalités d'édition de texte de Emacs 17, beaucoup de fonctionnalités de Emacs sont indisponibles (y compris des fonctionnalités de messagerie et de news et des modes d'édition pour Lisp, C++, Lex, Awk, Java, etc...)

8.16 - Apparemment, Ksh ne lit pas mon fichier .profile !

Il y a deux raisons possibles :

  • .profile n'est pas la propriété de l'utilisateur. Ceci est facile à corriger:

      # chown username ~username/.profile

  • Vous utilisez ksh à partir du système X Window

    Sous xterm, argv[0] pour ksh n'est pas précédé d'un tiret. L'utilisation d'un tiret (-) avant argv[0] permet de faire savoir à csh et à ksh qu'ils doivent interpréter leur fichiers de login. (Pour csh, c'est le fichier .login avec un fichier .cshrc séparé qui est toujours exécuté quand csh est chargé. Pour ksh, c'est plus visible car il n'y a qu'un seul script de démarrage, .profile. Ce fichier est ignoré si le shell n'est pas un shell de connexion.)

Pour corriger cela, créez un fichier dans votre répertoire HOME appelé .Xdefaults contenant la ligne XTerm*loginShell: true

    $ echo "XTerm*loginShell: true" > ~/.Xdefaults

Des fois, vous n'aurez pas besoin d"effectuer cette opération car quelques installations du système X Window ont ce paramètre par défaut. OpenBSD a choisi de respecter le comportement de XFree86.

8.17 - Pourquoi le contenu de /etc/motd est-il écrasé alors que je l'ai modifié ?

Le fichier /etc/motd est édité à chaque démarrage du système. La première ligne est alors remplacée par la version du noyau du système. Quand vous éditez ce fichier, assurez-vous de ne pas modifier les deux premières lignes afin d'éviter que /etc/rc ne les modifie quand il édite /etc/motd durant la phase de démarrage.

8.18 - Pourquoi le site www.openbsd.org est hébergé sur une machine Solaris ?

Bien qu'aucun des développeurs ne pense qu'elle soit pertinente, cette question revient assez souvent dans les listes de diffusion pour mériter une réponse ici. www.openbsd.org et le principal site ftp de OpenBSD sont hébergés par SunSITE à l'université d'Alberta au Canada. Ces sites sont hébergés sur un grand système Sun, qui a accès a beaucoup d'espace de stockage et de bande passante Internet. La présence du SunSITE permet au groupe OpenBSD d'avoir accès à cette bande passante. C'est pourquoi le site principal y est hébergé. Plusieurs miroirs OpenBSD sont sous OpenBSD, mais comme ils n'offrent aucune garantie d'avoir accès à cette large bande passante, le groupe a choisi d'héberger le site principal sur le SunSITE de l'université d'Alberta.

8.19 - J'ai des problèmes de détection de périphériques PCI sur OpenBSD 2.8

Sous OpenBSD 2.8, il y a un cas où certaines machines peuvent ne pas détecter correctement les périphériques PCI, ou mêeme se figer lorsqu'elles détectent plusieurs cartes réseau. C'est à cause du support PCIBIOS. La solution à ce problème est simple à mettre en oeuvre. Il suffit juste d'entrer dans le mode configuration durant la phase de démarrage et de désactiver le support PCIBIOS. Voici un exemple : 

    boot> boot -c

    deraadt@i386.openbsd.org:/usr/src/sys/arch/i386/compile/GENERIC
cpu0: Intel Pentium III (Coppermine) ("GenuineIntel" 686-class) 390 MHz
cpu0: FPU,V86,DE,PSE,TSC,MSR,PAE,MCE,CX8,SYS,MTRR,PGE,MCA,CMOV,PAT,PSE36,MMX,FXS
R,SIMD
real mem  = 66629632 (65068K)
avail mem = 56770560 (55440K)
using 838 buffers containing 3432448 bytes (3352K) of memory
UKC> disable pcibios
UKC> quit
[... snip ...]

Une fois cette opération effectuée, suivez les instructions indiquées dans FAQ 5, Configuration du noyau pour créer un nouveau noyau sans support PCIBIOS.
