Pour obtenir les fichiers de cette version :
  • Commandez un CDROM à partir de notre système de commandes.
  • Consultez les informations sur la page FTP pour une liste de machines miroirs.
  • Allez dans le répertoire pub/OpenBSD/4.4/ sur l'un des sites miroirs.
  • Lisez brièvement le reste de ce document.
  • Jetez un œil sur la page des errata 4.4 pour une liste des bogues et des solutions.
  • Regardez la liste détaillée des changements entre les versions 4.3 et 4.4.


Remarque : Tous les copyrights applicables et les différents crédits se trouvent dans les fichiers source contenus dans src.tar.gz, sys.tar.gz, xenocara.tar.gz ou dans les fichiers récupérés par l'intermédiaire de ports.tar.gz. Les fichiers utilisés pour construire les paquetages composant le fichier ports.tar.gz ne sont pas inclus sur le CDROM par manque de place.

Quoi de neuf ?

Ceci est une liste partielle des nouvelles fonctionnalités présentes dans OpenBSD 4.4. Pour une liste complète, voir le journal des changements qui ont mené à la création de la version 4.4.

  • Nouvelles (ou améliorations de) plates-formes :
    • OpenBSD/sparc64
      PLus de plates-formes supportées. Les machines utilisant le UltraSPARC IV/T1/T2 et le Fujitsu SPARC64-V/VI/VII sont désormais supportées.
    • OpenBSD/socppc
      Pour les machines basées sur le Freescale MPC8349E System-on-Chip (SoC) qui utilisent Das U-Boot comme boot loader.
    • OpenBSD/landisk
      Ajout du support des librairies partagées.

  • Meilleur support matériel incluant :
    • Plusieurs nouveaux/amélioration de pilotes pour les capteurs, incluant :
      • Nouveau pilote fins(4) pour le moniteur matériel Fintek F71805F LPC Super I/O.
      • Nouveau pilote andl(4) pour le moniteur matériel Andigilog aSC7611.
      • Nouveau pilote kate(4) pour le capteur de température embarqué AMD K8.
      • Nouveau pilote sdtemp(4) pour le capteur de température dans le module mémoire JEDEC JC-42.4.
      • Nouveau pilote adtfsm(4) pour le Analog Devices ADT7462.
      • Nouveau pilote km(4) pour le capteur de température embarqué AMD K10 (Phenom, Opteron Barcelona et alia).
      • Le pilote it(4) a maintenant une meilleure procédure de sondage.
      • Le pilote lmtemp(4) supporte maintenant le LM76.
    • Support pour les composants Intel G33 et G35 dans agp(4).
    • Nouveau pilote lii(4) pour les périphériques Ethernet Attansic L2 10/100.
    • Support préliminaire pour les webcams UVC USB : uvideo(4) et video(4).
    • Le pilote openchrome(4) remplace le pilote via X.Org pour les cartes vidéo VIA.
    • Le pilote X.Org pour la carte vidéo AMD Geode.
    • Nouveau pilote vmt(4) qui implémente les outils VMware.
    • Nouveau pilote auglx(4) pour les composants audio intégrés AMD Geode LX CS5536 AC'97.
    • Nouveau pilote ix(4) pour les cartes Ethernet PCI Express Intel 82598 10Gb.
    • Nouveau pilote acpithinkpad(4) fournissant un support ACPI additionnel sur les portables IBM/Lenovo ThinkPad.
    • Nouveau pilote acpiasus(4) fournissant un support ACPI additionnel sur les portables ASUS, incluant le EeePC.
    • Nouveau pilote gecko(4) supportant le port GeckoBOA BC GSC+ trouvé sur certains systèmes hppa.
    • Nouveau pilote tsec(4) supportant le contrôleur Ethernet Freescale Triple Speed.
    • Le pilote re(4) supporte maintenant les composants RTL8102E et RTL8168.
    • Le pilote cas(4) supporte maintenant les composants National Semiconductor Saturn.
    • Le pilote pccom(4) supprimé; toutes les plates-formes utilise maintenant com(4).
    • cardbus(4) et pcmcia(4) fonctionnent maintenant sur la plupart des machines sparc64.
    • Le pilote udcf(4) supporte maintenant les périphériques mouseCLOCK USB II.
    • Le pilote msk(4) supporte maintenant les composants 88E8040T.
    • Le pilote ath(4) supporte maintenant beaucoup plus de périphériques Atheros sans fil.
    • Le pilote ciss(4) supporte maintenant les périphériques HP Smart Array P212, P410, P411, P411i et P812.
    • Le pilote uftdi(4) supporte maintenant les périphériques ELV Elektronik et FTDI 2232L.
    • Le pilote umsm(4) supporte maintenant les périphériques Option GlobeTrotter 3G+, Huawei E220 et plus de périphériques HSDPA MSM devices.
    • Le pilote ubsa(4) supporte maintenant les périphériques ZTE CMDMA MSM.
    • Le pilote axe(4) supporte maintenant les périphériques Apple USB A1277.
    • Le pilote puc(4) supporte maintenant les périphériques Netmos.
    • Le pilote mgx(4) supporte maintenant l'accélération 2D surc certaines cartes.
    • Le firmware du pilote isp(4) pour certains contrôleurs a été mis à jour.
    • Le pilote isp(4) ne plante plus durant le sondage de certaines machines.
    • Le pilote bge(4) a un meilleur support des composants BCM5704 en mode fibre ce qui aide sur certains serveurs blade.
    • Le pilote bge(4) a un meilleur support des composants BCM5906 sur certains systèmes.
    • Le pilote bge(4) a un bien meilleure support des périphériques PCI Express ce qui a pour résultat d'en améliorer les performances de transmission.
    • Le pilote bge(4) supporte maintenant les périphériques BCM5714/5715/5780 utilisant une interface fibre.
    • Le pilote bnx(4) supporte maintenant les périphériques BCM5706/5708 utilisant une interface fibre.
    • Le pilote ral(4) supporte maintenant les périphériques Ralink Technology RT2700.
    • Les ports séries autre que com0 peuvent maintenant être utilisé comme console sur amd64.
    • La console série sur i386 et amd64 a une meilleure compatibilité avec les cartes de supervision serveur.

  • Nouveaux outils :
    • rpc.statd(8), le démon supervisant le statut des hotes a utiliser avec le NFS file locking daemon.
    • Import Initial de ypldap(8), un remplaçant pour ypserv pour connecter un annuaire LDAP à la famille de fonctions get{pw,gr}ent.
    • Dépréciation de slattach(8) et nmeaattach(8) en faveur de ldattach(8).
    • Import de tcpbench(1), a small TCP benchmarking tool.

  • Nouvelles fonctionnalités :
    • aucat(1) est maintenant capable de jouer et enregistrer un flux audio full-duplex, mixer un nombre illimité de flux, supporter jusqu'à 16 voix, re-echantillonner des flux à la volée, supporte plusieurs encodages 24-bit et 32-bit et fait de la conversion de format à la volée.
    • httpd(8) supporte maintenant IPv6.
    • dhcpd(8) supporte maintenant une synchronisation basique du fichier /var/db/dhcpd.leases pour permettre la redondance.
    • WPA/WPA2-PSK supporte plusieurs modèles de cartes sans-fil.
    • rpc.lockd(8) supporte maintenant le NLMv4 et fait le blocage.
    • ftp(1) supporte maintenant les transferts mget récursifs.
    • ftp(1) utilise maintenant les paquets keep alive par défaut.
    • ftp(1) accepte maintenant les mots de passe vide dans les URLs.
    • locate(1) supporte maintenant l'option -b pour faire une recherche uniquement sur le dernier composant du chemin.
    • cdio(1) accepte le mode TAO pour sélectionner la vitesse d'écriture.
    • cdio(1) n'efface plus deux fois le média.
    • Ajout de la possibilité dans cdio(1) de déterminer les capacités du média et l'afficher si le média supporte TAO ou l'effacement.
    • Version initiale de softraid(4) avec le support crypto.
    • dhcpd(8) possède maintenant les options tftp-config-file et auto-proxy-script dans dhcpd.conf(5).
    • dhclient(8) est bien plus résistant dans la gestion de ces options.
    • dhclient(8) est maintenant réactif à l'état du lien de l'interface et réagit aux changements.
    • Le support DIOCRLDINFO, DIOCGPDINFO et DIOCGPART a été ajouté aux périphériques de block qui n'en possédaient pas précédemment.
    • disklabel(8) ne supporte plus l'option '-r' et récupère toutes les information disklabel via ioctl.
    • disklabel(8) ne suggère plus les offsets et tailles qui pourraient résulter de partitions commençant ou finissant en dehors de la section du disque OpenBSD.
    • disklabel(8) lit maintenant correctement le champ 'vendor' des disklabels textes.
    • Le mode éditeur de disklabel(8) avec les commandes '?' et 'p' sont maintenant plus compactes et la commande 'l' a été ajoutée pour produire une sortie plus importante.
    • Les I/O's en dehors des limites de RAW_PART sont maintenant empêchée, permettant un détection propre des I/O invalides.
    • Les lecteurs de disquettes USB floppies possèdent maintenant un calcul du nombre de cylindres valide, à la place de 0.
    • newfs(8) peut maintenant créer des systèmes de fichiers sur des périphériques avec des tailles de secteur autre que 512, cependant ils ne peuvent pas encore être lus.
    • rtadvd(8) révoque maintenant ses privilèges et fonctionne avec son propre utilisateur _rtadvd.
    • Le sondage scsi(4) affiche moins de textes inutiles et moins de messages d'erreurs redondants.
    • Le périphérique st(4) peut maintenant être détaché.
    • Les périphériques ATAPI sont maintenant identifiés comme telles, à la place de périphériques SCSIn.
    • Les lecteurs de bandes SATA fonctionnent maintenant.
    • Le sondage scsi(4) affiche maintenant l'ID de l'initiateur sur le bus.
    • Les capacités de debug scsi(4) ont été améliorées pour afficher les commandes et les données en entrée/sortie de manière appropriée.
    • Le sondage scsi(4) fait un meilleur usage de la commande TEST UNIT READY pour effacer les erreurs et obtenir un attachement fructueux.
    • Le sondage scsi(4) peut maintenant trouver plus de périphériques fibre channel attachés.
    • Plusieurs corruptions de mbuf pool cache ont été corrigés.
    • identd(8) supporte maintenant l'IPv6 en mode standalone.
    • cal(1) affiche maintenant les numéros de semaine aussi.
    • In pf(4), implémente un sloppy tcpstate tracker qui ne fait pas du tout attention aux numéros des séquences.
    • Le compteur de règle pf(4) possède maintenant un compteur pour enregistrer combien d'état au total ont été crée par une règle.
    • La fonctionnalité kill states de pfctl(8) supporte maintenant deux cibles additionnelles : Kill by rule label ou state ID.
    • relayd(8) utilise l'état sloppy pf(4) pour les sessions routées (Direct Server Return).
    • Ajout du support dans relayd(8) des transparent L7 forwarding dans les relais.
    • Ajout du support des relais dynamiques IPv6-to-IPv4 ou IPv4-to-IPv6 TCP dans relayd(8).
    • Amélioration du mode DNS et de l'utilisation du code OpenBSD's Id shuffle de named(8). dans relayd(8).
    • Ajout dans awk(1) des opérations binaires.
    • Amélioration du code de l'affichage pour systat(1) qui ajoute des vues pour les états, règles et queues de pf(4).
    • Import initial du support pour IEEE 802.3ad/LACP dans trunk(4).

  • Améliorations et nettoyage de code :
    • Un grand changement dans le sous-système de buffer cache qui cache des pages seulement lorsqu'elles sont en service, ce qui améliorera les performances sur le système de fichiers, et permet l'utilisation effective d'un plus grand buffer cache.
    • Une implémentation grandement améliorée de malloc(3), l'alloueur de mémoire généraliste, qui récupère plus d'erreurs, réduit la fragmentation de l'espace d'adressage et est beaucoup plus rapide.
    • L'appel système statfs(2) a été amélioré pour supporter de plus grand système de fichiers.
    • La fonction strtof(3) a été ajoutée à la libc.
    • Beaucoup de travail a été effectué sur libm pour ajouter plusieurs fonctions pour tendre plus vers la compatibilié C99.
    • Beaucoup de fonctionnalités ont été implémentées dans OpenCVS, qui peut maintenant être utilisé réellement pour travailler.
    • De nouvelles APIs pour arc4random(3), une pour remplir un buffer avec des nombres aléatoires et une autre pour retourner un nombre aléatoire distribué uniformément sans effet de bord.

  • Changements dans la procédure d'installation/mise à jour :
    • Un nouvel outil sysmerge(8), dérivé du vieux port mergemaster, permet de fusionner plus facilement les fichiers de configuration durant une mise à jour.
    • Support complet de OpenBSD dans une partition étendue sur i386 et amd64.
    • Pendant l'installation 'dhcp' est maintenant la réponse par défaut durant la configuration réseau.
    • Récupérer les ensembles via FTP est maintenant plus robuste à cause de l'utilisation automatique du keep alive.
    • Récupérer les ensembles via NFS ne bloque plus en essayant un montage défaillant.
    • L'installation s'assure que les fichiers hostname.* sont installés avec le mode 600.
    • La configuration de la console série détecte maintenant automatiquement la vitesse.
    • Le support de la console série a été étendue à toutes les architectures..
    • L'affichage de la taille des partitions n'est plus limité à un taille sur 32 bit.
    • La taille des partitions est maintenant calculée et formatée avec des unités humainement lisible.
    • Les questions de configuration sur ntpd(8) ont été améliorées.
    • La détection des Sparc miniroot root disk a été corrigée.
    • L'invocation de disklabel(8) via les scripts est maintenant moins volubile.

  • OpenSSH 5.1:

    • Refus de lire les fichiers .shosts ou authorized_keys si ils ne sont pas de vrai fichiers.
    • Activation de l'échappement ~ pour les sessions esclaves multiplex.
    • Supporte l'adressage CIDR dans les blocs Match et les authorized_keys from="..." avec apostrophes.
    • Faire que le code du port forwarding teste un adressage additionnel quand il se connecte à une destination dont la résolution DNS possède plus d'une adresse.
    • Fait que le nombre maximum de sessions encours ssh(1) est contrôlé via MaxSessions dans ssh_config(5).
    • ssh-keyscan(1) utilise maintenant par défaut des clés RSA protocol 2, à la place des RSA1.
    • Ajout d'une extension au protocole sftp pour implémenter les opérations comme statvfs(3) et ajoute une commande df à sftp(1).
    • Désactive l'éxécution de ~/.ssh/rc pour les sessions ou une commande a été forcée par la directive ForceCommand dans sshd_config.
    • Et plusieurs corrections de bogues et d'amélioration des performances.

Les instructions pour faire une installation par FTP (ou tout autre type d'installation) sont très similaires ; les instructions du CDROM sont reproduites ici telles quelles, de sorte que vous puissiez voir combien cela aurait été plus facile si vous aviez acheté un CDROM.

OpenBSD/i386 :

    Manipulez les options du BIOS, et voyez si vous pouvez démarrer depuis le CDROM. La version OpenBSD/i386 est sur le CD1. Si votre BIOS ne supporte pas l'amorçage depuis le CDROM, vous devrez créer une disquette de boot pour l'installation. Pour faire une disquette de boot, copiez CD1:4.4/i386/floppy44.fs sur une disquette, et démarrez dessus.

    Utilisez CD1:4.4/i386/floppyB44.fs pour un meilleur support des contrôleurs SCSI, ou CD1:4.4/i386/floppyC44.fs pour un meilleur support des ordinateurs portables.

    Si vous ne pouvez pas démarrer à partir d'un CD ou d'une disquette, vous pouvez utiliser la procédure d'installation par le réseau en utilisant PXE tel que c'est décrit dans le document INSTALL.i386.

    Si vous désirez mélanger OpenBSD avec un autre système d'exploitation, vous aurez sans doute besoin de lire INSTALL.i386.

    Pour créer une disquette depuis MS-DOS, utilisez l'utilitaire "rawrite" disponible dans CD1:4.4/tools/rawrite.exe. Pour créer la disquette de démarrage sous Unix, utilisez l'utilitaire dd(1) L'exemple suivant est un exemple d'utilisation de dd(1), où le périphérique peut être "floppy", "rfd0c", ou "rfd0a". 

      # dd if=<file> of=/dev/<device> bs=32k

    Utilisez une disquette correctement formatée SANS BLOC DEFECTUEUX ou vous n'y arriverez pas. Pour de plus amples informations sur la création d'une disquette de démarrage et sur l'installation d'OpenBSD/i386, veuillez lire la FAQ 4.3.1.

OpenBSD/amd64 :

    La version 4.4 d'OpenBSD/amd64 se trouve sur le CD2. Démarrez à partir du CD pour commencer l'installation - vous aurez peut-être à ajuster vos options BIOS d'abord. Si vous ne pouvez pas démarrer à partir du CD, vous pouvez créer une disquette de démarrage pour faire l'installation. Pour faire cela, mettez CD2:4.4/amd64/floppy44.fs sur une disquette qui vous servira alors pour démarrer.

    Si vous ne pouvez démarrer ni à partir d'un CD, ni à partir d'une disquette, vous pouvez faire une installation par le réseau en utilisant PXE comme indiqué dans le document INSTALL.amd64 inclus.

    Si vous comptez mélanger OpenBSD avec un autre système d'exploitation, vous devez lire INSTALL.amd64.

OpenBSD/macppc :

    Placez le CD2 dans votre lecteur de CDROM et allumez votre machine, tout en maintenant la touche C enfoncée jusqu'à ce que l'affichage s'allume et affiche OpenBSD/macppc boot.

    Sinon, au prompt Open Firmware, entrez bootcd:,ofwboot /4.4/macppc/bsd.rd

OpenBSD/sparc64 :

    Placez le CD3 dans votre lecteur de CDROM et tapez boot cdrom.

    Si cela ne fonctionne pas ou si vous n'avez pas de lecteur de CDROM, vous pouvez copier CD3:4.4/sparc64/floppy44.fs ou CD3:4.4/sparc64/floppy44.fs sur une disquette et la démarrer avec boot floppy. Pour plus de détails, lisez INSTALL.sparc64.

    Assurez vous que vous utilisez une disquette correctement formatée et SANS BLOC DEFECTUEUX, ou votre installation échouera très probablement.

    Vous pouvez aussi copier CD3:4.4/sparc64/miniroot44.fs sur la partition de swap sur le disque et démarrer avec boot disk:b.

    Si rien ne fonctionne, vous pouvez démarrer à partir du réseau, comme décrit dans INSTALL.sparc64.

OpenBSD/alpha :

    Mettez FTP:4.4/alpha/floppy44.fs ou FTP:4.4/alpha/floppyB44.fs (selon votre machine) sur une disquette et saisissez boot dva0. Veuillez consulter INSTALL.alpha pour de plus amples informations.

    Assurez vous que vous utilisez une disquette correctement formatée et SANS BLOC DEFECTUEUX, ou votre installation échouera très probablement.

OpenBSD/armish :

    Connecté à un port série, le Thecus peut démarrer sur le réseau soit par tftp soit par http. Configurez le réseau en utilisant ifconfig et reset puis chargez bsd.rd ; lisez INSTALL.armish pour des informations plus précises. IOData HDL-G ne peut démarrer qu'à partir d'une partition EXT-2. Démarrez sous Linux, copiez les fichiers 'boot' et bsd.rd vers la première partition sur wd0 (hda1) puis lancez bsd.rd en préservant la partition ext2fs wdOi (hda1). Plus de détails sont disponibles dans INSTALL.armish.

OpenBSD/hp300 :

    Démarrez via le réseau en suivant les instructions incluses dans INSTALL.hp300.

OpenBSD/hppa :

    Démarrez via le réseau en suivant les instructions incluses dans INSTALL.hppa ou sur la page de la plate-forme hppa.

OpenBSD/landisk :

    Inscrivez miniroot44.fs au début de la carte CF ou du disque et bootez normalement.

OpenBSD/mac68k :

    Démarrez normalement MacOS et décompactez l'utilitaire Macside "BSD/Mac68k Booter" depuis FTP:4.4/mac68k/utils sur votre disque. Configurez le "BSD/Mac68k Booter" avec le chemin de votre noyau bsd.rd et bootez dans l'installeur. Veuillez consulter INSTALL.mac68k pour de plus amples informations. Référez vous aux instructions dans INSTALL.mac68k pour plus de détails.

OpenBSD/mvme68k :

    Vous pouvez créer une bande d'installation initialisable ou démarrer à partir du réseau.
    Le démarrage à partir du réseau requiert une version de MVME68K BUG qui supporte les commandes de débogage NIOT et NBO. Suivez les instructions dans INSTALL.mvme68k pour plus de détails.

OpenBSD/mvme88k :

    Vous pouvez créer une bande d'installation initialisable ou démarrer à partir du réseau.
    Le démarrage à partir du réseau requiert une version de MVME88K BUG qui supporte les commandes de débogage NIOT et NBO. Suivez les instructions dans INSTALL.mvme88k pour plus de détails.

OpenBSD/sparc :

    Démarrez à partir d'une des images ISO d'installation fournies, à l'aide d'une commande parmi celles décrites ci-après, suivant votre version de ROM. 
      ok boot cdrom 4.4/sparc/bsd.rd
or
> b sd(0,6,0)4.4/sparc/bsd.rd

    Si votre machine SPARC ne possède pas de lecteur cd-rom, vous pouvez démarrer à partir d'une disquette. A cette fin, vous devez écrire floppy44.fs sur une disquette. Pour de plus amples informations, veuillez prendre connaissance de l'entrée FAQ 4.3.1. Pour démarrer à partir de la disquette, utilisez une commande parmi celles décrites ci-après, suivant votre version de ROM. 

      ok boot floppy
or
> b fd()

    Assurez-vous de disposer d'une disquette correctement formatée et ne présentant AUCUN BLOC DEFECTUEUX ou votre installation échouera très probablement.

    Si votre machine SPARC n'est dotée ni d'un lecteur de disquette ni d'un lecteur cd-rom, vous pouvez soit mettre en place une bande de démarrage ou effectuer une installation par le réseau. Ces deux méthodes sont décrites dans le fichier INSTALL.sparc.

OpenBSD/sgi :

    Gravez cd44.iso et utilisez le pour démarrer depuis le lecteur CD de votre machine puis sélectionnez Install System Software à partir du menu "System Maintenance".

    Si votre machine n'est pas équipée avec un lecteur CD, vous pouvez mettre en place un serveur DHCP/tftp et démarrer en utilisant "bootp()/bsd.rd". Référez-vous aux instructions de INSTALL.sgi pour plus de détails.

OpenBSD/vax :

    Le démarrage à partir du réseau par mopbooting est décrit dans INSTALL.vax.

OpenBSD/zaurus :

    Utilisez l'installeur Linux graphique intégré ipkg, installez le paquetage openbsd44_arm.ipk. Rebootez, et lancez-le. Lisez INSTALL.zaurus pour davantage de détails.

Remarques à propos du code source :

    src.tar.gz contient une archive des sources se trouvant dans /usr/src. Ce fichier contient tout excepté les sources du noyau, qui sont dans une archive séparée. Pour l'extraire : 

      # mkdir -p /usr/src
# cd /usr/src
# tar xvfz /tmp/src.tar.gz

    sys.tar.gz contient une archive des sources se trouvant dans /usr/src/sys. Ce fichier contient toutes les sources nécessaires pour reconstruire un noyau. Pour l'extraire : 

      # mkdir -p /usr/src/sys
# cd /usr/src
# tar xvfz /tmp/sys.tar.gz

    Ces deux arborescences sont régulièrement mises à jour dans le CVS. En utilisant ces arborescences, il est possible d'utiliser les serveurs anoncvs immédiatement comme décrit ici. Utiliser ces fichiers est beaucoup plus rapide que d'obtenir la totalité des sources depuis un serveur anoncvs.

Procédure de mise à niveau

Si vous avez déjà un système OpenBSD 4.3 et vous ne voulez pas effectuer une réinstallation, la procédure à suivre est décrite dans le Guide de Mise à Niveau d'OpenBSD.

Arborescence des ports

Une archive de l'arborescence des ports est aussi fournie. Pour l'extraire : 

    # cd /usr
# tar xvfz /tmp/ports.tar.gz
# cd ports

Le sous-répertoire ports/ est un "checkout" de l'arborescence des ports OpenBSD. Veuillez lire la page http://www.OpenBSD.org/faq/faq15.html si vous ne savez pas de quoi il s'agit. Ce texte n'est pas un manuel pour utiliser les ports. C'est plutôt un ensemble de notes permettant à l'utilisateur de commencer rapidement à utiliser le système de ports OpenBSD.

Le répertoire ports/ représente une version CVS des ports (voir la page du manuel cvs(1) si vous n'êtes pas familier avec CVS). Comme pour l'arborescence des sources, notre arborescence des ports est disponible par anoncvs. Donc pour mettre à jour votre arborescence des ports/ depuis un média en lecture-écriture, il suffit de lancer la commande suivante : 

    # cd [portsdir]/; cvs -d anoncvs@server.openbsd.org:/cvs update -Pd -rOPENBSD_4_4

[Bien sûr, vous devez remplacer le répertoire local et le nom du serveur ici par l'emplacement de votre collection des ports et le serveur anoncvs le plus proche.]

Remarquez que la plupart des ports sont disponibles en tant que paquetages via FTP. Des mises à jour de paquetages pour la version 4.4 seront disponibles si des problèmes surviennent.

Si vous êtes intéressé par l'ajout d'un port, voulez aider, ou voulez juste en savoir plus, la liste de discussion ports@openbsd.org est la liste à connaître.
