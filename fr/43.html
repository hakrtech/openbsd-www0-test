Pour obtenir les fichiers de cette version :
  • Commandez un CDROM à partir de notre système de commandes.
  • Consultez les informations sur la page FTP pour une liste de machines miroirs.
  • Allez dans le répertoire pub/OpenBSD/4.3/ sur l'un des sites miroirs.
  • Lisez brièvement le reste de ce document.
  • Jetez un oeil sur la page des Errata 4.3 pour une liste des bogues et des solutions.
  • Regardez la liste détaillée des changements entre les versions 4.2 et 4.3.


Remarque : Tous les copyrights applicables et les différents crédits se trouvent dans les fichiers source contenus dans src.tar.gz, sys.tar.gz, xenocara.tar.gz ou dans les fichiers récupérés par l'intermédiaire de ports.tar.gz. Les fichiers utilisés pour construire les paquetages composant le fichier ports.tar.gz ne sont pas inclus sur le CDROM par manque de place.

Quoi de neuf ?

Ceci est une liste partielle des nouvelles fonctionnalités présentes dans OpenBSD 4.3. Pour une liste complète, voir le journal des changements qui ont mené à la création de la version 4.3.

  • Nouvelles (ou améliorations de) plates-formes :
    • OpenBSD/sparc64
      Support SMP. cela doit fonctionner sur tous les systèmes supportés à l'exception de la Sun Enterprise 10000.
    • OpenBSD/hppa
      Les serveurs K-class comme le K200 et le K410 sont maintenant supportés.
    • OpenBSD/mvme88k
      Support SMP sur les systèmes MVME188 et MVME188A.
      Le processeur 88110 et les cartes MVME197LE/SP/DP sont maintenant supportées.
    • OpenBSD/sgi
      Contient de nouveaux pilotes, cependant le noyau a besoin d'un important correctif d'errata.

  • Meilleur support matériel incluant :
    • Le pilote bge(4) supporte maintenant les périphériques Ethernet BCM5906/BCM5906M 10/100 et BCM5755 10/100/Gigabit.
    • Le pilote cas(4) supporte maintenant les périphériques Ethernet Cassini+ 10/100/Gigabit.
    • Le pilote em(4) supporte maintenant les périphériques Ethernet ICH9 10/100 et 10/100/Gigabit.
    • Le pilote gem(4) supporte maintenant l'interface embarquée 1000base-SX sur les serveurs Sun Fire V880.
    • Le pilote ixgb(4) supporte maintenant les périphériques Ethernet Sun 10Gb PCI-X.
    • Le pilote msk(4) supporte maintenant les périphériques Ethernet Yukon FE+ 10/100 et Yukon Supreme 10/100/Gigabit.
    • Le pilote nfe(4) supporte maintenant les périphériques Ethernet MCP73, MCP77 et MCP79 10/100/Gigabit.
    • Le pilote ral(4) supporte maintenant les périphériques réseau sans fil basé sur le RT2800.
    • Le pilote cmpci(4) supporte maintenant les adaptateurs audio basé sur le CMI8768.
    • Le pilote it(4) supporte maintenant les composants ITE IT8705F/8712F/8716F/8718F/8726F et SiS SiS950. Les fonctionnalités de temps Watchdog ont été ajoutées.
    • Le pilote mfi(4) supporte maintenant les contrôleurs RAID Dell CERC6/PERC6 et LSI SAS1078.
    • Le pilote viapm(4) supporte maintenant le contrôleur SMBus VIA VT8237S south bridges.
    • Le support des périphériques Expressecard hotplug a été ajouté.
    • Nouveau pilote amdpcib(4) pour les séries AMD-8111 LPC bridge et compteur de temps sur amd64.
    • Nouveau pilote pctr(4) pour les compteurs de performance CPU sur amd64.
    • Nouveau pilote bwi(4) pour les périphériques réseau sans fil Broadcom AirForce IEEE 802.11b/g.
    • Nouveau pilote envy(4) pour les périphériques audio VIA Envy24.
    • Nouveau pilote et(4) pour les périphériques réseau Ethernet Agere/LSI ET1310 10/100/Gigabit.
    • Nouveau pilote etphy(4) pour les périphériques réseau Ethernet PHY Agere/LSI ET1011 TruePHY Gigabit.
    • Nouveau pilote amdpcib(4) pour les séries AMD-8111 LPC bridge et compteur de temps sur i386.
    • Nouveau pilote glxpcib(4) pour les AMD CS5536 PCI-ISA bridge avec compteur de temps, watchdog timer et GPIO sur i386.
    • Nouveau pilote iwn(4) pour les périphériques réseau sans fil Link 4965AGN IEEE 802.11a/b/g/Draft-N.
    • Nouveau pilote de ligne msts(4) pour interfacer les périphériques Meinberg Standard Time String pour fournir un capteur de différence de temps.
    • Nouveau pilote gbe(4) pour le Frame Buffer SGI Graphics Back End (GBE) sur sgi.
    • Nouveau pilote mkbc(4) pour le contrôleur PS/2 Moosehead sur sgi.
    • Nouveau pilote power(4) pour le bouton marche/arrêt sur sgi.
    • Nouveau pilote ecadc(4) pour le capteur de température du sous-système de supervision d'environnement sur sparc64.
    • Nouveau pilote tda(4) pour le contrôleur de ventilation sur Sun Blade 1000/2000, permettant à ces machines d'être moins bruyantes.
    • Nouveau pilote spdmem(4) récupérant les informations sur les modules mémoire.
    • Nouveau pilote thmc(4) pour le capteur de température TI THMC50 et Analog ADM1022/1028.
    • Nouveau pilote uchcom(4) pour l'adaptateur série USB basé sur le WinChipHead CH341/340.
    • Nouveau pilote umbg(4) pour l'horloge radio Meinberg Funkuhren USB5131 qui fournit un capteur de différence de temps.
    • Nouveau pilote upgt(4) pour les périphériques réseau sans fil Conexant/Intersil PrismGT SoftMAC USB IEEE 802.11b/g.
    • Nouveau pilote wbng(4) pour le capteur de température, tension et ventilateur Winbond W83793G.
    • Nouveau pilote wbsio(4) pour les composants Winbond LPC Super I/O.
    • Nouveau pilote adl(4) pour le capteur de température, tension et ventilateur Andigilog aSC7621.
    • Le pilote siop(4) supporte maintenant le (non-PCI) NCR 53c720/770 en mode big-endian.
    • Nouveau pilote lmn(4) pour le capteur National Semiconductor LM93.

  • Nouveaux outils :
    • snmpd(8), implémentant le protocole de supervision réseau simple (SNMP).
    • Le programme snmpctl(8) qui contrôle le daemon SNMP.
    • L'utilitaire pcidump(8) qui affiche l'adresse, vendeur et nom du produit des périphériques PCI.
    • ldattach(8) est utilisé pour attacher une ligne discipline à une ligne série pour permettre les traitements dans le noyau de réception et/ou d'émission de données.

  • Nouvelles fonctionnalités :
    • eeprom(8) est maintenant capable d'afficher l'arbre du périphérique OpenPROM sur les systèmes qui le supportent.
    • Support X11 sur sgi a été ajouté.
    • Le rapport périodique security(8) inclut maintenant les changements de paquetage.
    • Le pilote cmpci(4) supporte maintenant la sortie audio multivoix si le matériel le supporte.
    • Le pilote auvia(4) supporte maintenant le sortie audio multivoix si le matériel le supporte.
    • Le pilote auich(4) supporte maintenant l'enregistrement du microphone aussi bien que le mode full-duplex.
    • Le pilote eso(4) supporte maintenant l'enregistrement aussi bien que le mode full-duplex.
    • La couche ffs est maintenant propre au niveau des adresses de blocs disques 64-bit. Cela signifie que les disques, partitions et systèmes de fichiers plus grand que 2To sont maintenant supportés à l'exception de statfs(2) et quotas.
    • DMA est maintenant activé sur les périphériques 1-sector comme les lecteurs flash fournissant une amélioration significative de la vitesse.
    • Les disklabels Sparc et Sparc64 fournissent maintenant une reconnaissance automatique des partitions ext2fs.
    • Les sytèmes de fichiers sur les périphériques USB sont automatiquement démonté si le périphérique est deconnecté.
    • La configuration de la répartition de charge carp(4) a été grandement simplifiée.
    • Les entrées fstab(5) se referrant à des points de montage non existent sont maintenant ignorées permettant aux entrées d'être traitées.
    • Des fichiers de configuration additionnels peuvent maintenant être inclus à pf.conf(5).
    • sppp(4) supporte maintenant IPv6.
    • ipsec.conf(5) supporte maintenant les clés définies sur 192 et 256 bit.

  • Améliorations et nettoyage de code :
    • Un support amélioré pour le sous-système lkm(4) sur amd64.
    • ossaudio(3) a reçu plusieurs correctifs de bogues et d'améliorations incluant, mais sans limite, le support amélioré de l'enregistrement et le support full-duplex.
    • audio(4) a reçu plusieurs correctifs de bogues et d'améliorations incluant, mais sans limite, le support amélioré de l'enregistrement et le support full-duplex.
    • make(1) a été grandement modifé, principalement pour améliorer le support des constructions paralléles. Les constructions paralléles exécutent maintenant les commandes de la même façon que les constructions séquentielles et la sortie de ces commandes est plus lisible. Une grande partie de l'arbre des sources, xenocara et quelques ports se construisent correctement avec make -j.
    • L'outil rcs a été amélioré et des bogues corrigés.
    • RTM_VERSION a été augmenté pour que tous les messages de routage puissent être modifiés en incluant des champs additionnels pour les futures fonctionnalités réseau.
    • sendbug(1) a un parseur de commentaires plus strict pour éviter les diffs douteux.
    • Le périphérique umass(4) ne détecte plus les LUNs boguées.
    • Les périphériques USB st(4) peuvent maintenant se deconnecter avec succés.
    • Plus de périphériques déviant umass sont pris en compte.
    • Les périphériques svnd(4) fonctionnent maintenant sur les périphériques de blocs.
    • disklabel(8) est maintenant au courant des partitions NTFS.
    • raidctl(8) prend maintenant en compte correctement les espaces en fin de ligne dans les fichiers de configuration.
    • mt(1) ne génére plus de panique quand il traite une commande 'rewoffl'.
    • Les périphériques raid(4) ne plantent plus quand ils recherchent un composant durant le démarrage.
    • Les périphériques sd(4) ne recoivent plus multiples commandes SYNCHRONIZE CACHE ce qui rend confus certains matériels.
    • sd(4) n'affirme plus que les commandes SYNCHRONIZE CACHE ont 16 octets de longueur alors qu'elles n'en possédent que 10. Certains périphériques le croient.
    • dhcpd(8) fournit maintenant toujours des paquets égales ou plus grand que le MTU IP minimum.
    • Le mode -E de disklabel(8) ne permet pas l'édition manuelle de la partition 'c' qui est toujours définie pour le disque complet.
    • Le mode -E de disklabel(8) ne permet pas de changer la valeur cpg de la partition.
    • Le mode -E de la commande 'r de disklabel(8) affiche maintenant la liste des espaces libres du disque.
    • Le mode -E de disklabel(8) ne permet plus d'assigner des tailles arbitraires aux partitions FS_BOOT et FS_UNUSED.
    • Les problèmes de pilote bge(4) recevant des jumbo frames a été résolu.
    • Plusieurs comparaisons dangereuses avec -1 quand on teste le résultat d'un appel en lecture ou écriture ont été éliminés.
    • Le flag M_ZERO pour malloc(9) remplace plusieurs combinaisons de malloc+bzero/memset, corrigeant un grand nombre de biogues dans l'initialisation de la mémoire et diminuant ainsi la taille du noyau.
    • dhcpd(8) construit maintenant correctement des paquets de réponse qui utilise les dépassements de tampons pour stocker des options.
    • Les pilotes SCSI sont plus fiables dans les machines MP à cause d'un meilleur blocage autour des fins de commande.
    • Les réponses TCP pour les paquets trés fragmentés sont maintenant construitent sans risque de corruption de la mémoire du noyau.
    • Les sockets permettent maintenant 4095 membres de groupes multicast.

  • Changements dans la procédure d'installation/mise à jour :
    • Toutes les plates-formes supporte maintenant la console série lors de l'installation.
    • La vitesse de la console série est détectée et les entrées appropriées crées automatiquement dans /etc/ttys.
    • OpenBSD/vax posséde aussi maintenant l'installation via des images CD ISO.
    • Les adreses des serveurs DNS sont maintenant récupérées si l'installation est relancée.
    • OpenBSD/sgi peut maintenant être installé avec une console graphique.

  • OpenBGPD 4.3:
    • Récupére correctement les préfixes qui peuvent créer une boucle de routage.
    • Les sorties détaillées RIB de bgpctl affichent des attributs additionnels comme les communautés étendues ou la liste des clusters id.

  • OpenNTPD 4.3:
    • Prend en compte les changements d'IP des clients plus gracieusement.
    • Enregistre les peers et statuts des capteurs sur syslog si la majorité d'entres eux sont mauvais ou si un signal SIGINFO est reçu.
    • Permet la compensation du temps de capteurs qui ont une erreur systématique.

  • OpenOSPFD 4.3:
    • Support du Equal cost multipath -- ne pas oublier de configurer les bons sysctls.
    • Les options de la ligne de commande et du parseur sont maintenant synchronisés avec bgpd.

  • relayd 4.3:
    • hoststated(8)/hoststatectl(8) a été renommé relayd(8)/relayctl(8).
    • La grammaire de configuration a été améliorée pour relayd.conf(5).
    • Permet l'envoi de traps SNMP via snmpd(8) quand l'hôte change d'état.
    • Amélioration du support pour le filtrage d'URL et des actions sur les protocoles.
    • Ajout du support pour le relai DNS basé sur UDP avec des ID requête aléatoires.
    • Plusieurs bogues corrigés, optimisations et nettoyage.
    • Amélioration du support de rechargement.

  • OpenSSH 4.8:
    • Nouvelles fonctionnalités :
      • Ajout du support chroot(2) pour sshd(8), controlé par une nouvelle option "ChrootDirectory". Merci de voir sshd_config(5) pour plus de détails et utilisez cette fonctionnalité avec prudence.
      • Lien de sftp-server(8) dans sshd(8). Le serveur sftp interne est utilisé quand la commande "internal-sftp" est spécifiée dans une déclaration Subsystem ou ForceCommand. Quand elle est utilisée avec ChrootDirectory, le serveur sftp interne ne necessite pas de configuration spéciale dans les fichiers l'environnemen chroot. Merci de consulter sshd_config(5) pour plus d'informations.
      • Ajout d'une méthode d'extension de protocole "posix-rename@openssh.com" sur sftp-server(8) pour faire des opérations atomique POSIX rename().
      • Suppression de la limite fixée à 100 fichiers pris en compte dans le sftp-server(8). Le serveur alloue maintenant dynamiquement la prise en compte du nombre de descripteurs de fichiers disponibles.
      • ssh(1) n'effectuera plus maintenant la génération de clés SSH éphémeres protocole 1 quand le mode inetd et les connexions en protocole 2 seront négociées. Cela accélerera les connexions en protocole 2 sur les serveurs en mode inetd qui autorise aussi le protocole 1.
      • Accepte la directive PermitRootLogin de sshd_config(5) dans un bloc Match. Permet par exemple uniquement le root sur le réseau local.
      • Reécriture de la découpe des arguments et échappements de sftp(1) pour être plus concis (entre commandes sftp) et plus concis avec sh(1). Il faut remarquer que cela changera l'interprétation de certaines chaines entre parenthéses, spécialement celles avec des séquences d'échappement de backslash embarqué.
      • Support de "Banner=none" dans sshd_config(5) pour désactiver l'envoi de baniére de pre-connexion (dans un bloc Match).
      • ssh(1) ProxyCommand sont maintenant éxécutées avec $SHELL plutôt que /bin/sh.
      • L'option ssh(1) ConnectTimeout est maintenant appliquée aussi bien sur les connexions TCP et l'échange de banniére SSH (précedemment cela couvrait uniquement les connexions TCP). cela permet à l'appelant de ssh(1) de mieux détecter et travailler avec des serveurs douteux qui acceptent une connexion TCP mais ne font pas progresser le protocole, et aussi faire de ConnectTimeout utile pour les connexions via un ProxyCommand.
      • Plusieurs nouveaux tests de regression, incluant des tests d'intéropérabilité avec le plink de PUTTY.
    • Les bogues significatifs suivants ont été corrigés dans cette version :
      • Les paquets SSH2_MSG_UNIMPLEMENTED ne remettent pas correctement à zéropackets la logique keepalive du client, causant des déconnexions des serveurs qui n'implémentent pas correctement "keepalive@openssh.com".
      • ssh(1) utilisait le SIG DNS RRtype obsoléte pour les clés hôte dans le DNS, à la place du RRSIG standard courant.
      • Récupére correctement les ACKs quand un upload sftp(1) a des problèmes en écriture en cours de transfert, évitant ainsi une sortie fatal() de ce qui devrait être une condition récupérable.
      • Corrige les avertissement sur les tailles de paquet. Précedemment TCP et l'agent de transfert avertissaient incorrectement sur la taille de la fenetre du canal comme de la taille du paquet, causant des erreurs fatales sous certaines conditions.
      • Beaucoup plus de bogues corrigés. Merci de lire les Notes sur la version.

Informations pour une installation rapide, faites pour les personnes familières avec OpenBSD, et l'utilisation de la commande "disklabel -E". Si vous n'êtes pas à l'aise avec l'installation d'OpenBSD, lisez les fichiers INSTALL.* correspondants cités ci-dessus !

OpenBSD/i386 :

    Manipulez les options du BIOS, et voyez si vous pouvez démarrer depuis le CDROM. La version OpenBSD/i386 est sur le CD1. Si votre BIOS ne supporte pas l'amorçage depuis le CDROM, vous devrez créer une disquette de boot pour l'installation. Pour faire une disquette de boot, copiez CD1:4.3/i386/floppy43.fs sur une disquette, et démarrez dessus.

    Utilisez CD1:4.3/i386/floppyB43.fs pour un meilleur support des contrôleurs SCSI, ou CD1:4.3/i386/floppyC43.fs pour un meilleur support des ordinateurs portables.

    Si vous ne pouvez pas démarrer à partir d'un CD ou d'une disquette, vous pouvez utiliser la procédure d'installation par le réseau en utilisant PXE tel que c'est décrit dans le document INSTALL.i386.

    Si vous désirez mélanger OpenBSD avec un autre système d'exploitation, vous aurez sans doute besoin de lire INSTALL.i386.

    Pour créer une disquette depuis MS-DOS, utilisez l'utilitaire "rawrite" disponible dans CD1:4.3/tools/rawrite.exe. Pour créer la disquette de démarrage sous Unix, utilisez l'utilitaire dd(1) L'exemple suivant est un exemple d'utilisation de dd(1), où le périphérique peut être "floppy", "rfd0c", ou "rfd0a". 

      # dd if=<file> of=/dev/<device> bs=32k

    Utilisez une disquette correctement formatée SANS BLOC DEFECTUEUX ou vous n'y arriverez pas. Pour de plus amples informations sur la création d'une disquette de démarrage et sur l'installation d'OpenBSD/i386, veuillez lire la FAQ 4.3.1.

OpenBSD/amd64 :

    La version 4.3 d'OpenBSD/amd64 se trouve sur le CD2. Démarrez à partir du CD pour commencer l'installation - vous aurez peut-être à ajuster vos options BIOS d'abord. Si vous ne pouvez pas démarrer à partir du CD, vous pouvez créer une disquette de démarrage pour faire l'installation. Pour faire cela, mettez CD2:4.3/amd64/floppy43.fs sur une disquette qui vous servira alors pour démarrer.

    Si vous ne pouvez démarrer ni à partir d'un CD, ni à partir d'une disquette, vous pouvez faire une installation par le réseau en utilisant PXE comme indiqué dans le document INSTALL.amd64 inclus.

    Si vous comptez mélanger OpenBSD avec un autre système d'exploitation, vous devez lire INSTALL.amd64.

OpenBSD/macppc :

    Placez le CD2 dans votre lecteur de CDROM et allumez votre machine, tout en maintenant la touche C enfoncée jusqu'à ce que l'affichage s'allume et affiche OpenBSD/macppc boot.

    Sinon, au prompt Open Firmware, entrez bootcd:,ofwboot /4.3/macppc/bsd.rd

OpenBSD/sparc64 :

    Placez le CD3 dans votre lecteur de CDROM et tapez boot cdrom.

    Si cela ne fonctionne pas ou si vous n'avez pas de lecteur de CDROM, vous pouvez copier CD3:4.3/sparc64/floppy43.fs ou CD3:4.3/sparc64/floppy43.fs sur une disquette et la démarrer avec boot floppy. Pour plus de détails, lisez INSTALL.sparc64.

    Assurez vous que vous utilisez une disquette correctement formatée et SANS BLOC DEFECTUEUX, ou votre installation échouera très probablement.

    Vous pouvez aussi copier CD3:4.3/sparc64/miniroot43.fs sur la partition de swap sur le disque et démarrer avec boot disk:b.

    Si rien ne fonctionne, vous pouvez démarrer à partir du réseau, comme décrit dans INSTALL.sparc64.

OpenBSD/alpha :

    Mettez FTP:4.3/alpha/floppy43.fs ou FTP:4.3/alpha/floppyB43.fs (selon votre machine) sur une disquette et saisissez boot dva0. Veuillez consulter INSTALL.alpha pour de plus amples informations.

    Assurez vous que vous utilisez une disquette correctement formatée et SANS BLOC DEFECTUEUX, ou votre installation échouera très probablement.

OpenBSD/armish :

    Connecté à un port série, le Thecus peut démarrer sur le réseau soit par tftp soit par http. Configurez le réseau en utilisant ifconfig et reset puis chargez bsd.rd ; lisez INSTALL.armish pour des informations plus précises. IOData HDL-G ne peut démarrer qu'à partir d'une partition EXT-2. Démarrez sous Linux, copiez les fichiers 'boot' et bsd.rd vers la première partition sur wd0 (hda1) puis lancez bsd.rd en préservant la partition ext2fs wdOi (hda1). Plus de détails sont disponibles dans INSTALL.armish.

OpenBSD/hppa :

OpenBSD/landisk :

    Inscrivez miniroot43.fs au début de la carte CF ou du disque et bootez normalement.

OpenBSD/mac68k :

    Démarrez normalement MacOS et décompactez l'utilitaire Macside "BSD/Mac68k Booter" depuis FTP:4.3/mac68k/utils sur votre disque. Configurez le "BSD/Mac68k Booter" avec le chemin de votre noyau bsd.rd et bootez dans l'installeur. Veuillez consulter INSTALL.mac68k pour de plus amples informations. Référez vous aux instructions dans INSTALL.mac68k pour plus de détails.

OpenBSD/mvme68k :

    Vous pouvez créer une bande d'installation initialisable ou démarrer à partir du réseau.
    Le démarrage à partir du réseau requiert une version de MVME68K BUG qui supporte les commandes de débogage NIOT et NBO. Suivez les instructions dans INSTALL.mvme68k pour plus de détails.

OpenBSD/sparc :

    Démarrez à partir d'une des images ISO d'installation fournies, à l'aide d'une commande parmi celles décrites ci-après, suivant votre version de ROM. 
      ok boot cdrom 4.3/sparc/bsd.rd
or
> b sd(0,6,0)4.3/sparc/bsd.rd

    Si votre machine SPARC ne possède pas de lecteur cd-rom, vous pouvez démarrer à partir d'une disquette. A cette fin, vous devez écrire floppy43.fs sur une disquette. Pour de plus amples informations, veuillez prendre connaissance de l'entrée FAQ 4.3.1. Pour démarrer à partir de la disquette, utilisez une commande parmi celles décrites ci-après, suivant votre version de ROM. 

      ok boot floppy
or
> b fd()

    Assurez-vous de disposer d'une disquette correctement formatée et ne présentant AUCUN BLOC DEFECTUEUX ou votre installation échouera très probablement.

    Si votre machine SPARC n'est dotée ni d'un lecteur de disquette ni d'un lecteur cd-rom, vous pouvez soit mettre en place une bande de démarrage ou effectuer une installation par le réseau. Ces deux méthodes sont décrites dans le fichier INSTALL.sparc.

OpenBSD/sgi :

    Gravez cd43.iso et utilisez le pour démarrer depuis le lecteur CD de votre machine puis sélectionnez Install System Software à partir du menu "System Maintenance".

    Si votre machine n'est pas équipée avec un lecteur CD, vous pouvez mettre en place un serveur DHCP/tftp et démarrer en utilisant "bootp()/bsd.rd". Référez-vous aux instructions de INSTALL.sgi pour plus de détails.

OpenBSD/zaurus :

    Utilisez l'installeur Linux graphique intégré ipkg, installez le paquetage openbsd43_arm.ipk. Rebootez, et lancez-le. Lisez INSTALL.zaurus pour davantage de détails.

Remarques à propos du code source :

    src.tar.gz contient une archive des sources se trouvant dans /usr/src. Ce fichier contient tout excepté les sources du noyau, qui sont dans une archive séparée. Pour l'extraire : 

      # mkdir -p /usr/src
# cd /usr/src
# tar xvfz /tmp/src.tar.gz

    sys.tar.gz contient une archive des sources se trouvant dans /usr/src/sys. Ce fichier contient toutes les sources nécessaires pour reconstruire un noyau. Pour l'extraire : 

      # mkdir -p /usr/src/sys
# cd /usr/src
# tar xvfz /tmp/sys.tar.gz

    Ces deux arborescences sont régulièrement mises à jour dans le CVS. En utilisant ces arborescences, il est possible d'utiliser les serveurs anoncvs immédiatement comme décrit ici. Utiliser ces fichiers est beaucoup plus rapide que d'obtenir la totalité des sources depuis un serveur anoncvs.

Arborescence des ports

Une archive de l'arborescence des ports est aussi fournie. Pour l'extraire : 

    # cd /usr
# tar xvfz /tmp/ports.tar.gz
# cd ports

Le sous-répertoire ports/ est un "checkout" de l'arborescence des ports OpenBSD. Veuillez lire la page http://www.OpenBSD.org/faq/faq15.html si vous ne savez pas de quoi il s'agit. Ce texte n'est pas un manuel pour utiliser les ports. C'est plutôt un ensemble de notes permettant à l'utilisateur de commencer rapidement à utiliser le système de ports OpenBSD.

Le répertoire ports/ représente une version CVS des ports (voir la page du manuel cvs(1) si vous n'êtes pas familier avec CVS). Comme pour l'arborescence des sources, notre arborescence des ports est disponible par anoncvs. Donc pour mettre à jour votre arborescence des ports/ depuis un média en lecture-écriture, il suffit de lancer la commande suivante : 

    # cd [portsdir]/; cvs -d anoncvs@server.openbsd.org:/cvs update -Pd -rOPENBSD_4_3

[Bien sûr, vous devez remplacer le répertoire local et le nom du serveur ici par l'emplacement de votre collection des ports et le serveur anoncvs le plus proche.]

Remarquez que la plupart des ports sont disponibles en tant que paquetages via FTP. Des mises à jour de paquetages pour la version 4.3 seront disponibles si des problèmes surviennent.

Si vous êtes intéressé par l'ajout d'un port, voulez aider, ou voulez juste en savoir plus, la liste de discussion ports@openbsd.org est la liste à connaître.
