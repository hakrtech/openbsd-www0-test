OpenBSD/hppa

OpenBSD/hppa fonctionne sur une large gamme de stations de travail et de serveurs Hewlett Packard Precision Architecture 32-bit (HP PA-RISC).

Une liste de diffusion dédiée au port OpenBSD/hppa est disponible à l'adresse hppa@openbsd.org. Pour vous inscrire à la liste de diffusion OpenBSD/hppa, envoyez un message dont le corps contient "subscribe hppa" à majordomo@openbsd.org. Veuillez relire notre politique des listes de diffusion avant de vous inscrire.

Le mainteneur actuel du port est Michael Shalayeff. Si d'autres personnes souhaitent contribuer, elles sont bien entendu les bienvenues !

Historique :

Ce projet a été initié à une époque où les seuls systèmes d'exploitation libres pour les machines HP PA-RISC étaient Lites et MkLinux. Ces deux sources fûrent des fournisseurs majeurs d'information et de code pour le développement initial du port OpenBSD/hppa. HPBSD est encombré avec du code propriétaire HP et des licences USL licensing. De plus, il n'est pas publiquement disponible. Un des principaux bénéfices directs des efforts de portage précédents (en plus du code bien entendu) est le support de HP PA-RISC par GCC et binutils, ce qui a permit de construire rapidement un environnement basé sur des outils de développement croisés.

Plus tard, HP a décidé de sponsoriser un portage Linux sur des machines PA-RISC qui a abouti dans les faits à la publication d'une documentation propriétaire, même si les machines ciblées sont des classes différentes de machines 9000.

Etat actuel :

Bien qu'aucune version officielle de ce port n'ait vu le jour, les systèmes supportés démarrent en mode multi-utilisateur et des snapshots complets sont mis à disposition périodiquement.

Projets (sans ordre particulier) :

  • Améliorer la stabilité générale
  • Supporter le contrôleur SCSI Narrow SE NCR53C700
  • Supporter le contrôleur SCSI FWD NCR53C720
  • Procédures d'installation
  • Améliorer la documentation spécifique au port
  • Supporter le démarrage à partir d'une bande et d'un CD-ROM
  • Support des processeurs 7300LC et 8x00
  • Support des processeurs 7000 (les machines "original snakes")
  • Emulation HP-UX
  • Emulation PARISC/Linux
  • Librairies partagées
  • Support X11

Matériel supporté :

Modèles supportés

  • A base de PA7100 : 715/33, 715/50, 715/75, 725/50, 725/75, 735/100, 755/100
  • A base de PA7100LC : 712, 715/64, 715/80, 715/100, 715/100XC, 725/64, 725/100
  • A base de PA7150 : 735/125, 755/125
  • A base de PA7200 : J200, J210, J210XC, C100, C110

Périphériques supportés

Cette liste du matériel supporté n'a pour but d'être exhaustive. Les développeurs ont besoin de matériel afin de poursuivre le développement et de faire plus de tests. Veuillez consulter notre "Liste de souhaits" (contactez mickey).

    • CRX-8, CRX-24, HCRX-8, HCRX-24
  • Ports série
    • ns16550 et compatibles intégrés (com)
  • Ports parallèle
    • Port parallèle intégré (lpt)
  • Contrôleurs SCSI
    • Contrôlleur SCSI Narrow SE NCR53C710 (osiop)
  • Ethernet
    • Intel i82596 ethernet intégré (ie)
  • Périphériques audio
    • Harmony CS4215/AD1849 intégré (harmony)
  • Périphériques d'entrée
    • Claviers HIL keyboards (hilkbd)
    • Souris et tablette graphique HIL (hilms)

    • Souris PS/2 (pms)
  • Périphériques temps réel
    • Horloge temps réel avec batterie de secours.

Modèles non supportés

Les systèmes suivants pourraient fonctionner :

  • A base de PA7000 : 705, 710, 720, 730, 750
  • A base de PA7300LC : A180[C], B132L[+], B160L, B180L+, C132L, C160L
  • Systèmes PA2.0 (en mode 32-bit)
  • HP 9000 836, 844, 845, 846
  • HP 9000 E-Class (avec E/S limitée)
  • HP 9000 D-Class, au moins les modèles 32-bit
  • Hitachi VQ200, VQ210, et quelques autres stations de travail non-HP
  • Hitachi 9000V 3050RX/200, 3050RX/100C et 3500/*
  • PrecisionBook de RDI Computer Corporation
  • Systèmes à carte unique HP VME et VXI (nécessitent au moins VME)

Les systèmes suivants ne sont très probablement pas supportés vu qu'ils ont des firmwares incompatibles (les 3000) ou possèdent des sous-systèmes d'E/S pratiquement non documentés :

  • HP 3000 900
  • HP 9000 8x7 (F, G, H, I-Class)

Les systèmes suivants ne seront, probablement, jamais supportés par ce port vu qu'ils ont une architecture PA-RISC 1.0, architecture substantiellement inférieure à la 1.1 :

  • HP 9000 600, 635, 645
  • HP 9000 800, 808, 815, 822, 825, 832, 834, 835, 836, 840, 842, 844, 845, 846, 850, 852, 855, 860, 865, 870, 890
  • HP 9000 900

Obtenir et installer OpenBSD/hppa :

Des snapshots sont mis à disposition de temps à autre, à cet endroit ainsi que sur quelques miroirs. Voici de même les instructions d'installation d'OpenBSD/hppa snapshot.

Démarrage du média d'installation par le réseau :

Le fichier lif33.fs contient un chargeur de démarrage et une installation miniroot. Pour le démarrer à partir du réseau, une machine serveur devra être configurée pour permettre à la machine hppa de le télécharger :

  1. Configuration serveur
    • Pour les anciennes machines (705, 7[1235]0, old 715) rbootd doit être configuré et démarré.
      • /etc/rbootd.conf contient une seule ligne par machine devant démarrer à partir du serveur. Chaque ligne contient l'adresse ethernet de la machine hppa cliente, et le nom de fichier à charger à partir du serveur tftp (relatif à /usr/mdec/rbootd). Par exemple : 
        	8:0:9:70:c4:11    lif33.fs    # 9000/720
	8:0:9:19:fb:a0    lif33.fs    # 9000/725
    • Les machines récentes (712, 715 récentes, 725) n'ont pas besoin de rbootd, mais plutôt d'une configuration bootp.
      • /etc/bootptab contient un bloc d'informations par machine. Par exemple : 
        	# 9000/715
	boron:\
        	ha=0800095f5a6e:\
        	bf=lif33.fs:\
        	hd=:\
        	ht=ethernet:\
        	ip=10.0.1.143:\
        	sm=255.255.255.0:\
        	td=/tftpboot:
      • Créer un répertoire /tftpboot et y mettre le fichier lif33.fs.
      • Mettre une entrée pour la machine hppa dans le fichier /etc/ethers : 
        	08:00:09:5f:5a:6e	boron
      • Mettre une autre entrée pour la machine hppa dans le fichier /etc/hosts: 
        	10.0.1.143	boron
      • La dernière étape consiste à activer les services nécessaires pour que cela fonctionne. bootpd et tftpd sont en général démarrés par inetd, donc /etc/inetd.conf devra être modifié et inetd redémarré. De même, rarpd doit être activé au niveau de rc.conf.
    • Pour de plus amples informations concernant la configuration du serveur pour un démarrage par le réseau, veuillez consulter la page de manuel diskless(8).

  2. Démarrage du client par le réseau
    • BOOT_ADMIN
      Afin d'obtenir le prompt BOOT_ADMIN, tapez sur la touche <Echap> au démarrage de la machine. Une fois les périphériques de démarrage sondés, un menu sera affiché; si le prompt n'est pas BOOT_ADMIN, il sera nécessaire de saisir ``a'' pour y accèder.
    • Commande de démarrage par le réseau
      Une fois au prompt 'BOOT_ADMIN', la commande 
      	BOOT_ADMIN> boot lan isl
      lancera le processus de démarrage à partir de lif33.fs.
