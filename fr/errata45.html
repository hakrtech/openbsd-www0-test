Les correctifs ci-dessous sont disponibles sur CVS via la branche OPENBSD_4_5 stable.

Pour des informations plus détaillées à propos de l'installation de correctifs sur OpenBSD, consultez la FAQ OpenBSD.

  • 006: CORRECTIF DE FIABILITE : 24 Juin 2009   Toutes les architectures
    Une erreur "off-by-one" dans la fonction inflate de Zlib.xs du module perl Compress::Raw::Zlib avant la 2.017 (CVE-2009-1391), qui est utilisé dans AMaViS, SpamAssassin et surement d'autres produits, permet à un attaquant en fonction du contexte de causer un déni de service (blocage ou crash) via un flux compressé fait spécialement pour cela qui déclenchera un dépassement de tampon du tas.
    Un correctif corrigeant le problème au niveau du code source est disponible.

  • 005: CORRECTIF DE FIABILITE : 24 Avril 2009   Toutes les architectures
    Sur un système très chargé, une interruption audio peut intervenir pendant que le processus audio remplit les tampons audio. Ceci déclenche de faux (et inutile) code de correction dans le pilote audio(4) causant l'application audio d'être désynchronisée, et causant à son tour un bégaiement continu jusqu'a ce que l'application soit redémarrée.
    Un correctif corrigeant le problème au niveau du code source est disponible.

  • 004: CORRECTIF DE FIABILITE : 24 Avril 2009   Toutes les architectures
    En mode serveur quand en mode full-duplex (celui par défaut) aucat(1) enverra chaque message de synchronisation deux fois, faisant croire aux applications clientes un probléme sur les tampons. Selon l'application, cette cause fait bégayer le son.
    Un correctif corrigeant le problème au niveau du code source est disponible.

  • 003: CORRECTIF DE FIABILITE : 24 Avril 2009   seulement i386
    Quand la mémoire DMA est mappée par pilotes de périphériques, les drapeaux de mappage et de protection sont partiellement non initialisé. En fonction du contexte d'appel, cela peut causer un mauvais fonctionnement des périphériques, comme audio(4) qui bégaye, mais d'autres anomalies peuvent être observées pour d'autres types de périphériques.
    Un correctif corrigeant le problème au niveau du code source est disponible.

  • 002: CORRECTIF DE FIABILITE : 11 Avril 2009   Toutes les architectures
    Lorsque pf tente d'effectuer la traduction d'un datagramme IP spécialement conçue, la déréférence d'un pointeur nul se produira, entraînant une panique du noyau. Dans certaines configurations cela peut-être déclenché par un attaquant distant.
    Restreindre les règles de traduction aux protocoles qui sont spécifiques aux versions d'IP utilisés est un contournement valide tant que le correctif n'est pas installé. Comme par exemple, pour des règles de nat/binat/rdr IPv4 vous pouvez utiliser : 
        nat/rdr ... inet proto { tcp udp icmp } ...
    Ou pour les règles de nat/binat/rdr IPv6 vous pouvez utiliser : 
        nat/rdr ... inet6 proto { tcp udp icmp6 } ...
    Un correctif corrigeant le problème au niveau du code source est disponible.

  • 001: CORRECTIF DE FIABILITE : 8 Avril 2009   Toutes les architectures
    Le code OpenSSL qui gére ASN.1 peut-être forcé à faire des accés mémoire invalides avec l'utilisation de certaines chaines de caractéres invalides (CVE-2009-0590) ou sous certaines conditions d'erreurs déclenchées par des structures ASN.1 invalides (CVE-2009-0789). Ces vulnérabilités peuvent être exploitées pour faire du dénis de service. Une description plus détaillée de ces problèmes est disponible sur OpenSSL security advisory, mais il est à remarquer que d'autres problèmes décris dans "Incorrect Error Checking During CMS verification" sont relatifs à du code non activé dans OpenBSD.
    Un correctif corrigeant le problème au niveau du code source est disponible.
