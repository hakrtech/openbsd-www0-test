Pour des informations plus détaillées à propos de l'installation de correctifs sur OpenBSD, consultez la FAQ OpenBSD.

  • 008: CORRECTIF DE SECURITE : 14 Janvier 2009   Toutes les architectures
    named(8) ne teste pas correctement la valeur de retour de la fonction de vérification DSA, potentiellement cela permet de contourner la vérification des signatures DNSSEC DSA. CVE-2009-0025.
    Un correctif corrigeant le problème au niveau du code source est disponible.

  • 007: CORRECTIF DE SECURITE : 9 Janvier 2009   Toutes les architectures
    La librairie OpenSSL ne vérifie pas correctement la valeur de retour de certaines fonctions de vérification, permettant à la validation d'être contournée et permettant à un attaquant distant de faire une attaque "man in the middle" sur les connexions SSL/TLS si le serveur est configuré avec un certificat DSA ou ECDSA. CVE-2008-5077.
    Un correctif corrigeant le problème au niveau du code source est disponible.

  • 006: CORRECTIF DE FIABILITE : 19 Novembre 2008   Toutes les architectures
    A cause d'un changement dans la gestion des options, cela cause des problémes avec certains clients DHCP comme Solaris/OpenSolaris et certains routeurs embarqués qui n'acceptent pas les offres DHCP.
    Un correctif corrigeant le problème au niveau du code source est disponible.

  • 005: CORRECTIF DE FIABILITE : 7 Novembre 2008   Toutes les architectures
    Un bogue logiciel peut entrainer une allocation mémoire qui entraine une panique du noyau accédant à un tableau en dehors de ces limites, quand la mémoire physique est épuisée.
    Un correctif corrigeant le problème au niveau du code source est disponible.

  • 004: CORRECTIF DE FIABILITE : 6 Novembre 2008   Toutes les architectures
    Corrige le module httpd(8) mod_proxy qui ne fonctionne pas sur les architectures 64-bit. A cause de ce bogue, il en résultera un crash des processus enfants quand des régles proxy seront utilisées durant une session HTTP.
    Un correctif corrigeant le problème au niveau du code source est disponible.

  • 003: CORRECTIF DE FIABILITE : 6 Novembre 2008   Toutes les architectures
    Corrige la pile TCP/IP IPv4 dans le recyclage de la socket TIME_WAIT. A cause de ce bogue une connexion entre deux IP peut-être remise à zéro au lieu d'être acceptée si elle est reçue sur une socket en état TIME_WAIT.
    Un correctif corrigeant le problème au niveau du code source est disponible.

  • 002: CORRECTIF DE FIABILITE : 2 Novembre 2008   Toutes les architectures
    A cause d'un bogue dans le pilote vr(4), il est possible pour un système utilisant le pilote vr(4) de paniquer sous une forte charge si le chemin RX se retrouve sans mbufs.
    Un correctif corrigeant le problème au niveau du code source est disponible.

  • 001: CORRECTIF DE SECURITE : 2 Novembre 2008   Toutes les architectures
    Le protocole ndp (Neighbor Discovery Protocol) ne vérifie pas correctement les requétes de sollicitation des voisins permettant peut-être à un attaquant non loin d'intercepter le trafic. L'attaquant doit posséder une connexion IPv6 sur le même routeur que sa cible pour exploiter cette vulnérabilité. CVE-2008-2476.
    Un correctif corrigeant le problème au niveau du code source est disponible.
